설치/삭제 파이프라인을 BFF가 직접 운전한다

모든 Terraform 호출이 사람 수동이던 것을, 브라우저 세션 없이 자동·복구가능하게 만들고 실행 히스토리를 일급으로 조회·알림한다.

메커니즘은 BFF 내부 durable state machine + 30초 reconciler tick 하나다. DB가 유일한 상태.

이 문서는 ADR-016을 처음 보는 아키텍트를 위한 시각화 요약이다. 정본은 docs/adr/ADR-016-*.md + design/pipeline/*.md이며, 충돌 시 정본이 우선한다.

핵심 결정과 거부한 대안

결정. BFF 내부 durable state machine + reconciler tick을 둔다. DB가 유일한 상태, 30초 tick이 각 task를 전진시키고 pipeline 상태는 task에서 파생된다. 다중 replica는 advisory-lock 리더 + 모든 전이 CAS로 안전하다.
옵션결정이유
A. BFF 내부 durable reconciler (DB row + tick)채택실제 워크로드에 부합 · 재시작 안전 · 관리자 조정형 폴링이 next_check_at에 자연 대응 · 최소 운영 footprint
B. 별도 오케스트레이터 마이크로서비스보류모듈 분량 로직에 서비스 분량 오버헤드. 부하가 강제하면 이 길
C. 워크플로 엔진 (Temporal/Airflow)거부≥10분 폴링의 2–4 step 선형 체인이 그 비용을 정당화 못 함
D. BFF 인메모리 async 체인거부재시작/배포에 run 유실; durable 큐·수일 대기 표현 불가
E. Backend Manager에 파이프라인 상태거부로직·상태 분산; 원격 API 경유 원자적 slot 회계는 racy
F. IM 측 동시성 제한 (지금)보류현재 자동화 caller는 BFF뿐; 다른 caller 등장 시 IM 429/503 재검토

가장 비싼 감수: BFF가 더 이상 stateless proxy가 아니다(DB·백그라운드 루프·리더 선출). restart-safety·히스토리·조회성이 같은 뿌리라 감수한다.

설계를 이루는 결정들

ADR 본문 결정 1·3·4·5·6·7 + task-model 결정 2. 결정 8은 없다(v1에서 unique 제약으로 대체됨 — v1/v2).

결정 1 · 오케스트레이션

DB 단일 상태 + tick. API는 외부 호출도 task 전진도 안 한다 — 전적으로 reconciler의 몫. 단 하나의 예외: RUNNING→CANCELLING(중단)은 Admin API가 직접.

결정 2 · 작업 모델 (task-model)

TaskKind 2종: TERRAFORM_JOB(dispatch→poll), CONDITION_CHECK(조건 폴링). 새 task = 새 코드 class(대개 기존 kind 재사용). 임의 훅 조합·범용 컨텍스트 없음.

결정 3 · 정합성

exactly-once 기계 없이 idempotency-by-construction + at-least-once dispatch. 모든 dispatch는 멱등이어야 한다(INSTALL 이미존재=성공, DELETE 이미부재=성공) — task 등록 계약.

결정 4 · Liveness

무한 대기 없음 — execution timeout + WAIT_EXTERNAL TTL. 죽일 수 없거나 systemic한 실패는 corruption이 아니라 delay. circuit breaker 없음(timeout+retry+알림 롤업).

결정 5 · 수동 개입

retry = 새 run 생성(재개 아님; terminal 부활 없음). 완료분은 terraform 수렴으로 사실상 no-op. + 동일 target 중복은 unique 제약으로 1건만(기존 반환).

결정 6 · 외부 호출 실행

tick은 외부 호출을 비블로킹 async로 발사(200초+ 호출이 tick을 안 멈춤). 상태=tick, 관측=호출 스레드(쓰기 책임 분리).

결정 7 · Definition

recipe = (type,provider)당 코드 default 1개. 실행 구성은 불변 snapshot으로 박제해 재현. (per-target custom override는 v2.)

결정 4b · 동시성

동시 실행은 고정 풀 M이 hard-cap. BFF N-cap(N≈M)은 pubsub 큐를 얕게 하는 제출 throttle이지 동시성 보장이 아니다.

tick과 단일 writer 분리

이 ADR에서 가장 자주 오해되는 핵심. crash 안전성·동시성·장시간 호출이 전부 이 한 규칙에서 나온다.

규칙. 상태 전이(task.status)는 tick만 쓴다. 관측(task_check)과 산출(attempt.response)은 호출 스레드가 쓴다. 실행 주체는 status를 절대 건드리지 않는다 → "전이 도중 죽음" 같은 중간 상태가 없다.

dispatch 5단계 (writer 분리의 정수)

  1. tick tx · task → DISPATCHING (CAS) + task_attempt 행 생성 + next_check_at
  2. 호출 스레드 tx · task_check(kind=DISPATCH) PENDING 선기록
  3. 호출 스레드 · dispatch API 호출 (IM run)
  4. 호출 스레드 tx · attempt.response 영속화 + 같은 task_check 행 UPDATE
  5. 다음 tick · 적재된 관측 보고 → RUNNING (CAS)

■ 상태 전이 = tick   ■ 관측·산출 = 호출 스레드

중간에 죽어도: status는 직전 값으로 일관 유지되고, 복구는 재호출로 끝난다(poll/조건확인은 읽기라 반복 안전). "호출 시도 vs 미시도"는 task_check 행 존재 여부(PENDING)로 일급 사실이 된다.

왜 async 발사인가

tick이 200초 호출을 순차로 기다리면 같은 tick의 다른 task를 전부 200초씩 민다. 그래서 비블로킹 발사하고 루프는 안 막힌다. 구현(Virtual Thread)은 아키텍처 불변식이 아니다 → implementation-notes.md.

leader는 효율 장치일 뿐

모든 전이가 CAS이고 모든 side effect가 반복 안전이라, advisory lock이 misfire해도 중복 호출·no-op 쓰기를 낭비할 뿐 상태는 일관하다. 정확성은 leader에 의존하지 않는다.

정합성 — exactly-once 없이 안전

worker에 dedup이 없다는 제약(#3)을 BFF가 흉내 내지 않는다. 안전성은 작업의 멱등성에서 온다.

멱등성 불변식 (계약)

모든 dispatch 작업은 멱등이어야 한다 — 중복 실행되어도 인프라가 손상되지 않고 "이미 원하는 상태"를 성공으로 반환. BFF는 이를 런타임에 검증 못 하므로 task 등록 계약 + 리뷰로 강제한다. 비멱등 작업은 거부.

fail_count = 성공하지 못한 시도 수

IM 거부·check 에러·crash 미수신을 구분 없이 센다. K = max_fail_count = 초기 dispatch 포함 최대 attempt 수. K번째도 성과 없으면 task FAILED. (총 attempt ≤ K, 재dispatch ≤ K−1.)

crash & N-pod walkthrough (요약)

Crash 지점복구
dispatch 호출 후, response 영속 전DISPATCHING 잔류 → 재dispatch(fail_count++); 멱등성이 중복 흡수
poll/check 응답 후, 기록 전관측 1회 유실; 다음 cadence 재독(읽기는 반복 안전)

두 불변식이 이 표를 닫는다 — ① DB가 유일 상태이고 모든 전이는 guarded write(CAS), ② 모든 외부 side effect는 반복 안전.

동시성 — M · N · K를 헷갈리지 말 것

세 글자가 각각 다른 일을 한다. 가장 흔한 오독 지점이라 명시적으로 가른다.

M — 고정 worker 풀

동시성 hard cap
동시 terraform 실행은 풀 크기 M을 넘지 못한다. 초과 제출은 pubsub 큐가 흡수. 진짜 상한은 여기.

N — slot cap (N≈M)

제출 throttle
BFF가 active TF task(DISPATCHING|RUNNING)를 COUNT해 N 미만일 때만 admit. 큐를 얕게 유지할 뿐 동시성 안전장치가 아니다(split-brain 일시 초과도 풀이 흡수).

K — max_fail_count

재시도 한도
task당 최대 attempt 수. N·K는 retry/orphan worst-case 제출량 산정 참고값이지 동시성 상한이 아니다.
BFF가 보장하는 것: BFF-visible active TF task ≤ N (happy-path soft target). 보장하지 않는 것: 실제 global worker job concurrency ≤ N 또는 ≤ N·K (그건 고정 풀 M이 책임). 그래서 admission을 CAS화하지 않는다 — soft여도 풀이 받쳐주므로 무해.

poll 부하는 slot과 무관(수천 WAIT_EXTERNAL 가능)하므로 별도 축 max_external_calls_per_tick(tick당 발사 상한)으로 burst만 완화한다.

Liveness와 취소

무한 대기 없음

층만료 시
호출별 HTTP deadline그 호출을 CALL_TIMEOUT으로 끊음(task 실패 아님)
Dispatch 복구response 없는 DISPATCHING 재dispatch
Execution timeoutattempt 실패(EXECUTION_TIMEOUT) · slot 해제
WAIT_EXTERNAL TTLtask EXPIRED → pipeline FAILED
worker outage는 일반 경로(timeout+retry)로 처리하고 알림만 롤업(WORKER_OUTAGE_SUSPECTED). 감지 latency ~30분+는 구조적 상수.

취소 = 죽이지 않고 전진만 멈춤

IM에 cancel API가 없다(확정). 그래서 forward edge만 gate(readying·dispatching·retrying), drain edge는 절대 gate 안 함.
  • 미dispatch task · DISPATCHING(job_id 미영속) → 즉시 CANCELLED
  • CONDITION_CHECK(read-only) → 폴링 중이어도 즉시 CANCELLED
  • RUNNING TERRAFORM_JOB(job_id 영속) → 자연 terminal까지 drain(slot 보유)
결론: 취소는 실패·leak할 수 없고 늦을 수만 있다(execution timeout이 bound).
CANCELLING precedence(결정 1.1). 파생은 매 tick 우선순위 순 — ① CANCELLING이 최우선 > ② fail_count==K FAILED > ③ TTL EXPIRED FAILED > ④ 전 task DONE. 판정 기준은 실패 시각이 아니라 파생 시점의 pipeline.status(상태 기준). 그래서 "취소 중 task가 실패해도 pipeline은 CANCELLED로 단일 수렴".

Pipeline Definition과 snapshot

"어떤 task를 어떤 순서로"를 무엇이 정의하고, 실행된 구성을 어떻게 재현하는가.

Task catalog (코드)

각 task = 코드 class. TaskKind 2종 위의 실행 단위. 데이터로 동작을 정의하지 않는다.

Default recipe (코드)

(type,provider)당 코드 1개 · release version. 단일 출처라 변경이 전 target에 자동 반영(drift 없음).

Snapshot (불변 기록)

생성 시 1회 박제(write-once, 1:1). 그 run의 definition 원본 — 재현·이력 권위.
절연 범위 — 과장 금지. snapshot·task row가 freeze하는 것은 recipe/config (task 목록·순서·ttl·polling·execution_timeout·max_fail_count; 호출별 HTTP deadline은 task별 아닌 전역+TaskKind 설정이라 비포함)다. default release를 올려도 in-flight·과거 run의 구성은 불변. 그러나 task class 코드(핸들러 동작)는 freeze 대상이 아니다 — reconciler는 현재 배포 코드로 실행한다 (코드=실행 권위). 비호환 변경은 새 task class로 도입한다.

중복 pipeline 방지는 별도 큐가 아니라 부분 unique 제약: unique(target_source_id) WHERE status NOT IN (DONE,FAILED,CANCELLED). target당 non-terminal pipeline 1건을 강제하고, 중복 생성(재시도 포함)은 새 행 대신 기존 1건을 반환한다 (created=false). "target당 실행자 1" 전제가 이로써 성립한다.

데이터 모델 (BFF DB)

현재 상태는 CAS로 갱신(pipeline·task), 일어난 일은 행 추가(task_attempt·task_check·pipeline_event). 4 grain: run → task → attempt → observation.

Pipeline 상태기계 (5종)

파생 우선순위(매 tick). ① CANCELLING이 최우선 > ② fail_count==K > ③ TTL EXPIRED > ④ 전 task DONE. ①이 ②③을 누르므로 "취소 중 task가 실패해도 pipeline은 CANCELLED로 단일 수렴".
From→ To트리거 / Guard
(생성)RUNNINGpipeline 생성 즉시
RUNNINGCANCELLING[중단] API → CAS prior=RUNNING (terminal이면 0행 no-op = 부활 차단)
RUNNINGFAILEDtick 파생 ②/③: ¬CANCELLING ∧ (status=FAILED task[재시도 소진 fail_count==K 또는 HANDLER_NOT_FOUND 영구 실패] · 또는 TTL EXPIRED task)
RUNNINGDONEtick 파생 ④: ¬CANCELLING ∧ 전 task DONE

terminal(DONE·FAILED·CANCELLED)에서 나가는 전이는 없다("terminal은 terminal"). retry는 전이가 아니라 새 pipeline 생성.

Task 상태기계 (9종)

비terminal 5종 + terminal 4종. 전진 경로는 kind가 정한다. slot 큐 대기는 별도 상태가 아니라 READY ∧ kind=TERRAFORM_JOB로 표현(WAITING_SLOT 제거).

비terminal 5종(BLOCKED READY DISPATCHING RUNNING WAITING_EXTERNAL) · terminal 4종(DONE FAILED EXPIRED CANCELLED). slot 큐 대기 = READY ∧ kind=TERRAFORM_JOB(별도 상태 아님). 분기(실패·취소·재시도)는 아래 표.

종류전이규칙
종결RUNNING → FAILEDpoll FAILED / execution timeout / IM 거부로 fail_count++ == K
종결WAITING_EXTERNAL → EXPIREDWAIT_EXTERNAL TTL 초과 → pipeline FAILED 파생 (EXPIRED는 TTL 전용)
재시도RUNNING → READYfail++ < K → slot 반납 후 재큐(admission 재통과)
재시도DISPATCHING (in-place)dispatch 복구 timeout → fail++; slot 보유한 채 재dispatch
취소BLOCKED·READY → CANCELLED미dispatch → 즉시 CANCELLED (forward edge gate)
취소DISPATCHING (TF) → CANCELLEDjob_id 미영속 → 폴링할 handle 없음 → 즉시 CANCELLED(원 dispatch 멱등이라 무해; BFF 추적 중단이라 worker terraform 자연 종료가 bound, BFF execution timeout 아님)
취소RUNNING (TF) → drainjob_id 영속된 죽일 수 없는 job → 자연 terminal까지 drain; pipeline만 CANCELLED로 수렴
task CANCELLED ≠ pipeline CANCELLED. task의 CANCELLED는 한 번도 dispatch되지 않은 task에만 붙는다. dispatch한 in-flight job은 자기 terminal(DONE/FAILED)로 drain되고 그 사실은 보존된다 — pipeline 파생만 CANCELLED로 수렴.

v1 / v2 경계

v1은 "고정 풀에서 설치/삭제 자동화 + 재시작 안전"에 집중한다. 아래는 의도적으로 v2로 미룬 표면이다(YAGNI — 구체 need가 생길 때 설계). v1 결함이 아니다.

v2로 미룬 것왜 v2인가
scheduling 예약 실행v1은 모든 pipeline이 즉시 실행 — 예약 요구가 아직 없다
per-target 직렬화 큐 (구 결정 8)v1은 unique 제약으로 target당 1건만 허용해 충분 — 큐·게이트 불요
custom per-target recipev1은 코드 default로 전 target 커버 — 커스터마이즈 요구가 아직 없다
postCheck + 로그 캡처terminal 로그 보존은 안전한 write-once 캡처법(redaction)이 확정된 뒤
알림 라우팅 + Slack/Emailv1은 인앱 알림(pipeline_event)만으로 충분
skip-completed (부분 재실행)v1 retry = full re-run이 terraform 수렴으로 안전
GENERAL_JOB (비-terraform job kind)v1 task는 TERRAFORM_JOB + CONDITION_CHECK로 모든 흐름 표현
ADR 범위 밖: 생성 트리거의 정확한 endpoint·주체는 기존 install/delete 흐름(ADR-006/009)이 소유하므로 구현 시 배선한다. ADR이 못 박는 생성 계약은 resolution → 원자적 task row+snapshot → 중복 차단(기존 반환) 셋뿐이다.

Infra Manager(Terraform job API) / Backend Manager(연동·승인·도메인 상태).
TERRAFORM_JOB
dispatch→job_id→poll→terminal. slot(N) 소비.
CONDITION_CHECK
dispatch 없이 외부 조건 MET까지 폴링. attempt 없음, slot 불요.
drain
취소 시 죽일 수 없는 in-flight job을 자연 terminal까지 폴링하는 것(forward edge만 막음).
outbox
상태 변경과 같은 tx에 쓰인 pipeline_event가 알림의 단일 원천. 전달은 at-least-once(인앱은 read-dedup으로 effectively-once).
WAIT_EXTERNAL TTL
CONDITION_CHECK이 외부 대기에 머무는 총 시간 상한(기본 7일). 초과 시 EXPIRED.
