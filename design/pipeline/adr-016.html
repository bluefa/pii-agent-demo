ADR-016: Install/Delete Pipeline — 내구성 상태 머신 도메인 모델

제안됨
제안됨 — 2026-06-27

install/delete 파이프라인 설계의 도메인 절반: 지속 상태, 데이터 모델, 유일성 규칙, 장애 시맨틱, lifecycle. 실행 모델 — 상태 머신이 어떻게 전진 구동되는지 (러너, 워커 풀, 동시성, 크래시 복구) — 은 ADR-021에서 별도의 독립적으로 재검토 가능한 결정으로, 도메인 모델을 다시 열지 않고도 실행 전략을 변경할 수 있다.

배경

현재 고객 인프라를 설치하거나 삭제하려면 운영자가 브라우저 세션에서 InfraManager를 통해 Terraform job을 수동으로 실행해야 한다. 이를 자동화하고자 한다: Admin 콘솔에서 시작한 후 재시작에도 살아남으며 스스로 완료되고, 실행 이력이 가시적으로 남는다.

  • Admin 콘솔은 파이프라인을 생성한 후 닫힌다; 브라우저가 계속 열려 있지 않는다.
  • 오케스트레이터가 파이프라인을 전진 구동한다 (런타임은 ADR-021).
  • InfraManager는 Terraform job을 비동기로 실행한다: 하나의 디스패치가 N개의 job id 세트를 반환하고, Kubernetes worker pod가 각각의 apply를 나중에 실행한다.
  • BackendManager는 통합/승인 및 타겟-소스 데이터를 소유한다.

규모: ~2,000 타겟; ~12 파이프라인 형태 (provider × install/delete). Terraform job은 수 분 소요; 조건 확인은 수 초에서 수 일까지 폴링한다.

제약 사항
  1. InfraManager는 중복 제거가 없다 — 동일 job을 두 번 제출하면 두 번 실행되지만, 모든 실행 API는 멱등적이므로 인프라 결과는 손상되지 않는다.
  2. 결과가 소실될 수 있다 (드문 워커 장애); '여전히 실행 중'과 '소실됨'을 구분하지 않는다 — 실행 타임아웃이 둘 모두를 흡수한다.

결정

1 DB가 유일한 상태

파이프라인의 상태는 데이터베이스 행에 존재하며, 소실될 수 있는 인메모리 권위(authority)는 없다. 파이프라인을 구동하는 것(ADR-021)은 진행 상황에 대해 무상태(stateless)이며, 행을 읽고 재개한다. 아래의 모든 결정은 이 원칙에 의존한다.

2 두 도메인 테이블, 소규모 내구성 상태 머신

pipeline과 task는 도메인 상태 테이블이다 (전체 스키마는 아래 스키마 섹션 참고).

Task:      BLOCKED ──▶ READY ──▶ IN_PROGRESS ──▶ DONE | FAILED | CANCELLED
Pipeline:  RUNNING ─────────────────────────▶ DONE | FAILED | CANCELLED

현재 태스크는 가장 낮은 seq를 가진 READY/IN_PROGRESS 태스크이며, 그 앞의 태스크들은 선행 태스크가 DONE에 도달할 때까지 명시적으로 BLOCKED 상태이다 (태스크는 BLOCKED로 생성되어 READY로 전환되며, 첫 번째 태스크는 READY로 시작). 파이프라인 상태는 저장된 프로젝션으로, 상태를 변경하는 태스크 전환과 동일한 트랜잭션에서 기록되어 스캔 시 저렴하게 필터링할 수 있다.

다섯 개의 핵심 enum (TaskStatus, PipelineStatus, TaskKind, PipelineType, ErrorCode), 그리고 연산 집합이 폐쇄(closed)일 때 조건부 TaskOperation이 추가된다 (개방 집합은 레지스트리로 검증). 태스크의 kind는 실행자(executor)를 선택하고, operation은 그 안에서 도메인 액션을 선택한다. 파이프라인의 레시피(정렬된 태스크 목록)는 (type, provider)별 코드 기본값이다.

3 관측은 상태와 분리

두 개의 관측 테이블 — task_attempt (재시도 시도별 결과)와 task_check (시도별 폴링 요약) — 은 운영자가 장애를 초기 진단하는 데 필요한 정보를 담는다: 시도별 실행된 job, 최종 결과, TTL 만료 조건이 NOT_MET인지 API 실패인지, 폴링 횟수, 마지막 외부 응답. 이 테이블들은 태스크 완료 여부를 판단하는 완료 check(attempt, task)가 읽는 결과도 보관한다 — 리컨사일러는 최신 attempt 행만, 그것도 오직 완료 판정을 위해서만 읽는다; 클레임, 스케줄링, 파이프라인 전환은 관측 테이블을 절대 읽지 않는다. 행이 소실되더라도 상태는 절대 손상되지 않는다: 최신 결과가 없으면 executionTimeout으로 낙하하여 재디스패치(멱등적)되며, 비용은 지연이지 정확성 손실이 아니다 (세 가지 불변 조건은 스키마 섹션에 있다). 상태나 enum을 추가하지 않는다.

4 타겟당 하나의 활성 파이프라인

유일성 규칙은 타겟당 하나의 비종단(non-terminal) 파이프라인만 허용한다. 중복 생성 요청은 — 어떤 유형이든 — 오류를 반환하는 대신 기존 활성 실행을 반환하며, 트리거 엔드포인트는 이 계약을 준수해야 한다. 이것이 ADR-021이 파이프라인당 단일 소유자로 추론할 수 있는 전제이다.

5 정확성은 exactly-once가 아닌 멱등성에 의존

모든 디스패치는 멱등적이다: 중복 제출이 발생해도 인프라는 여전히 올바른 상태를 유지한다 ("이미 원하는 상태"도 성공으로 간주). 이를 통해 실행 모델은 at-least-once 방식이면서도 정확할 수 있다 — "InfraManager가 job을 시작했다"와 "우리가 attempt 결과를 기록했다" 사이에 크래시가 발생해도 재디스패치로 복구된다 — 그리고 상태 머신에서 DISPATCHING 상태를 제거할 수 있다. InfraManager는 중복 제거를 하지 않으므로 (제약 1), 재디스패치로 무해한 중복 job이 생성될 수 있다. 단일 TERRAFORM_JOB 디스패치는 N개의 job id 세트를 생성하며, attempt 결과 (job id 세트와 응답)는 task_attempt에 기록된다. 태스크 완료는 해당 결과에 대한 코드 수준 check — check(attempt, task) → done? — 이며, 도메인 job 컬럼이 아니다. 결과가 소실되어도 태스크는 멈추지 않는다: 태스크별 executionTimeout이 발동하여 새로운 실행(fresh run)으로 재디스패치(멱등적)되므로, 정확성은 job id 보존에 의존하지 않는다. 우리는 절대 prior job을 "재확보"하지 않는다; (task_id, attempt_no)는 논리적 시도 식별자이며, InfraManager 키가 아니다.

6 제한된 대기와 재시도

  • 태스크별 fail_count. 디스패치 또는 폴링 실패 시 증가하며, maxFailCount 미만이면 새로운 실행(fresh run)으로 재시도하고 (완료된 작업은 no-op — Terraform이 수렴), 이상이면 태스크는 FAILED 상태가 된다.
  • 두 가지 데드라인: 호출별 타임아웃과 태스크별 executionTimeout (TF) / ttl (조건); 둘 다 별도 상태가 아닌 정식 ErrorCode 값으로 매핑된다.
  • 서킷 브레이커 없음 — 시스템적 장애는 지연(타임아웃 + 재시도 + 알림)이지 데이터 손상이 아니다.

7 최소 lifecycle

두 가지 태스크 종류 (TERRAFORM_JOB, CONDITION_CHECK). 재시도는 새로운 실행이다. 취소는 직접 CANCELLED로 수렴하며 — CANCELLING 상태는 없다 — 모든 비종단 태스크 (BLOCKED/READY/IN_PROGRESS → CANCELLED)를 종단화한다; FAILED 파이프라인은 실패 태스크를 FAILED로, 나머지를 CANCELLED로 표시한다. 종단 상태는 절대 부활하지 않는다. 취소가 실행 중인 워커에 어떻게 적용되는지는 실행 관심사이다 (ADR-021).

검토한 대안

옵션 판정 이유
A. 내구성 DB 상태 머신 — 두 도메인 테이블, 상태는 행 채택 행이 곧 상태이다; 재시작 안전; 멱등성으로 at-least-once 실행이 정확해진다.
B. 최대 모델 (관측 원장, 시도 로그, 이벤트 아웃박스, 스냅샷; 6 테이블) 기각 '두 가지 태스크 종류의 정렬된 체인을 하나의 타겟에 실행'하기에 지나치게 크다 (히스토리 참고).
C. 임베디드 JSON 태스크 목록이 있는 단일 행 기각 태스크별 쿼리/인덱스 손실 (현재 태스크, 기한 스캔, 재시도 횟수); 자식 task 테이블이 더 저렴하다.

결과

장점
  • 현재 상태는 하나의 규칙: 행. 멱등 재디스패치를 통해 크래시와 재배포에서 자가 복구 — exactly-once 메커니즘 불필요.
  • 소규모이고 안정적: 두 도메인 테이블, 다섯 핵심 enum, 두 태스크 종류. 실행 전략(ADR-021)이 변경되어도 모델은 변하지 않는다.
감수할 비용
  • 호출별 전체 감사 원장이나 이벤트 아웃박스 없음. 감사 = 로그/메트릭 + pipeline/task 행 + 두 관측 테이블. 워커 중단 및 큐 대기 알림은 연기됨.
  • 타겟별 유일성은 동일 타겟에 대한 동시 INSTALL과 DELETE를 설계상 거부 — 의도된 동작이지 제한이 아니다.

스키마

도메인 상태 테이블

pipeline(id, type, target, status, created_at, last_activity_at)
  — 실행 시 추가: next_due_at, claimed_by, claimed_until, cancel_requested (ADR-021 참고)
task(id, pipeline_id, seq, kind, operation, status, fail_count, error_code,
     started_at, ready_at, finished_at, next_check_at, ttl, polling_interval,
     execution_timeout, max_fail_count)
  — job_id 컬럼 없음: 하나의 디스패치가 생성하는 N개의 job id는 task_attempt에 저장되며,
    완료는 최신 attempt 결과에 대한 코드 수준 check(attempt, task)로 판정한다.

관측 테이블 (시도별; 최신 행만 — 완료 check에 의해서만 — 읽힘; 그 외는 없음)

task_attempt(id, task_id, attempt_no, job_ids, status, error_code, dispatch_response_code,
            dispatch_response_summary, started_at, finished_at)
  — 재시도 시도당 하나의 행; attempt_no = task.fail_count + 1.
    job_ids는 해당 디스패치의 InfraManager job id 세트를 보관한다 (디스패치 1회 → N개 job);
    dispatch_response_code / dispatch_response_summary는 HTTP 상태와 응답 요약을 담는다.
    최신 attempt 행이 완료 check의 입력이다.
task_check(id, task_attempt_id, call_count, not_met_count, api_error_count,
           call_timeout_count, last_external_status, last_response_code,
           last_response_summary, last_checked_at)
  — 시도당 최대 하나의 행 (1:0..1), 제자리 UPDATE; 행 수는 폴링이 아닌 시도 수와 함께 증가.
    두 태스크 종류 모두 (TF-job 상태 폴링 및 조건 확인)에 대한 시도의 폴링/확인 단계를 요약.

관계: pipeline 1:N task 1:N task_attempt 1:0..1 task_check.

관측 불변 조건
  1. 리컨사일러는 최신 task_attempt/task_check 행만 읽으며, 오직 태스크 완료 평가(check(attempt, task))를 위해서만 읽는다; 클레임, 스케줄링, 파이프라인 전환은 오직 pipeline/task에만 의존한다.
  2. task_check는 제자리 UPDATE (시도당 하나의 행): 폴링별 삽입 없음, RLE 없음, pruner 없음.
  3. 관측 행이 소실되더라도 상태는 절대 손상되지 않는다: 최신 결과가 없으면 check가 태스크별 executionTimeout으로 낙하하여 새로운 실행(fresh run)을 재디스패치(멱등적)한다 — 소실의 비용은 재디스패치(지연 + 무해한 중복 job)이지, 정확성 손실이 아니다.

열거형 (정식 값)

열거형 값
TaskStatus BLOCKED, READY, IN_PROGRESS, DONE, FAILED, CANCELLED
PipelineStatus RUNNING, DONE, FAILED, CANCELLED
TaskKind TERRAFORM_JOB, CONDITION_CHECK
PipelineType INSTALL, DELETE
ErrorCode JOB_FAILED, EXECUTION_TIMEOUT, TTL_EXPIRED, CHECK_ERROR, CALL_TIMEOUT

TaskOperation은 조건부 여섯 번째 enum으로, 연산 집합이 폐쇄(closed)일 때만 존재하며, 개방/설정 집합은 레지스트리를 대신 사용한다.

용어

  • InfraManager — Terraform job을 실행 (비동기; 하나의 디스패치가 N개의 job id 세트를 반환; worker pod가 각각의 apply 수행).

  • Terraform job — 하나의 인프라 apply; 수 분 소요.
  • 현재 태스크 — RUNNING 파이프라인에서 가장 낮은 seq를 가진 READY/IN_PROGRESS 태스크.
