ADR-021: Install/Delete Pipeline — Claim-Pull 실행 모델

제안됨
제안됨 — 2026-06-27

이것은 install/delete 파이프라인 설계의 실행 절반이다: ADR-016의 내구성 상태 머신이 실제로 어떻게 전진 구동되는지를 다룬다. 도메인 모델(테이블, 상태, 유일성, 라이프사이클, 멱등성 계약)은 ADR-016에 있으며 이 결정이 바뀌어도 변하지 않는다. 실행 모델을 별도 ADR로 관리하면 미래의 변경이 도메인 모델을 건드리지 않고 이 ADR만 대체할 수 있다.

배경

ADR-016은 데이터베이스 행이 파이프라인의 상태 그 자체이며 모든 디스패치가 멱등적임을 확립했다 (따라서 at-least-once 전달이 정확하다). 그것이 결정하지 않은 것은 런타임이다: 오케스트레이터가 어디서 동작하는지, 몇 개의 인스턴스가 돌아가는지, 기한이 된 작업을 어떻게 찾는지, 동시 외부 호출을 어떻게 제한하는지.

가장 지배적인 런타임 제약은 외부 호출 지연 시간의 무한한 가변성이다. InfraManager의 "run" API는 비동기적이다 — 짧은 호출이 job id를 반환하지만, 실제 Terraform job은 수 분 동안 실행되고, 조건 확인은 수 초에서 수 일까지 폴링한다. 실행 모델은 이를 막힘 없이 흡수해야 하고, 프로세스 재시작에서도 살아남아야 한다 (ADR-016이 상태 손실 없음을 보장하며, 진행만 일시 중단된다).

선택을 형성하는 두 가지 동시 쓰기 사실:

  • 취소는 Admin/API 경로에서 발행된다 — 워커가 틱(tick)을 실행하는 동안 별도 프로세스, 자체 단기 트랜잭션에서 언제든지 발행될 수 있다. "단일 프로세스만 쓰기 권한을 가진다"고 가정하는 모델은 구조적으로 깨진다.
  • 프로세스 중복은 일상적이다: 롤링 배포, 스케일 이벤트, 재시작이 둘 이상의 인스턴스가 동시에 살아 있는 창(window)을 만든다. 인메모리 in-flight 가드는 프로세스 단위이므로 이 경계를 넘어 보호를 제공하지 못한다.

운영상 사실:

  • InfraManager는 자체 고정 워커 풀을 가진다 — 이것이 동시 Terraform job의 실질적 상한이다. 오케스트레이터는 V1에서 자체 하드 캡이 필요 없다.
  • InfraManager에 과잉 제출은 무해하다 (멱등적; InfraManager의 큐만 깊어질 뿐).
  • 규모는 소규모: ~2,000 타겟, 분 단위 job.

1 워커가 DB에서 작업을 풀(Pull)한다; 단일 인스턴스 제약 없음, 리더 선출 없음

오케스트레이터는 독립적인 배포 가능한 서버로 실행된다. N개의 워커 스레드 — 선택적으로 여러 파드/레플리카에 걸쳐 — 가 각자 데이터베이스에서 기한이 된 파이프라인을 풀한다. 레플리카 수 제약 없음, 인메모리 in-flight 가드 없음, 리더 선출 없음. 수평 스케일과 HA가 설계에 내재되어 있다.

2 FOR UPDATE SKIP LOCKED + 리스를 통한 파이프라인 레벨 클레임

워커는 외부 시스템에 닿기 전 짧은 전용 트랜잭션(tx1)을 실행하여 기한이 된 파이프라인 하나를 클레임한다:

BEGIN;
SELECT id FROM pipeline
 WHERE status = 'RUNNING'
   AND next_due_at <= now()
   AND (claimed_until IS NULL OR claimed_until < now())
 ORDER BY next_due_at
 LIMIT 1
 FOR UPDATE SKIP LOCKED;
UPDATE pipeline
   SET claimed_by = :claim_token,
       claimed_until = now() + (:lease_seconds * interval '1 second')
 WHERE id = :pipeline_id;
COMMIT;

SKIP LOCKED는 동일 스캔을 경쟁하는 두 워커가 서로를 차단하지 않고 서로 다른 파이프라인을 클레임하도록 보장한다. claimed_by / claimed_until 스탬프 — 프로세스 카운트가 아닌 — 가 프로세스와 파드 경계를 넘어 파이프라인이 한 번에 하나의 워커에 소유됨을 보장한다.

Fencing token — a per-claim UUID, not a stable pod id.

claimed_by는 클레임 순간에 발행되는 새 UUID(:claim_token)이며, 재사용되는 파드 id나 스레드 이름이 아니다. 이는 스테일 낙오자 거부에 중요하다: 워커의 리스가 만료되고 같은 파드가 나중에 동일 파이프라인을 재클레임하면 다른 토큰을 받으므로, 이전 클레임의 in-flight tx2는 구 토큰을 보유하여 claimed_by = :claim_token 가드에서 no-op된다. 재사용되는 stable identity를 쓰면 그 가드를 통과하여 스테일 쓰기가 허용된다.

ORDER BY next_due_at + SKIP LOCKED는 근사적/공정한 FIFO이지, 엄격한 순서가 아니다. 다른 워커가 보유한 행은 건너뛰므로 서로 다른 동시 워커가 서로 다른 상대적 순서를 관측할 수 있다. 파이프라인들이 서로 독립적이므로 이는 수용 가능하다.

빈 클레임 결과가 백로그가 비었다는 의미는 아니다. SKIP LOCKED는 다른 워커의 리스가 보유한 행을 조용히 생략한다. 스캔 루프는 0행 결과를 유휴 신호로 처리해서는 안 된다; 정상적으로 다음 폴 이터레이션으로 진행해야 한다.

실행 스키마 참고. 이 모델은 pipeline 테이블에 실행 조정 컬럼 네 개를 추가한다 — next_due_at, claimed_by, claimed_until, cancel_requested (boolean) — 이들은 ADR-016의 도메인 상태 컬럼과 구별되는 이 ADR 소유의 실행 메타데이터이다.

3 두 트랜잭션 분리

tx1: claim     →     external call (OUTSIDE any transaction)     →     tx2: report

외부 호출 (InfraManager, 조건 확인)은 200 ms–60 s가 소요된다. 그 창 동안 행 잠금을 유지하면 동일 행을 스캔하려는 다른 모든 워커가 차단된다. 두 트랜잭션 분리로 이를 피한다: tx1이 파이프라인을 클레임하고 즉시 커밋; 외부 호출이 잠금 없이 실행; tx2가 결과를 커밋한다.

4 소유권 보호 쓰기 (Ownership-guarded write-back)

한 줄 요약

보고(tx2)는 워커가 여전히 클레임을 소유하고 있을 때만 (WHERE claimed_by = :claim_token) 반영된다; 그렇지 않으면 no-op된다. 이 단일 소유권 가드 — 취소 케이스별 status에 단일 쓰기 권한 규칙과 함께 — 가 쓰기 안전성의 전부이므로, status 가드는 필요 없다.

쓰기. 보고 트랜잭션(tx2)은 다음으로 태스크와 파이프라인을 전환한다:

UPDATE pipeline
   SET status = :new_status, ...
 WHERE id = :pipeline_id
   AND claimed_by = :claim_token;

tx2는 단일 트랜잭션 내에서 고정 순서로 실행된다: (1) 파이프라인 행을 SELECT ... FOR UPDATE; (2) claimed_by = :claim_token 검증 — 실패 시 전체 보고를 no-op; (3) cancel_requested 읽기; (4) 태스크 및 파이프라인 status 쓰기 — 플래그가 설정된 경우 파이프라인과 모든 비종단 태스크 (현재 태스크 + 아직 BLOCKED인 후속 태스크)를 ADR-016 §7에 따라 CANCELLED로 유도; 그렇지 않으면 정상 현재 태스크 전환 적용; (5) 다음 next_due_at 쓰기 및 클레임 해제.

소유권은 파이프라인 레벨에서 한 번만 검증된다. claimed_by 확인은 잠긴 파이프라인 행에서 이루어지며, 태스크와 파이프라인 쓰기는 그 검증된 단일 쓰기 권한 클레임 하에서 발생하므로, 행별 클레임 가드가 필요 없다 — task 테이블은 claimed_by 컬럼을 갖지 않는다.

성공 시 tx2는 클레임을 해제하고 next_due_at을 원자적으로 전진한다. 태스크/파이프라인 상태를 전진하는 트랜잭션이 동시에 claimed_by/claimed_until을 지우고 새 next_due_at을 쓴다. 이 해제가 없으면 단계를 완료했지만 RUNNING으로 남은 파이프라인이 claimed_until이 지날 때까지 잠긴 채 다른 워커를 차단한다. 현재 태스크가 DONE에 도달하면 같은 tx2가 다음 태스크를 BLOCKED → READY로 전환한다. 디스패치 시 워커는 task_attempt 행을 쓰고 (시도의 job_ids 집합 + 응답), 각 폴에서 해당 시도의 task_check 요약 (횟수 + 마지막 응답)을 제자리 UPDATE한다; 모두 검증된 파이프라인 클레임 하에서 (단일 쓰기 권한, 태스크 레벨 클레임 불필요). 태스크 완료는 최신 시도 행에 대한 코드 레벨 check(attempt, task)로 판단한다 — 리컨사일러는 완료 확인을 위해서만 해당 행을 읽으며, 클레임·스케줄링·파이프라인 전환은 시도 테이블을 읽지 않는다. 소실된 시도 결과가 태스크를 멈추지 않는다: executionTimeout으로 흘러내려 새 N-job 실행을 재디스패치한다 (멱등적).

위가 메커니즘이고; 아래는 왜 레이스 안전한지를 다룬다.

왜 status 가드가 필요 없는가. 단독 claimed_by 가드가 모든 낙오자를 차단한다: 리스가 만료된 워커가 파이프라인이 재클레임된 후 재개되면 claimed_by가 더 이상 일치하지 않으므로 업데이트가 no-op된다 (클로버 없음). status = :expected_status 가드는 아무것도 추가하지 않는데, 행이 종단화되는 동안 일치하는 라이브 토큰을 보유한 다른 쓰기 권한자가 절대 없기 때문이다 — 취소는 즉시 경로(결정 6)에서만 status를 쓰고, 이는 클레임이 null/만료 상태일 때만 발동하며 종단화하면서 claimed_by를 지운다; 라이브 리스 파이프라인에서 취소는 플래그만 설정하고 클레임 보유자가 유일한 status 쓰기 권한자이다. tx2 쓰기가 취하는 동일 행 잠금 하에서 cancel_requested를 읽는 것(3단계)이 협력적 취소를 레이스 안전하게 만드는 것이다 — 3단계 직전에 커밋한 취소는 관측되고; 직후에 커밋하는 것은 tx2가 커밋될 때까지 차단된 후 다음 클레임에서 반영된다.

왜 중복 외부 호출이 무해한가. 두 트랜잭션 분리는 InfraManager가 두 번 호출될 수 있는 세 가지 유계(bounded) 창을 만든다:

  • (a) 클라이언트 타임아웃 — 호출자가 타임아웃되지만 InfraManager가 이미 job을 수락했다.
  • (b) 디스패치 후 크래시 — 워커가 디스패치 후 tx2가 반환된 job_ids를 기록하기 전에 크래시한다.
  • (c) 지연 중 리스 만료 — 워커 스레드 풀 큐 대기 또는 GC 일시정지가 리스를 초과; 다른 워커가 재클레임하여 동일 파이프라인을 디스패치한다.

세 경우 모두에서 중복 외부 호출은 인프라 멱등성으로 안전하다 — TF API는 중복 무해이다 (재디스패치가 무해한 중복 job을 만들 수 있고; 시도에 기록되는 job_ids는 최신 디스패치의 집합이다). 보호된 DB 쓰기는 다른 문제를 해결한다: 스테일 낙오자가 DB 상태를 클로버하는 것을 막는다. 둘은 상호보완적이지 대체 가능하지 않다 — 멱등성은 외부 중복을 커버하고, 가드는 DB 쓰기를 커버한다.

5 리스 만료를 통한 크래시 복구

크래시된 워커가 클레임한 파이프라인은 claimed_until < now()가 되면 다시 기한이 된다. 리더 없음, 인간 개입 없음 — 다음 스캔이 재클레임한다. 하드 제약:

lease_seconds > max_single_call_timeout + pool_queue_wait + scheduling_margin

클레임된 파이프라인은 외부 호출이 시작되기도 전에 워커 스레드 풀 큐에서 대기할 수 있다; 그 큐 대기는 호출 자체와 마찬가지로 리스 시간을 소비한다. 리스는 클레임에서 tx2 커밋까지 큐잉 지연을 포함한 전체 벽시계 시간(wall-clock time)을 커버해야 한다.

6 취소: 유휴 파이프라인은 즉시, 실행 중인 것은 협력적으로

취소는 Admin/API 경로 (별도 프로세스, 자체 단기 트랜잭션)에서 발행된다. 적용 방식은 한 가지 사실에 따라 갈린다 — 지금 이 파이프라인을 실행 중인 워커가 있는가?

한 줄 요약

Case A (라이브 클레임 없음) — API 경로가 직접 터미널 status를 쓴다; Case B (라이브 클레임 있음) — API 경로는 cancel_requested만 올리고, 유일한 클레임 보유 워커가 CANCELLED를 적용한다. 어느 쪽도 워커의 tx2와 라이브 클레임 토큰을 공유하지 않으므로, status 가드가 필요 없다.

Case A — 파이프라인이 실행 중이 아님 (클레임 없음, 또는 리스 만료): 즉시 취소

이것이 흔한 "대기 중 / 아직 픽업 전" 파이프라인이다. API 경로가 워커 왕복 없이 자체 트랜잭션에서 직접 종단화한다:

-- A1: 파이프라인을 종단화하고 클레임을 지운다 (만료 리스의 낙오자 tx2가
--     claimed_by 가드에서 실패하도록); 라이브 워커가 행을 소유하지 않을 때만 발동
UPDATE pipeline SET status = 'CANCELLED', claimed_by = NULL, claimed_until = NULL
 WHERE id = :pid AND status = 'RUNNING'
   AND (claimed_by IS NULL OR claimed_until < now());
-- A2: 모든 비종단 태스크를 종단화 (A1이 행을 업데이트했을 때만 실행)
UPDATE task SET status = 'CANCELLED'
 WHERE pipeline_id = :pid AND status IN ('BLOCKED', 'READY', 'IN_PROGRESS');
Case B — 워커가 파이프라인을 실행 중 (라이브 리스): 협력적 취소

API 경로는 status를 쓸 수 없다 (워커가 유일한 status 쓰기 권한자이므로), 플래그만 올리고 파이프라인을 깨운다 — 이 경로는 Case A의 A1이 0행을 업데이트했을 때 발동한다:

UPDATE pipeline SET cancel_requested = true, next_due_at = now()
 WHERE id = :pid AND status = 'RUNNING';

클레임 보유 워커는 안전 지점 — 클레임 직후, 디스패치 전, 보고 트랜잭션(tx2) 내부 — 에서 cancel_requested를 읽고, 설정된 경우 모든 비종단 태스크 (현재 태스크 + 아직 BLOCKED인 후속 태스크)와 파이프라인을 CANCELLED로 종단화한 후 클레임을 해제한다. next_due_at = now()는 잠자던 파이프라인 (예: 긴 폴/조건 대기 중인 것)을 깨워 다음 스캔이 신속히 클레임하도록 한다.

왜 두 케이스 모두 레이스 안전한가:

  • 동시 클레임에 대해 — 둘 모두 파이프라인 행을 경쟁하므로, 먼저 커밋하는 쪽이 이긴다: 클레임은 Case A가 취소한 후 RUNNING을 더 이상 보지 않는다; Case A의 A1은 워커가 클레임한 후 자유 클레임을 더 이상 보지 않으므로 0행을 업데이트하고 Case B로 내려간다.
  • 만료 리스 낙오자에 대해 — Case A는 클레임이 null/만료 상태일 때만 발동하고 동일 구문에서 claimed_by/claimed_until을 지운다, 따라서 GC로 일시정지된 낙오자가 재개되면 토큰이 사라져 tx2의 claimed_by 가드가 no-op된다 — 부활 없음. 이것이 두 케이스 모두 status 가드가 필요 없는 이유이다.

수용된 지연 엣지 (Case B): (i) 워커 풀에 클레임됐지만 여전히 큐에 있는 작업은 tx2가 실행될 때만 플래그를 관측하므로, 취소 지연은 리스가 아닌 큐 대기 시간에 바운드된다; (ii) 워커의 최종 디스패치 전 플래그 확인 이후 도착하는 취소는 여전히 하나의 외부 호출이 발동되도록 허용한다 — 멱등성으로 무해하며, tx2에서 CANCELLED로 기록된다. 이미 디스패치된 job은 완료되도록 내버려 둔다 (멱등 인프라); InfraManager 측 취소는 정확성에 필요하지 않은 별도 후속 조치이다.

안전 메커니즘 및 조정 파라미터 (아키텍처 결정 아님)

정확한 값은 이 ADR이 아닌 운영 설정에 있다:

  • 워커 수 N은 동시 외부 호출을 (≤ min(N, 기한 파이프라인 수))로 캡한다. 429/503는 next_due_at을 앞으로 밀어 백오프한다.
  • 리스 기간 (lease_seconds)은 최대 단일 호출 타임아웃 + 풀 큐 대기 + 스케줄링 여유를 초과해야 한다 (결정 5 참조). 너무 짧은 리스의 효과 두 가지: (1) 보호된 쓰기가 스테일 낙오자의 DB 상태 클로버를 막고; (2) 멱등성 (ADR-016)이 InfraManager에 대한 중복 외부 호출을 무해하게 만든다. 어느 쪽도 데이터 손상을 유발하지 않지만, 중복 InfraManager 호출은 할당량을 소비한다.
  • 클레임 조건 인덱스 — 클레임 조건 (status='RUNNING' AND next_due_at <= now() AND (claimed_until IS NULL OR claimed_until < now()))에는 지원 인덱스가 필요하다; 없으면 모든 클레임이 동시 다중 워커 폴링 하에서 전체 순차 스캔 + 정렬로 저하된다. (next_due_at) WHERE status = 'RUNNING'의 부분 btree 인덱스가 핫 경로를 커버한다.
  • Terraform-job 동시성 캡 (slotCap) — 소프트 게이트 동작은 결정 7에 정의되며; 하드 캡 적용 (counter-CAS)은 연기됨. InfraManager의 고정 풀이 실질적 상한이며; 과잉 제출은 멱등 큐만 깊게 만든다.
  • DB 폴링 부하 제어 — 워커가 자가 폴링하므로, 조용한 시간대에 DB를 과다 요청하지 않도록 클레임 루프에 명시적 부하 제어가 필요하다: 빈 클레임 결과 후 즉시 재시도 없음; 지터(jitter) 포함 적응형 백오프 (예: 200 ms → 500 ms → 1 s → 2–5 s, 작업 발견 시 초기화); 가장 가까운 기한 파이프라인 시간이 알려진 경우 sleep = min(nextDueAt − now(), maxIdleSleep)으로 과잉 폴링 방지; 워커 간 동기화 웨이크업 방지를 위해 수면에 지터 분산. (클레임 조건 부분 인덱스가 이미 스캔당 비용을 제한하며; 이것은 그를 보완한다.)

7 어드미션 제어 및 TF 슬롯 게이트 (소프트 캡)

runningPipelineCap은 소프트 픽업 목표이지 하드 불변 조건이 아니며, 생성이 아닌 클레임을 게이트한다. 파이프라인 생성은 타겟별 유일성만 강제한다 (ADR-016) — 생성된 파이프라인은 즉시 RUNNING에 진입한다. 캡은 동시에 클레임된 작업 중인 RUNNING 파이프라인 수를 제한한다: 캡을 초과하여 클레임하기 전에 워커가 활성 클레임된 파이프라인 수를 확인한다. 캡 초과 파이프라인은 단순히 RUNNING에서 클레임되지 않은 채로 있다가 슬롯이 비면 next_due_at 순서로 나중에 픽업된다 — QUEUED나 WAITING_SLOT 상태가 없다. 확인이 count-read이므로 초과될 수 있다: 캡 M과 C개의 동시 클레임 워커에서 최악 동시 클레임은 M + C − 1. 이 유계 초과는 V1에서 수용된다. 하드 캡은 원자적 어드미션이 필요하다 (예: UPDATE pipeline_admission_counter SET used = used + 1 WHERE used < cap, 종단 시 해제) — 연기됨.

TF 슬롯 게이트. TF 슬롯은 Terraform-job 실행 점유 (수 분), API 호출 동시성 (수 밀리초~수 초)과 다른 자원이다. slotCap은 InfraManager/Terraform 압박을 완화하기 위해 TF 디스패치를 늦추는 소프트 어드미션 목표이다. 워커 루프의 소프트 게이트 동작: TF 디스패치 전에 워커가 슬롯 가용성을 확인; 가용하면 디스패치; 아니면 태스크를 READY로 유지하고 next_due_at = now() + slotRetry로 설정하고 클레임을 해제 (워커가 다른 파이프라인을 픽업할 수 있도록). 유계 초과는 runningPipelineCap과 동일한 이유로 수용된다; 여기서 경쟁 행위자 C는 파드가 아닌 슬롯을 확인하는 동시 워커들 (totalWorkerCount로 바운드)이다. 하드 캡은 tf_slot_counter CAS 또는 InfraManager 측 어드미션 제한을 사용할 것이다 — 연기됨.

파드 수는 정확성 파라미터가 아니다. 클레임 메커니즘은 구조적으로 다중 파드 안전이다 — DB 클레임이 프로세스 수가 아닌 조정 기본 단위 — 이 모델은 몇 개의 파드가 실행되는지에 무관하다: 하나 또는 여러 개, 정확성은 변하지 않으며 replicas 제약도 없다. 유일한 파드 수 영향은 수용된 소프트 캡 초과에 있다: totalWorkerCount = activePodCount × workerPerPod이고, 초과는 더 많은 파드가 올릴 수 있는 동시 어드미션 행위자 수 C와 함께 스케일한다. 그 초과가 수용되므로 파드 수 거버넌스가 불필요하다.

워커 루프 의사코드 (설명용; 슬롯 게이트 단계 포함):

loop:
  row = claim_one_due_pipeline()          // tx1: SKIP LOCKED + lease stamp
  if row is None:
    sleep(backoff_with_jitter())
    continue

  if row.cancel_requested:
    cancel_pipeline(row)                  // tx2: set CANCELLED, release claim
    continue

  if is_tf_dispatch_step(row) and not slot_available():
    reschedule(row, delay=slotRetry)      // tx2: release claim, push next_due_at
    continue

  result = execute_step(row)              // external call, outside any transaction

  report_result(row, result)              // tx2: guarded write-back, release claim

검토한 대안

옵션 판정 이유
A. 다중 워커 클레임-풀 (SKIP LOCKED + 리스 + 보호된 쓰기) 채택 구조적으로 다중 프로세스 안전; HA + 수평 스케일 내재; 취소-vs-워커 및 배포 중복 레이스 생존; 리스 만료를 통한 크래시 복구.
B. 단일 서버 (replicas=1) + 인메모리 in-flight 가드 기각 두 번째 쓰기 권한자가 생기는 순간 실패: 취소가 워커와 경쟁하여 터미널 부활을 일으키고, 프로세스 중복 (롤링 배포/스케일/재시작)이 이중 상태 전환을 유발한다. 멱등성은 이중 디스패치는 커버하지만 상태 전환 레이스는 커버하지 못한다.
C. 워크플로우 엔진 (Temporal / Airflow / 브로커) 기각 분 단위 폴의 2–4단계 선형 체인이 운영 비용을 정당화하지 못한다.
D. 인메모리 비동기 체인 (스캔 없음) 기각 재시작/배포 시 실행이 소실됨; 수 일의 대기를 내구적으로 표현할 수 없다. ADR-016이 이미 DB를 유일한 상태로 요구한다.

결과

장점
  • 구조적으로 다중 프로세스 안전: DB 클레임이 조정 기본 단위이지, 프로세스 수가 아니다.
  • HA 및 수평 스케일 — 운영하거나 디버깅할 리더 없음.
  • 취소 안전 (두 케이스, status 가드 없음): 라이브 리스 파이프라인의 경우 클레임 보유 워커가 유일한 status 쓰기 권한자로 cancel_requested 플래그를 통해 CANCELLED를 적용한다 (Case B); 유휴 파이프라인의 경우 API 경로가 터미널 status를 직접 쓰지만, 클레임 null/만료 조건으로 파이프라인 행 경쟁에서 이기고 지운 후에만 한다 (Case A). 어느 쪽도 워커의 tx2와 라이브 클레임 토큰을 공유하지 않으므로 부활이 불가능하다.
  • 크래시 복구는 리스 만료를 통해 자동 — 복구 저널 없음, 수동 단계 없음.
감수할 비용
  • 리스 튜닝: lease_seconds는 최대 호출 시간을 초과해야 한다; 너무 짧은 리스는 중복 (안전한) 작업을 유발하고; 너무 긴 리스는 크래시 후 복구를 지연시킨다.
  • 두 트랜잭션 분리는 단일 인프로세스 쓰기보다 움직이는 부품이 많다: 클레임 tx, 외부 호출, 보고 tx를 모두 별도로 추론해야 한다.
  • 리스 만료 창: 크래시된 워커의 파이프라인은 최대 하나의 리스 기간 동안 일시 중단된다.
  • 소프트 어드미션 목표 — runningPipelineCap과 slotCap 모두 소프트 게이트이다; 동시 어드미션 읽기가 최대 C − 1까지 초과할 수 있다 (여기서 C = 동시 어드미션 행위자 수: runningPipelineCap에는 어드미션 트랜잭션, 슬롯 게이트에는 워커). 하드 캡은 명시적 counter-CAS 어드미션이 필요하다 — 연기됨.

운영 참고 (파라미터 & 메트릭)

이 섹션은 참고 카탈로그이다. 이들은 정적 서버 설정 (예: Spring application.yaml) — API로 변경 가능한 런타임 파라미터가 아니다 — 이며 실제 값은 운영 설정에 있다. 특히 리스와 동시성 값은 설계 시점에 고정되지 않고 관측된 InfraManager 지연 시간에 맞게 운영 중에 튜닝된다.

파라미터 (Knobs)

파라미터 의미

필수 관계: leaseDuration > maxApiCallTimeout + poolQueueWait + safetyMargin (결정 5의 바운드와 동일).

주요 메트릭

  • active workers — 현재 단계를 실행 중인 워커 스레드
  • total worker count — activePodCount × workerPerPod
  • concurrent API calls — InfraManager / 조건 확인에 대한 in-flight 외부 호출
  • RUNNING pipeline count — 현재 status = 'RUNNING' 행 수
  • pipeline-cap overshoot count — runningPipelineCap 초과 어드미션
  • empty-claim rate — 행을 반환하지 않은 클레임 폴의 비율
  • claim QPS — 클레임 폴 처리량
  • claim latency — tx1 클레임 쿼리의 p50/p99 지속 시간
  • due-pipeline lag — 클레임되지 않은 기한 행에 대해 now() − min(next_due_at)
  • stale-report discard count — claimed_by 가드 실패로 tx2 no-op된 횟수
  • lease-expired reclaim count — 리스 타임아웃 후 재클레임된 파이프라인
  • slot-full retry count — 슬롯 게이트로 연기된 TF 디스패치
  • TF-slot overshoot count — slotCap 초과 TF 디스패치
  • 429/503 count — InfraManager의 백프레셔 응답
  • API latency — 외부 호출 지속 시간의 p50/p99
  • DB lock wait — 클레임 스캔 중 행 잠금 대기 시간
  • pipeline completion latency — 파이프라인 생성에서 터미널 상태까지의 벽시계 시간

링크

  • ADR-016 — 이 모델이 구동하는 내구성 도메인 모델

용어

  • 클레임 (Claim) — 클레임별 펜싱 토큰 (클레임 시 발행되는 새 UUID)을 claimed_by에, 데드라인을 claimed_until에 스탬프하는 짧은 커밋 트랜잭션(tx1)의 행위. 각 클레임은 새로운 유일 토큰을 생성하며; 리스 만료 후 같은 워커가 동일 파이프라인을 재클레임하면 다른 토큰을 받아, 이전 클레임의 in-flight tx2가 소유권 가드에서 거부된다. 클레임 시 워커는 진행 전 cancel_requested도 확인한다.
  • 협력적 취소 (Cooperative cancel, Case B) — 라이브 리스 하의 파이프라인의 경우, Admin/API 경로는 cancel_requested = true만 쓰고 (next_due_at = now()로 잠자는 파이프라인을 깨움); 클레임 보유 워커가 안전 지점에서 플래그를 읽고 터미널 CANCELLED 전환을 직접 적용하여, 해당 라이브 리스 파이프라인에서 유일한 status 쓰기 권한자로 남는다. (유휴 파이프라인은 대신 API 경로에서 즉시 종단화 — 결정 6의 Case A.) 어느 케이스도 터미널 부활을 막기 위한 쓰기별 status 가드가 필요 없다.
  • 기한 파이프라인 (Due pipeline) — next_due_at <= now()이고 리스가 만료됐거나 (혹은 한 번도 설정되지 않은) 파이프라인.
  • 소유권 보호 쓰기 (Guarded write-back) — 소유권 가드가 실패하면 no-op되는 UPDATE ... WHERE id = :id AND claimed_by = :token. 리스 만료 낙오자 클로버를 방어한다; 취소가 협력적이고 status가 단일 쓰기 권한자를 가지므로 (결정 6) status 가드가 필요 없다.
  • 리스 (Lease) — claimed_until 타임스탬프; 만료가 모든 워커의 재클레임을 위해 자동으로 클레임을 해제한다.
  • 두 트랜잭션 분리 (Two-transaction split) — tx1 (클레임)과 tx2 (보고)는 별도의 커밋 트랜잭션이며; 외부 호출은 그 사이에서 어떤 트랜잭션 밖에서 실행된다.

개정 이력

  • 2026-06-27: ADR-016을 분할하여 생성; 도메인 모델과 독립적으로 대체될 수 있도록 실행 모델을 여기로 추출.
  • 2026-06-27: 단일 쓰기 권한 전제가 동시 취소 및 다중 세션 하에서 깨지는 것으로 밝혀진 후 단일 서버/인메모리 모델을 다중 워커 클레임-풀로 교체.
