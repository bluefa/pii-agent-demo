ADR-021: Install/Delete Pipeline — 클레임-풀 실행 모델

제안됨
제안됨 — 2026-06-27

이것은 install/delete 파이프라인 설계의 실행 절반: ADR-016의 내구성 상태 머신이 실제로 어떻게 전진 구동되는지를 다룬다. 도메인 모델(테이블, 상태, 유일성, 라이프사이클, 멱등성 계약)은 ADR-016에 있으며 이 결정이 변경되어도 달라지지 않는다. 런타임 모델을 별도의 ADR에 두면, 미래에 변경이 생길 때 오직 이 ADR만 대체되고 도메인 모델은 그대로 유지된다.

배경

ADR-016은 데이터베이스 행이 파이프라인 상태 그 자체임을 확립하고, 모든 디스패치가 멱등적임을(따라서 at-least-once 전달이 정확함을) 보장한다. 결정하지 않은 것은 런타임: 오케스트레이터가 어디서 실행되는지, 인스턴스가 몇 개인지, 기한이 된 작업을 어떻게 찾는지, 동시 외부 호출을 어떻게 제한하는지이다.

지배적인 런타임 제약은 무한한 외부 호출 지연이다. InfraManager의 "run" API는 비동기 — 짧은 호출이 job id를 반환하지만, 시작된 Terraform job은 수 분 동안 실행되며, 조건 확인은 수 초에서 수 일까지 폴링한다. 실행 모델은 이를 흡수해야 하고, 프로세스 재시작에서도 살아남아야 한다 (ADR-016은 상태가 소실되지 않고 진행만 일시 정지됨을 보장한다).

선택을 형성하는 두 가지 동시 쓰기 팩트:

  • 취소는 Admin/API 경로에서 발행됨 — 별도의 프로세스, 자체 짧은 트랜잭션으로 — 워커가 틱 중간에 있을 때 언제든지. "단일 프로세스만 유일한 쓰기자"를 가정하는 어떤 모델도 구조적으로 깨진다.
  • 프로세스 겹침은 일상적: 롤링 배포, 스케일 이벤트, 재시작은 둘 이상의 인스턴스가 동시에 실행되는 시간 창을 만든다. 인메모리 in-flight 가드는 프로세스 범위이며, 이러한 경계를 넘어 보호를 제공하지 않는다.

운영 사실:

  • InfraManager는 자체 고정 워커 풀을 갖는다 — 이것이 동시 Terraform job의 실질적 상한이다. 오케스트레이터는 V1에서 자체 하드 캡이 필요 없다.
  • InfraManager에 과다 제출은 무해하다 (멱등적; InfraManager의 큐를 깊게 할 뿐이다).
  • 규모가 작다: ~2,000 타겟, 분 단위 job.

1 워커가 DB에서 작업을 풀링; 단일 인스턴스 제약 없음, 리더 선출 없음

오케스트레이터는 자체 배포 가능한 서버로 실행된다. N개의 워커 스레드 — 선택적으로 여러 파드/레플리카에 걸쳐 — 가 각각 데이터베이스에서 기한이 된 파이프라인을 풀링한다. 레플리카 수 제약 없음, 인메모리 in-flight 가드 없음, 리더 선출 없음. 수평 확장과 HA가 설계에 내재되어 있다.

2 FOR UPDATE SKIP LOCKED + 리스를 통한 파이프라인 레벨 클레임

워커는 외부 시스템에 접근하기 전, 짧은 전용 트랜잭션(tx1)을 실행하여 기한이 된 파이프라인 하나를 클레임한다:

BEGIN;
SELECT id FROM pipeline
 WHERE status = 'RUNNING'
   AND next_due_at <= now()
   AND (claimed_until IS NULL OR claimed_until < now())
 ORDER BY next_due_at
 LIMIT 1
 FOR UPDATE SKIP LOCKED;
UPDATE pipeline
   SET claimed_by = :claim_token,
       claimed_until = now() + (:lease_seconds * interval '1 second')
 WHERE id = :pipeline_id;
COMMIT;

SKIP LOCKED는 동일한 스캔에서 경쟁하는 두 워커가 서로를 블로킹하지 않고 서로 다른 파이프라인을 클레임하도록 보장한다. claimed_by / claimed_until 스탬프 — 프로세스 카운트가 아닌 — 가 프로세스와 파드 전반에 걸쳐 파이프라인이 한 번에 하나의 워커에 의해 소유됨을 보장한다.

claimed_by는 클레임별 펜싱 토큰이다 — 클레임 시점에 새로 발급된 UUID(위 SQL의 :claim_token)이며, 재사용되는 파드 id나 스레드 이름이 아니다. 이것은 스테일-스트래글러 거부에 중요하다: 워커의 리스가 만료되고 동일한 파드가 나중에 동일한 파이프라인을 재클레임하면, 다른 토큰을 받는다. 이전 클레임에서 진행 중인 tx2는 이전 토큰을 보유하므로 claimed_by = :claim_token 가드에서 no-op된다. 재사용되는 안정적인 식별자는 해당 가드를 통과하여 스테일 쓰기를 허용한다.

ORDER BY next_due_at + SKIP LOCKED는 엄격한 순서가 아닌 근사/공정한 FIFO를 제공한다. 다른 워커가 보유한 행은 건너뛰므로, 서로 다른 동시 워커가 다른 상대적 순서를 관찰할 수 있다. 파이프라인은 서로 독립적이므로 이는 허용된다.

빈 클레임 결과가 백로그가 비어 있음을 의미하지 않는다. SKIP LOCKED는 다른 워커가 리스를 보유한 행을 조용히 제외한다. 스캔 루프는 결과가 없는 것을 유휴 신호로 처리해서는 안 되며, 정상적으로 다음 폴링 이터레이션을 진행해야 한다.

실행 스키마 참고. 이 모델은 pipeline 테이블에 네 개의 실행 조정 컬럼을 추가한다 — next_due_at, claimed_by, claimed_until, 그리고 cancel_requested(불리언) — 이것들은 이 ADR이 소유하는 실행 메타데이터이며, ADR-016의 도메인 상태 컬럼과 구별된다.

3 두 트랜잭션 분리

tx1: claim     →     external call (OUTSIDE any transaction)     →     tx2: report

외부 호출(InfraManager, 조건 확인)은 200 ms–60 s가 소요된다. 그 시간 동안 행 락을 유지하면 동일한 행을 스캔하려는 다른 모든 워커를 블로킹한다. 두 트랜잭션 분리는 이를 방지한다: tx1이 파이프라인을 클레임하고 즉시 커밋하며; 외부 호출이 락 없이 실행되고; tx2가 결과를 커밋한다.

4 가드된 write-back (소유권 가드)

한 줄 요약

report(tx2)는 워커가 여전히 클레임을 소유할 때만 (WHERE claimed_by = :claim_token) 반영된다; 그렇지 않으면 no-op된다. 이 단일 소유권 가드 — 그리고 취소 케이스별 status 단일 작성자 규칙 — 가 write-back 안전성의 전부이므로, status 가드가 필요 없다.

쓰기. report 트랜잭션(tx2)은 다음을 통해 태스크와 파이프라인을 전환한다:

UPDATE pipeline
   SET status = :new_status, ...
 WHERE id = :pipeline_id
   AND claimed_by = :claim_token;

tx2는 단일 트랜잭션 내에서 고정된 순서로 실행된다: (1) 파이프라인 행을 SELECT ... FOR UPDATE; (2) claimed_by = :claim_token 검증 — 실패하면 report 전체를 no-op; (3) cancel_requested 읽기; (4) 태스크와 파이프라인 status 쓰기 — 플래그가 설정된 경우 파이프라인과 모든 비종단 태스크 (현재 태스크 및 아직 BLOCKED 상태인 후속 태스크들)를 ADR-016 §7에 따라 CANCELLED로 구동; 그렇지 않으면 정상적인 현재 태스크 전환 적용; (5) 다음 next_due_at 쓰기 및 클레임 해제.

소유권은 파이프라인 레벨에서 한 번 검증된다. claimed_by 확인은 락된 파이프라인 행에서 이루어지며; 태스크와 파이프라인 쓰기는 그 검증된 단일 작성자 클레임 아래에서 이루어지므로, 행별 클레임 가드가 필요 없다 — task 테이블은 claimed_by 컬럼을 갖지 않는다.

성공 시 tx2는 클레임을 해제하고 next_due_at을 원자적으로 전진시킨다. 태스크/파이프라인 상태를 전진시키는 동일한 트랜잭션이 claimed_by/claimed_until을 제거하고 새로운 next_due_at을 쓴다 (트리거 엔드포인트가 생성 시 시드). 이 해제 없이는 스텝을 완료했지만 RUNNING 상태를 유지하는 파이프라인이 claimed_until이 지날 때까지 잠긴 채 다른 워커를 블로킹한다. 현재 태스크가 DONE에 도달하면, 동일한 tx2가 다음 태스크를 BLOCKED → READY로 전환한다. 디스패치 시 워커는 task_attempt 행을 기록하고(시도의 job_ids 집합 + 응답), 각 폴링 시 해당 시도의 task_check 요약(카운트 + 마지막 응답)을 제자리 UPDATE한다; 모두 검증된 파이프라인 클레임 아래에서(단일 작성자, 태스크 레벨 클레임 불필요). 태스크 완료는 최신 시도 행에 대한 코드 레벨 check(attempt, task)다 — 리컨사일러는 오직 완료를 위해 그 행만 읽으며, 클레임·스케줄링·파이프라인 전환은 시도 테이블을 읽지 않는다. 소실된 시도 결과는 태스크를 멈추지 않는다: executionTimeout으로 넘어가고 신선한 N-job 실행을 재디스패치한다 (멱등적).

위의 내용은 메커니즘이다; 이하는 왜 레이스 없는지에 대한 설명이다.

레이스 안전성 분석

왜 status 가드가 필요 없는가. 단독 claimed_by 가드가 모든 스트래글러를 막는다: 리스가 만료된 후 재개된 워커는 claimed_by가 더 이상 일치하지 않음을 발견하고 update가 no-op된다 (덮어쓰기 없음). status = :expected_status 가드는 아무것도 추가하지 않는다. 왜냐하면 행이 종단화되는 동안 일치하는 라이브 토큰을 보유하는 다른 작성자가 절대 없기 때문이다 — 취소는 즉시 경로(결정 6)에서만 status를 쓰며, 이는 클레임이 null/만료 상태일 때만 발생하고 종단화 시 claimed_by를 제거한다; 라이브 리스 파이프라인에 대해 취소는 플래그만 설정하고 클레임 보유자가 유일한 status 작성자다. tx2 쓰기가 취하는 동일한 행 락 아래에서 cancel_requested를 읽는 것(3단계)이 협력적 취소를 레이스 없이 만드는 요소다 — 3단계 직전에 커밋된 취소는 관찰되고; 직후에 커밋된 취소는 tx2가 커밋될 때까지 블로킹된 후 다음 클레임에 적용된다.

왜 중복 외부 호출이 무해한가. 두 트랜잭션 분리는 InfraManager가 두 번 호출될 수 있는 세 가지 경계 시간 창을 만든다:

  • (a) 클라이언트 타임아웃 — 호출자가 타임아웃되지만 InfraManager는 이미 job을 수락한 경우.
  • (b) 디스패치 후 크래시 — 워커가 디스패치하고 tx2가 반환된 job_ids를 기록하기 전에 크래시하는 경우.
  • (c) 멈춤 중 리스 만료 — 워커 스레드 풀 큐 대기 또는 GC 일시 중지가 리스를 초과; 다른 워커가 재클레임하고 동일한 파이프라인을 디스패치하는 경우.

세 경우 모두 중복 외부 호출은 인프라 멱등성으로 안전하다 — TF API는 중복에 무해하다 (재디스패치로 무해한 중복 job이 생성될 수 있으며; 시도를 위해 기록된 job_ids는 최신 디스패치의 집합이다). 가드된 DB 쓰기는 다른 문제를 해결한다: 스테일 스트래글러가 DB 상태를 덮어쓰는 것을 방지. 둘은 상호 보완적이며 대체 가능하지 않다 — 멱등성은 외부 중복을 커버하고, 가드는 DB 쓰기를 커버한다.

5 리스 만료를 통한 크래시 복구

크래시된 워커가 클레임한 파이프라인은 claimed_until < now()가 되면 다시 기한이 된다. 리더 없음, 수동 개입 없음 — 다음 스캔이 재클레임한다. 하드 제약:

lease_seconds > max_single_call_timeout + pool_queue_wait + scheduling_margin

클레임된 파이프라인은 외부 호출이 시작되기 전 워커 스레드 풀 큐에서 대기할 수 있다; 그 큐 대기는 호출 자체만큼 리스 시간을 소모한다. 리스는 클레임에서 tx2 커밋까지의 전체 경과 벽시계 시간 — 큐잉 지연 포함 — 을 커버해야 한다.

6 취소: 유휴 파이프라인은 즉시, 실행 중인 경우 협력적으로

취소는 Admin/API 경로(별도의 프로세스, 자체 짧은 트랜잭션)에서 발행된다. 적용 방식은 하나의 사실로 나뉜다 — 지금 이 파이프라인을 실행 중인 워커가 있는가?

Case A — 파이프라인이 실행 중이 아닌 경우 (클레임 없음 또는 리스 만료): 즉시 취소

흔한 "대기 중 / 아직 픽업되지 않은" 파이프라인의 경우다. API 경로가 워커 라운드트립 없이 자체 트랜잭션에서 직접 종단화한다:

-- A1: 파이프라인 종단화 AND 클레임 제거 (만료된 리스 스트래글러의 tx2가
--     claimed_by 가드에서 실패하도록); 라이브 워커가 없을 때만 실행
UPDATE pipeline SET status = 'CANCELLED', claimed_by = NULL, claimed_until = NULL
 WHERE id = :pid AND status = 'RUNNING'
   AND (claimed_by IS NULL OR claimed_until < now());
-- A2: 모든 비종단 태스크 종단화 (A1이 행을 업데이트한 경우에만 실행)
UPDATE task SET status = 'CANCELLED'
 WHERE pipeline_id = :pid AND status IN ('BLOCKED', 'READY', 'IN_PROGRESS');
Case B — 워커가 파이프라인을 실행 중인 경우 (라이브 리스): 협력적 취소

API 경로는 status를 쓸 수 없으므로 (워커가 유일한 status 작성자), 플래그만 올리고 파이프라인을 깨운다 — 이 경로는 Case A의 A1이 0 행을 업데이트했을 때 실행된다:

UPDATE pipeline SET cancel_requested = true, next_due_at = now()
 WHERE id = :pid AND status = 'RUNNING';

클레임을 보유한 워커는 안전 포인트 — 클레임 직후, 디스패치 전, report 트랜잭션(tx2) 내부 — 에서 cancel_requested를 읽고, 설정되어 있으면 모든 비종단 태스크 (현재 태스크 및 아직 BLOCKED 상태인 후속 태스크들)와 파이프라인을 CANCELLED로 종단화하고 클레임을 해제한다. next_due_at = now()는 잠자고 있는 파이프라인 (예: 긴 폴링/조건 대기 중인)을 깨워서 다음 스캔이 신속하게 클레임하도록 한다.

두 케이스가 모두 레이스 없는 이유:

  • 동시 클레임에 대해 — 둘 다 파이프라인 행에서 경쟁하므로, 먼저 커밋된 쪽이 이긴다: Case A가 취소한 후에는 클레임이 더 이상 RUNNING을 보지 못하고; Case A의 A1은 워커가 클레임한 후에는 빈 클레임을 보지 못한다 (0 행을 업데이트하고 Case B로 넘어간다).
  • 만료된 리스 스트래글러에 대해 — Case A는 클레임이 null/만료일 때만 실행되고 동일한 구문에서 claimed_by/claimed_until을 제거하므로, GC로 일시 중지된 스트래글러가 재개되면 토큰이 사라지고 tx2의 claimed_by 가드가 no-op된다 — 부활 없음. 이 때문에 어느 케이스도 status 가드가 필요 없다.

허용된 지연 엣지 (Case B): (i) 워커 풀에서 클레임됐지만 여전히 큐에 있는 작업은 tx2가 실행될 때만 플래그를 관찰하므로, 취소 지연은 큐 대기 — 호출별 타임아웃이 아닌 리스에 의해 경계됨; (ii) 워커의 마지막 디스패치 전 플래그 확인 이후 도착한 취소는 여전히 하나의 외부 호출을 발생시킬 수 있다 — 멱등성에 의해 무해하며, tx2에서 CANCELLED로 기록된다. 이미 디스패치된 job은 완료되도록 둔다 (멱등 인프라); InfraManager 측 취소는 별도의 후속 조치이며, 정확성에는 필요 없다.

취소가 동일한 행 경쟁 아래에서만 status를 쓰므로 (Case A) 또는 유일한 작성자인 워커를 통해서만 (Case B), 그리고 워커의 tx2와 라이브 클레임 토큰을 공유하지 않으므로, 종단 상태 부활을 방지하기 위한 쓰기별 status 가드가 필요 없다.

안전 메커니즘 & 튜닝 파라미터 (아키텍처 결정이 아님)

정확한 값은 이 ADR이 아닌 운영 설정에 있다:

  • 워커 수 N은 동시 외부 호출을 상한 제한한다 (≤ min(N, due pipelines)). 초당 요청 수 보장이 아니다 — 429/503은 next_due_at을 앞으로 밀어 백오프한다.
  • 리스 기간 (lease_seconds)은 최대 단일 호출 타임아웃 + 풀 큐 대기 + 스케줄링 마진을 초과해야 한다 (결정 5 참고). 보수적으로 튜닝하라; 너무 짧은 리스는 두 가지 별개의 효과를 갖는다: (1) 가드된 쓰기가 스테일 스트래글러의 DB 상태 덮어쓰기를 방지; (2) 멱등성(ADR-016)이 InfraManager에 대한 중복 외부 호출을 무해하게 한다. 어느 쪽도 손상을 유발하지 않지만, 불필요한 InfraManager 호출은 쿼터를 소비한다.
  • 클레임 조건 인덱스 — 클레임 조건 (status='RUNNING' AND next_due_at <= now() AND (claimed_until IS NULL OR claimed_until < now()))은 지원 인덱스가 필요하다; 없으면 모든 클레임이 동시 멀티워커 폴링 시 전체 순차 스캔 + 정렬로 저하된다. (next_due_at) WHERE status = 'RUNNING'에 대한 부분 btree 인덱스가 핫 패스를 커버한다.
  • Terraform job 동시성 캡 (slotCap) — 소프트 게이트 동작은 결정 7에 정의되어 있다; 하드캡 강제(카운터-CAS)는 연기됨. InfraManager의 고정 풀이 실질적 상한이며; 과다 제출은 멱등 큐를 깊게 할 뿐이다.
  • DB 폴링 부하 제어 — 워커가 자체 폴링하므로, 클레임 루프는 조용한 기간에 DB를 폭격하지 않도록 명시적 부하 제어가 필요하다: 빈 클레임 결과 후 즉시 재시도 금지; 지터를 포함한 적응형 백오프 (예: 200 ms → 500 ms → 1 s → 2–5 s, 작업 발견 시 초기화); 가장 가까운 기한 파이프라인 시간이 알려진 경우, 과다 폴링을 피하기 위해 sleep = min(nextDueAt − now(), maxIdleSleep); 워커 전반에서 동기화된 웨이크업을 방지하기 위한 수면 전반에 걸친 지터. (클레임 조건 부분 인덱스가 이미 스캔당 비용을 제한하며; 이것은 그것을 보완한다.)

7 어드미션 제어와 TF 슬롯 게이트 (소프트 캡)

runningPipelineCap은 하드 불변 조건이 아닌 소프트 픽업 타겟이며, 생성이 아닌 클레임을 게이팅한다. 파이프라인 생성은 타겟별 유일성만 강제한다 (ADR-016) — 생성된 파이프라인은 즉시 RUNNING에 진입한다. 캡은 얼마나 많은 RUNNING 파이프라인이 작업을 위해 동시에 클레임되는지를 제한한다: 캡을 초과하여 클레임하기 전, 워커가 활성 클레임된 파이프라인 수를 확인한다. 캡을 초과한 파이프라인은 단순히 RUNNING 상태로 미클레임 상태를 유지하며 슬롯이 해제됨에 따라 next_due_at 순서로 나중에 픽업된다 — QUEUED 또는 WAITING_SLOT 상태가 없다. 확인은 카운트 읽기이므로 오버슈트가 가능하다: 캡 M과 C개의 동시 클레임 워커에 대해, 최악의 경우 동시 클레임은 M + C − 1이다. 이 경계된 오버슈트는 V1에서 허용된다. 하드 캡은 원자적 어드미션이 필요하다 (예: UPDATE pipeline_admission_counter SET used = used + 1 WHERE used < cap, 종단 시 해제) — 연기됨.

TF 슬롯 게이트. TF 슬롯은 Terraform job 실행 점유율(분 단위)이며, API 호출 동시성(밀리초~초 단위)과 다른 리소스다. slotCap은 InfraManager/Terraform 압력을 완화하기 위해 TF 디스패치를 늦추는 소프트 어드미션 타겟이다. 워커 루프의 소프트 게이트 동작: TF 디스패치 전, 워커가 슬롯 가용성을 확인; 가용하면 디스패치; 그렇지 않으면 태스크를 READY로 유지하고, next_due_at = now() + slotRetry를 설정하며 클레임을 해제 (워커가 스스로를 해제하여 다른 파이프라인을 픽업할 수 있음). 동일한 이유로 오버슈트가 허용된다; 여기서 경쟁 주체 C는 슬롯을 확인하는 동시 워커 (총 워커 수로 경계됨)이며, 파드가 아니다. 하드 캡은 tf_slot_counter CAS 또는 InfraManager 측 어드미션 제한이 필요하다 — 연기됨.

파드 수는 정확성 파라미터가 아니다. 클레임 메커니즘은 구조적으로 멀티파드 안전하다 — DB 클레임이 프로세스 카운트가 아닌 조정 기본 요소이므로 — 이 모델은 실행 파드 수에 무관하다: 하나든 여럿이든, 정확성은 변하지 않으며 어느 방향으로도 replicas 제약이 없다. 파드 수의 유일한 효과는 허용된 소프트 캡 오버슈트에 있다: totalWorkerCount = activePodCount × workerPerPod이며, 오버슈트는 더 많은 파드가 올릴 수 있는 동시 어드미션 주체 C 수에 따라 확대된다. 그 오버슈트는 허용되므로 (결정 7), 파드 수는 여기서 거버넌스가 필요 없다.

워커 루프 의사코드 (참고용; 슬롯 게이트 단계 포함):

loop:
  row = claim_one_due_pipeline()          // tx1: SKIP LOCKED + lease stamp
  if row is None:
    sleep(backoff_with_jitter())
    continue

  if row.cancel_requested:
    cancel_pipeline(row)                  // tx2: set CANCELLED, release claim
    continue

  if is_tf_dispatch_step(row) and not slot_available():
    reschedule(row, delay=slotRetry)      // tx2: release claim, push next_due_at
    continue

  result = execute_step(row)              // external call, outside any transaction

  report_result(row, result)              // tx2: guarded write-back, release claim

검토한 대안

옵션 판정 이유
A. 멀티워커 클레임-풀 (SKIP LOCKED + 리스 + 가드된 쓰기) 채택 구조적으로 멀티프로세스 안전; HA + 수평 확장이 내재됨; 취소-vs-워커 및 배포-겹침 레이스에서 살아남음; 리스 만료를 통한 크래시 복구.
B. 단일 서버 (replicas=1) + 인메모리 in-flight 가드 기각 두 번째 작성자가 생기는 순간 실패한다: 취소가 워커와 레이스하여 종단 상태 부활을 일으키고, 모든 프로세스 겹침(롤링 배포/스케일/재시작)이 이중 status 전환을 유발한다. 멱등성은 이중 디스패치를 커버하지만 status 전환 레이스는 커버하지 않는다.
C. 워크플로우 엔진 (Temporal / Airflow / 브로커) 기각 분 단위 폴링의 2–4 스텝 선형 체인은 운영 비용을 정당화하지 않는다.
D. 인메모리 비동기 체인 (스캔 없음) 기각 재시작/배포 시 실행이 소실됨; 다일 대기를 내구적으로 표현할 수 없음. ADR-016은 이미 DB가 유일한 상태임을 요구한다.

결과

장점
  • 구조적으로 멀티프로세스 안전: DB 클레임이 프로세스 카운트가 아닌 조정 기본 요소다.
  • HA와 수평 확장이 운영하거나 디버그할 리더 없이 제공된다.
  • 취소 안전 (두 케이스, status 가드 없음): 라이브 리스 파이프라인의 경우 클레임 보유 워커가 유일한 status 작성자이며 cancel_requested 플래그를 통해 CANCELLED를 적용한다 (Case B); 유휴 파이프라인의 경우 API 경로가 종단 status를 직접 쓰지만, 클레임이 null/만료 상태에서 파이프라인 행 경쟁에서 이긴 후 이를 제거한 후에만 한다 (Case A). 어느 케이스도 워커의 tx2와 라이브 클레임 토큰을 공유하지 않으므로 부활이 불가능하다.
  • 크래시 복구는 리스 만료를 통해 자동 — 복구 저널 없음, 수동 단계 없음.
감수할 비용
  • 리스 튜닝: lease_seconds는 최대 호출 시간을 초과해야 한다; 너무 짧은 리스는 중복(안전한) 작업을 유발하고; 너무 긴 리스는 크래시 후 복구를 지연시킨다.
  • 두 트랜잭션 분리는 단일 인프로세스 쓰기보다 움직이는 부품이 많다: 클레임 tx, 외부 호출, report tx가 모두 별도로 추론되어야 한다.
  • 리스 만료 창: 크래시된 워커의 파이프라인은 최대 한 리스 기간 동안 재클레임 전까지 일시 중지된다.
  • 소프트 어드미션 타겟 — runningPipelineCap과 slotCap 모두 소프트 게이트이며; 동시 어드미션 읽기가 최대 C − 1만큼 오버슈트할 수 있다 (여기서 C = 동시 어드미션 주체 수: runningPipelineCap에 대한 어드미션 트랜잭션, 슬롯 게이트에 대한 워커). 하드 캡은 명시적 카운터-CAS 어드미션이 필요하다 — 연기됨.

운영 레퍼런스 (파라미터 & 지표)

이 섹션은 참고 카탈로그다. 이것들은 정적 서버 설정 (예: Spring application.yaml) — API 변경 가능한 런타임 파라미터가 아님 — 이며, 값은 이 ADR이 아닌 운영 설정에 있다. 특히 리스 및 동시성 값은 설계 시점에 고정되는 것이 아니라 관찰된 InfraManager 지연에 맞게 운영 중에 튜닝된다.

slotCap 동시 TF 디스패치 슬롯에 대한 소프트 어드미션 타겟
slotRetry 슬롯이 꽉 찼을 때 슬롯 가용성 재확인 전 지연
leaseDuration 클레임 리스 창; 아래 필요 관계 참고
apiCallTimeout InfraManager / 조건 확인 호출에 대한 호출별 타임아웃
maxIdleSleep 클레임 폴링 간 유휴 수면 상한
backoffBase 빈 클레임 결과 후 초기 백오프 간격
backoffMax 최대 백오프 간격 (지터 적용 전)
jitterRatio 수면 간격을 랜덤화할 비율 (예: ±20%)

필요 관계: leaseDuration > maxApiCallTimeout + poolQueueWait + safetyMargin (결정 5와 동일한 경계).

핵심 지표

  • active workers — 현재 스텝을 실행 중인 워커 스레드
  • total worker count — activePodCount × workerPerPod
  • concurrent API calls — InfraManager / 조건 확인에 대한 진행 중 외부 호출
  • RUNNING pipeline count — 현재 status = 'RUNNING' 행 수
  • pipeline-cap overshoot count — runningPipelineCap 초과 어드미션
  • empty-claim rate — 행을 반환하지 않은 클레임 폴링 비율
  • claim QPS — 클레임-폴링 처리량
  • claim latency — tx1 클레임 쿼리의 p50/p99 소요 시간
  • due-pipeline lag — 미클레임 기한 행에 대한 now() − min(next_due_at)
  • stale-report discard count — claimed_by 가드 실패로 no-op된 tx2 수
  • lease-expired reclaim count — 리스 타임아웃 후 재클레임된 파이프라인
  • slot-full retry count — 슬롯 게이트로 연기된 TF 디스패치
  • TF-slot overshoot count — slotCap 초과 TF 디스패치
  • 429/503 count — InfraManager의 백프레셔 응답
  • API latency — 외부 호출 소요 시간의 p50/p99
  • DB lock wait — 클레임 스캔 중 행 락 대기 시간
  • pipeline completion latency — 파이프라인 생성에서 종단 상태까지의 벽시계 시간

링크

  • ADR-016 — 이 실행 모델이 구동하는 내구성 도메인 모델

용어

  • 클레임 (Claim) — 클레임별 펜싱 토큰(클레임 시점에 발급된 신선한 UUID)을 claimed_by에, 데드라인을 claimed_until에, 짧은 커밋된 트랜잭션(tx1)에서 스탬핑하는 행위. 각 클레임은 새로운 고유 토큰을 생성하며; 동일한 워커가 리스 만료 후 동일한 파이프라인을 재클레임하면 다른 토큰을 받아, 이전 클레임에서 진행 중인 tx2가 소유권 가드에 의해 거부됨을 보장한다. 클레임 시 워커는 진행 전에 cancel_requested도 확인한다.
  • 협력적 취소 (Cooperative cancel, Case B) — 라이브 리스를 보유한 파이프라인의 경우, Admin/API 경로는 cancel_requested = true만 쓰고 (잠자고 있는 파이프라인을 깨우기 위해 next_due_at = now() 설정); 클레임 보유 워커가 안전 포인트에서 플래그를 읽고 스스로 종단 CANCELLED 전환을 적용하며, 해당 라이브 리스 파이프라인의 유일한 status 작성자로 유지된다. (유휴 파이프라인은 대신 API 경로가 즉시 종단화한다 — 결정 6의 Case A.) 어느 케이스도 종단 상태 부활을 방지하기 위한 쓰기별 status 가드가 필요 없다.
  • 기한 파이프라인 (Due pipeline) — next_due_at <= now()이고 리스가 만료됐거나(또는 설정된 적 없는) 파이프라인.
  • 가드된 write-back (Guarded write-back) — 소유권 가드가 실패하면 no-op되는 UPDATE ... WHERE id = :id AND claimed_by = :token. 리스 만료 스트래글러의 덮어쓰기를 방지한다; 취소가 협력적이고 status에 단일 작성자가 있으므로 status 가드가 필요 없다 (결정 6).
  • 리스 (Lease) — claimed_until 타임스탬프; 만료 시 자동으로 클레임을 해제하여 어느 워커든 재클레임할 수 있게 한다.
  • 두 트랜잭션 분리 (Two-transaction split) — tx1(클레임)과 tx2(report)는 별도의 커밋된 트랜잭션이며; 외부 호출은 그 사이에서, 어느 트랜잭션 밖에서 실행된다.

개정 이력

  • 2026-06-27: ADR-016을 분리하여 생성; 실행 모델을 여기에 추출하여 도메인 모델과 독립적으로 대체 가능하도록 함.
  • 2026-06-27: 취소와 멀티세션 하에서 단일 작성자 전제가 깨짐이 발견된 후, 단일 서버/인메모리 모델을 멀티워커 클레임-풀로 대체.
