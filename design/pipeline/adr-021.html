ADR-021: Install/Delete Pipeline — Claim-Pull 실행 모델

제안됨
제안됨 — 2026-06-27

install/delete 파이프라인 설계의 실행 절반이다: ADR-016의 내구성 상태 머신이 실제로 어떻게 전진 구동되는지를 다룬다. 도메인 모델 — 테이블, 상태, 유일성, lifecycle, 멱등성 계약 — 은 ADR-016에 있으며, 이 결정이 바뀌더라도 변하지 않는다. 실행 모델을 별도 ADR로 분리하면, 미래에 이 ADR만 대체하면서 도메인 모델은 그대로 유지할 수 있다.

배경

ADR-016은 데이터베이스 행이 곧 파이프라인의 상태이며, 모든 디스패치가 멱등적이라는 것을 확립한다 (따라서 at-least-once 전달이 올바르다). 그러나 런타임에 대해서는 결정하지 않는다: 오케스트레이터가 어디서 실행되는지, 인스턴스는 몇 개인지, 어떻게 처리해야 할 작업을 찾는지, 동시 외부 호출을 어떻게 제한하는지.

지배적인 런타임 제약은 무한할 수 있는 외부 호출 레이턴시다. InfraManager의 "run" API는 비동기이다 — 짧은 호출이 job id를 반환하지만, 그것이 시작하는 Terraform job은 수 분 동안 실행되며, 조건 확인은 수 초에서 수 일까지 폴링한다. 실행 모델은 이를 멈추지 않고 흡수해야 하며, 프로세스 재시작에서도 살아남아야 한다 (ADR-016은 상태가 소실되지 않음을 보장하며, 다만 진행이 일시 멈출 뿐이다).

설계 선택에 영향을 미치는 두 가지 동시 쓰기 사실:

  • 취소는 Admin/API 경로가 발행한다 — 별도 프로세스, 자체 단기 트랜잭션 — 워커가 tick 중간에 있을 때 언제든지. "단일 프로세스만 유일한 쓰기자"라는 가정을 하는 모델은 구조적으로 깨진다.
  • 프로세스 중복은 일상적이다: 롤링 배포, 스케일 이벤트, 재시작이 두 개 이상의 인스턴스가 동시에 살아 있는 구간을 만든다. 인메모리 in-flight 가드는 프로세스 단위이며, 이 경계를 가로질러 아무런 보호도 제공하지 않는다.

운영 사실:

  • InfraManager는 자체 고정 워커 풀을 가진다 — 이것이 동시 Terraform job의 실질적 상한이다. 오케스트레이터는 V1에서 자체 하드 캡이 필요하지 않다.
  • InfraManager에 과도하게 제출하는 것은 무해하다 (멱등적; 단지 큐를 깊게 만들 뿐이다).
  • 규모는 작다: ~2,000 타겟, 분 단위 job.

결정

1 워커는 DB에서 작업을 풀(pull)한다; 단일 인스턴스 제약 없음, 리더 선출 없음

오케스트레이터는 독자적인 배포 가능한 서버로 실행된다. N개의 워커 스레드 — 선택적으로 여러 pod/replica에 걸쳐 — 각각 데이터베이스에서 처리 대상 파이프라인을 풀(pull)한다. 레플리카 수 제약, 인메모리 in-flight 가드, 리더 선출이 없다. 수평 확장과 고가용성이 설계에 내재한다.

2 FOR UPDATE SKIP LOCKED + 리스를 통한 파이프라인 레벨 클레임

워커는 외부 시스템에 접촉하기 전에 짧고 전용적인 트랜잭션(tx1)을 실행하여 처리 대상 파이프라인 하나를 클레임한다:

BEGIN;
SELECT id FROM pipeline
 WHERE status = 'RUNNING'
   AND next_due_at <= now()
   AND (claimed_until IS NULL OR claimed_until < now())
 ORDER BY next_due_at
 LIMIT 1
 FOR UPDATE SKIP LOCKED;
UPDATE pipeline
   SET claimed_by = :claim_token,
       claimed_until = now() + (:lease_seconds * interval '1 second')
 WHERE id = :pipeline_id;
COMMIT;

SKIP LOCKED는 동일 스캔을 경쟁하는 두 워커가 서로 블로킹 없이 서로 다른 파이프라인을 클레임하게 보장한다. 프로세스 수가 아닌 claimed_by / claimed_until 스탬프가 — 프로세스와 pod를 가로질러 — 한 파이프라인이 한 워커에 의해 소유됨을 보장한다.

claimed_by는 클레임별 펜싱 토큰이다 — 각 클레임 순간에 새로 생성된 UUID (위 SQL의 :claim_token), 재사용된 pod id나 스레드 이름이 아니다. 이것은 오래된 스트래글러(stale straggler) 거부에 중요하다: 워커의 리스가 만료되고 동일 pod가 나중에 동일 파이프라인을 재클레임하면 다른 토큰을 받는다. 이전 클레임의 in-flight tx2는 구토큰을 가지므로 claimed_by = :claim_token 가드에서 no-op이 된다. 재사용된 안정적 식별자라면 해당 가드를 통과시켜 오래된 쓰기가 적용될 것이다.

ORDER BY next_due_at + SKIP LOCKED는 근사/공정한 FIFO를 산출하며, 엄격한 순서는 아니다. 현재 다른 워커가 보유 중인 행은 건너뛰므로 서로 다른 동시 워커는 상대적 순서를 다르게 관찰할 수 있다. 파이프라인들이 서로 독립적이므로 이는 허용된다.

빈 클레임 결과는 백로그가 비었다는 의미가 아니다. SKIP LOCKED는 다른 워커가 리스를 보유 중인 행을 조용히 생략한다. 스캔 루프는 zero-row 결과를 유휴 신호로 처리해서는 안 되며, 정상적으로 다음 폴 이터레이션을 진행해야 한다.

실행 스키마 노트

이 모델은 pipeline 테이블에 네 개의 실행 조정 컬럼을 추가한다 — next_due_at, claimed_by, claimed_until, cancel_requested (boolean) — 이것들은 이 ADR이 소유하는 실행 메타데이터이며, ADR-016의 도메인 상태 컬럼과 구별된다.

3 두 트랜잭션 분리 (two-transaction split)

tx1: 클레임     →     외부 호출 (트랜잭션 밖에서 실행)     →     tx2: 결과 기록

외부 호출 (InfraManager, 조건 확인)은 200 ms–60 s 동안 실행된다. 그 구간 동안 행 락을 유지하면 동일 행을 스캔하려는 다른 모든 워커가 블로킹된다. 두 트랜잭션 분리가 이를 방지한다: tx1이 파이프라인을 클레임하고 즉시 커밋; 외부 호출은 락 없이 실행; tx2가 결과를 커밋한다.

4 소유권 보호 쓰기백 (ownership-guarded write-back)

결과 기록 트랜잭션(tx2)은 다음과 같이 태스크와 파이프라인을 전환한다:

UPDATE pipeline
   SET status = :new_status, ...
 WHERE id = :pipeline_id
   AND claimed_by = :claim_token;

하나의 가드가 정확성을 지킨다:

  • claimed_by 가드 — 리스가 만료된 후 파이프라인이 다른 워커에 의해 재클레임된 뒤에 재개되는 오래된 스트래글러는 claimed_by가 더 이상 일치하지 않아 업데이트가 no-op이 된다. 덮어쓰기 없음.

취소 안전성을 위해 status = :expected_status 가드는 필요하지 않다. 취소는 즉시 경로 (결정 6 Case A)에서만 status를 쓰며, 그 경로는 클레임이 null이거나 만료된 경우에만 실행되고 종단화하면서 claimed_by를 지운다 — 따라서 워커의 tx2와 라이브 클레임 토큰을 공유할 수 없다. 라이브 리스 파이프라인의 경우 취소는 플래그만 설정하고 클레임 보유자가 유일한 status 쓰기자다. 어느 쪽이든 claimed_by 가드 하나가 모든 스트래글러를 차단하며, 행이 종단화되는 동안 일치하는 토큰을 가진 동시 쓰기자는 없다.

tx2의 소유권 검증은 파이프라인 레벨이며, 행별이 아니다. tx2에서 워커는 파이프라인 소유권을 한 번 검증하고 (잠긴 파이프라인 행에 대한 claimed_by = :claim_token 확인); 태스크와 파이프라인 상태 쓰기는 검증된 단일 쓰기자 클레임 하에서 발생하므로 추가적인 행별 클레임 가드가 필요 없다. task 테이블은 claimed_by 컬럼을 가지지 않는다.

tx2는 단일 트랜잭션 내에서 고정 순서로 실행된다: (1) 파이프라인 행을 SELECT ... FOR UPDATE; (2) claimed_by = :claim_token 검증 — 실패하면 전체 결과 기록을 no-op; (3) cancel_requested 읽기; (4) 태스크와 파이프라인 status 쓰기 — 플래그가 설정된 경우, ADR-016 §7에 따라 파이프라인과 모든 비종단 태스크 (현재 태스크 및 아직 BLOCKED인 후행 태스크들)를 CANCELLED로 구동; 그렇지 않으면 정상 현재 태스크 전환 적용; (5) 다음 next_due_at 쓰기 및 클레임 해제. cancel_requested를 쓰기가 취하는 동일 행 락 하에서 읽는 것이 협력적 취소 (결정 6 Case B)를 레이스-프리로 만든다 — 3단계 직전에 커밋한 취소는 관찰되고, 직후에 커밋한 취소는 tx2가 커밋될 때까지 블로킹되었다가 다음 클레임에서 반영된다.

성공 시 tx2는 클레임을 해제하고 next_due_at을 원자적으로 전진시킨다. 태스크/파이프라인 상태를 전진시키는 동일 트랜잭션이 claimed_by/claimed_until을 지우고 새 next_due_at을 쓴다. 이 해제 없이는 단계를 완료했지만 여전히 RUNNING인 파이프라인이 claimed_until 만료까지 잠겨 있어 다른 모든 워커가 집어 올 수 없다. 현재 태스크가 DONE에 도달하면, 동일 tx2가 다음 태스크를 BLOCKED → READY로 전환한다.

디스패치 시 워커는 task_attempt 행을 기록한다 (시도의 job_ids 집합 + 응답); 각 폴링 시 해당 시도의 task_check 요약 (카운트 + 마지막 응답)을 제자리 UPDATE한다; 둘 다 검증된 파이프라인 클레임 하에서 기록된다 (단일 쓰기자, 태스크 레벨 클레임 불필요). 태스크 완료는 최신 시도 행에 대한 코드 레벨 check(attempt, task)이다 — 리컨사일러는 해당 행만 읽으며, 완료 결정을 위해서만 읽는다; 클레임, 스케줄링, 파이프라인 전환은 시도 테이블을 절대 읽지 않는다. 시도 결과가 소실되어도 태스크가 멈추지 않는다: executionTimeout으로 흘러내려가고 새로운 N-job 실행을 재디스패치한다 (멱등적).

중복 외부 호출 구간

두 트랜잭션 분리는 InfraManager에 중복 호출이 발생할 수 있는 여러 제한된 구간을 만든다:

  • (a) 클라이언트 타임아웃 — 호출자는 타임아웃됐지만 InfraManager는 이미 job을 받아들였다.
  • (b) 디스패치 후 크래시 — 워커가 디스패치 후, tx2가 반환된 job_ids를 기록하기 전에 크래시.
  • (c) 리스 만료 중 정지 — 워커 스레드 풀 큐 대기 또는 GC 일시중지가 리스를 초과; 다른 워커가 재클레임하고 동일 파이프라인을 디스패치.

세 경우 모두 중복 외부 호출은 인프라 멱등성에 의해 안전해진다 — TF API는 중복 무해: 재디스패치는 부작용 없이 무해한 중복 job을 만들 수 있다. InfraManager는 중복 제거를 하지 않으며; 시도에 기록된 job_ids는 최신 디스패치의 집합이다. 가드된 DB 쓰기는 별도의 문제를 처리한다: 소유권이 만료된 오래된 스트래글러가 DB 상태를 덮어쓰는 것을 방지한다. 두 안전 장치는 상호 보완적이며 교환 가능하지 않다.

5 리스 만료를 통한 크래시 복구

크래시된 워커가 클레임한 파이프라인은 claimed_until < now()가 되면 다시 처리 대상이 된다. 리더도, 인간 개입도 없다 — 다음 스캔이 재클레임한다. 하드 제약:

lease_seconds > max_single_call_timeout + pool_queue_wait + scheduling_margin

클레임된 파이프라인은 외부 호출이 시작되기 전에 워커 스레드 풀 큐에서 대기할 수 있다; 그 큐 대기는 호출 자체와 마찬가지로 리스 시간을 소모한다. 리스는 클레임부터 tx2 커밋까지의 전체 경과 벽시계 시간 — 모든 큐잉 지연 포함 — 을 커버해야 한다.

6 취소: 유휴 파이프라인은 즉시, 실행 중인 파이프라인은 협력적

취소는 Admin/API 경로가 발행한다 (별도 프로세스, 자체 단기 트랜잭션). 적용 방식은 하나의 사실에 따라 갈린다 — 지금 이 파이프라인을 실행 중인 워커가 있는가?

Case A — 파이프라인이 실행 중이 아닌 경우 (클레임 없음 또는 리스 만료): 즉시 취소

일반적인 "대기 중 / 아직 픽업되지 않은" 파이프라인 케이스다. API 경로는 워커 왕복 없이 자체 트랜잭션에서 직접 종단시킨다:

-- A1: 파이프라인을 종단하고 클레임을 지운다 (만료된 리스 스트래글러의 tx2가
--     claimed_by 가드에서 실패하도록); 라이브 워커가 행을 소유하지 않을 때만 실행
UPDATE pipeline SET status = 'CANCELLED', claimed_by = NULL, claimed_until = NULL
 WHERE id = :pid AND status = 'RUNNING'
   AND (claimed_by IS NULL OR claimed_until < now());
-- A2: 모든 비종단 태스크를 종단화한다 (A1이 행을 업데이트한 경우에만 실행)
UPDATE task SET status = 'CANCELLED'
 WHERE pipeline_id = :pid AND status IN ('BLOCKED', 'READY', 'IN_PROGRESS');
Case B — 워커가 파이프라인을 실행 중인 경우 (라이브 리스): 협력적 취소

API 경로는 status를 쓸 수 없다 (워커가 유일한 status 쓰기자). 플래그만 올리고 파이프라인을 깨운다 — 이 경로는 Case A의 A1이 0행을 업데이트했을 때 실행된다:

UPDATE pipeline SET cancel_requested = true, next_due_at = now()
 WHERE id = :pid AND status = 'RUNNING';

클레임 보유 워커는 안전 지점 — 클레임 직후, 디스패치 전, 결과 기록 트랜잭션(tx2) 내 — 에서 cancel_requested를 읽으며, 설정된 경우 모든 비종단 태스크 (현재 태스크 및 아직 BLOCKED인 후행 태스크들)와 파이프라인을 CANCELLED로 종단화한 후 클레임을 해제한다. next_due_at = now()은 슬리핑 중인 파이프라인 (예: 긴 폴/조건 대기)을 깨워 다음 스캔이 신속하게 클레임할 수 있게 한다.

두 케이스 모두 레이스-프리인 이유:

  • 동시 클레임과의 경합 — 둘 다 파이프라인 행에서 경합하므로, 먼저 커밋하는 것이 이긴다: 클레임은 Case A가 취소한 후 RUNNING을 더 이상 보지 못한다; Case A의 A1은 워커가 클레임한 후 더 이상 빈 클레임을 보지 못하므로 (0행 업데이트) Case B로 넘어간다.
  • 만료된 리스 스트래글러와의 경합 — Case A는 클레임이 null/만료된 경우에만 실행되며 동일 구문에서 claimed_by/claimed_until을 지우므로, GC로 일시중지된 스트래글러가 재개되면 토큰이 없어 tx2 claimed_by 가드가 no-op — 부활 없음. 이것이 어느 케이스도 status 가드가 필요 없는 이유다.

허용된 레이턴시 엣지 (Case B): (i) 워커 풀에서 클레임됐지만 아직 큐 대기 중인 작업은 tx2가 실행될 때만 플래그를 관찰하므로, 취소 레이턴시는 큐 대기 — 콜별 타임아웃이 아닌 리스로 제한; (ii) 워커의 마지막 사전 디스패치 플래그 확인 이후에 도착한 취소는 하나의 외부 호출이 실행되도록 허용 — 멱등성에 의해 무해하며, tx2에서 CANCELLED로 기록된다. 이미 디스패치된 job은 완료되도록 남겨둔다 (멱등적 인프라); InfraManager 측 취소는 별도 후속 작업이며, 정확성에 필수는 아니다.

취소가 status를 쓰는 경우가 동일 행 경합 (Case A) 또는 유일한 쓰기자 워커 (Case B)를 통해서뿐이며, 워커 tx2와 라이브 클레임 토큰을 공유하지 않으므로, 종단 부활을 방지하기 위한 쓰기별 status 가드는 필요 없다.

7 어드미션 제어와 TF 슬롯 게이트 (소프트 캡)

runningPipelineCap은 하드 불변 조건이 아닌 소프트 픽업 목표이며, 생성이 아닌 클레임을 게이팅한다. 파이프라인 생성은 타겟별 유일성만 강제하며 (ADR-016) — 생성된 파이프라인은 즉시 RUNNING에 진입한다. 캡은 동시에 작업을 위해 클레임된 RUNNING 파이프라인 수를 제한한다: 캡을 초과하여 클레임하기 전에 워커는 활성 클레임된 파이프라인 수를 확인한다. 캡을 초과한 파이프라인은 단순히 RUNNING에서 클레임되지 않은 상태로 남아 있다가 슬롯이 비면 next_due_at 순서에 따라 나중에 픽업된다 — QUEUED 또는 WAITING_SLOT 상태 없음. 확인은 카운트 읽기이므로 오버슈트 가능: 캡 M과 C개의 동시 클레임 워커에 대해, 최악의 경우 동시 클레임은 M + C − 1. 이 제한된 오버슈트는 V1에서 허용된다. 하드 캡은 원자적 어드미션이 필요하다 (예: UPDATE pipeline_admission_counter SET used = used + 1 WHERE used < cap, 종단 시 해제) — 연기됨.

TF 슬롯 게이트. TF 슬롯은 Terraform-job 실행 점유 (분 단위)이며, API 호출 동시성 (밀리초~초)과는 다른 리소스다. slotCap은 InfraManager/Terraform 압력을 완화하기 위해 TF 디스패치를 늦추는 소프트 어드미션 목표다. 워커 루프의 소프트 게이트 동작: TF 디스패치 전에 워커는 슬롯 가용성을 확인; 가용하면 디스패치; 그렇지 않으면 태스크를 READY로 유지하고 next_due_at = now() + slotRetry로 설정하고 클레임을 해제한다 (워커가 다른 파이프라인을 픽업할 수 있도록). 제한된 오버슈트는 runningPipelineCap과 동일한 이유로 허용된다; 여기서 경합 액터 C는 슬롯을 확인하는 동시 워커들이다 (totalWorkerCount로 제한), pod가 아니다. 하드 캡은 tf_slot_counter CAS 또는 InfraManager 측 어드미션 제한을 사용 — 연기됨.

Pod 수는 정확성 파라미터가 아니다. 클레임 메커니즘은 구조적으로 멀티-pod 안전 — DB 클레임이 조정 기본 요소이며, 프로세스 수가 아님 — 이므로, 이 모델은 실행 중인 pod 수에 무관하다: 하나든 여럿이든, 정확성은 변하지 않으며 어느 쪽으로도 replicas 제약이 없다. pod 수 효과는 오직 허용된 소프트 캡 오버슈트에만 있다: totalWorkerCount = activePodCount × workerPerPod이며, 오버슈트는 동시 어드미션 액터 수 C와 함께 증가하고, 더 많은 pod가 이를 높일 수 있다. 그 오버슈트는 허용되므로 (결정 7), pod 수는 여기서 별도 거버넌스가 필요 없다.

워커 루프 의사코드 (예시; 슬롯 게이트 단계 포함):

loop:
  row = claim_one_due_pipeline()          // tx1: SKIP LOCKED + lease stamp
  if row is None:
    sleep(backoff_with_jitter())
    continue

  if row.cancel_requested:
    cancel_pipeline(row)                  // tx2: set CANCELLED, release claim
    continue

  if is_tf_dispatch_step(row) and not slot_available():
    reschedule(row, delay=slotRetry)      // tx2: release claim, push next_due_at
    continue

  result = execute_step(row)              // external call, outside any transaction

  report_result(row, result)              // tx2: guarded write-back, release claim

안전 메커니즘 & 튜닝 파라미터 (아키텍처 결정 아님)

정확한 값은 이 ADR이 아닌 운영 설정에 있다:

  • 워커 수 N은 동시 외부 호출을 제한한다 (≤ min(N, due pipelines)). 초당 요청 보장이 아님 — 429/503은 next_due_at을 앞으로 밀어 백오프한다.
  • 리스 기간 (lease_seconds)은 최대 단일 호출 타임아웃 + 풀 큐 대기 + 스케줄링 여유를 초과해야 한다 (결정 5 참고). 보수적으로 튜닝; 너무 짧은 리스는 두 가지 별개 효과가 있다: (1) 가드된 쓰기가 오래된 스트래글러의 DB 상태 덮어쓰기를 방지; (2) 멱등성 (ADR-016)이 InfraManager에 대한 중복 외부 호출을 무해하게 만든다. 어느 쪽도 손상을 초래하지 않지만, 불필요한 InfraManager 호출은 쿼터를 소모한다.
  • 클레임 술어 인덱스 — 클레임 술어 (status='RUNNING' AND next_due_at <= now() AND (claimed_until IS NULL OR claimed_until < now()))는 지원 인덱스가 필요하다; 없으면 모든 클레임이 동시 멀티워커 폴링 하에서 전체 순차 스캔 + 정렬로 저하된다. (next_due_at) WHERE status = 'RUNNING'에 대한 부분 btree 인덱스가 핫 경로를 커버한다.
  • Terraform-job 동시성 캡 (slotCap) — 소프트 게이트 동작은 결정 7에 정의됨; 하드 캡 강제 (counter-CAS)는 연기됨. InfraManager의 고정 풀이 실질적 상한; 과도 제출은 멱등적 큐를 깊게 할 뿐이다.
  • DB 폴링 부하 제어 — 워커는 자체 폴링하므로, 클레임 루프는 조용한 기간 동안 DB를 과도하게 두드리지 않도록 명시적 부하 제어가 필요하다: 빈 클레임 결과 후 즉시 재시도 없음; 지터(jitter)가 있는 적응형 백오프 (예: 200 ms → 500 ms → 1 s → 2–5 s, 작업 발견 시 리셋); 가장 가까운 처리 대상 파이프라인 시간이 알려진 경우 sleep = min(nextDueAt − now(), maxIdleSleep)로 과도 폴링 방지; 워커 전반에 걸친 동기화된 깨어남을 방지하기 위한 슬립 구간 지터 분산. (클레임 술어 부분 인덱스가 이미 스캔당 비용을 제한; 이것이 보완한다.)

검토한 대안

옵션 판정 이유
A. 멀티워커 claim-pull (SKIP LOCKED + 리스 + 가드된 쓰기) 채택 구조적으로 멀티프로세스 안전; 고가용성 + 수평 확장 내재; 취소-대-워커 및 배포-중복 레이스 생존; 리스 만료를 통한 크래시 복구.
B. 단일 서버 (replicas=1) + 인메모리 in-flight 가드 기각 두 번째 쓰기자가 생기는 순간 실패: 취소가 워커와 경쟁하여 종단 부활을 유발하고, 모든 프로세스 중복 (롤링 배포 / 스케일 / 재시작)이 이중 상태 전환을 초래한다. 멱등성은 이중 디스패치를 커버하지만 상태 전환 레이스는 아니다.
C. 워크플로우 엔진 (Temporal / Airflow / broker) 기각 분 단위 폴링의 2–4단계 선형 체인이 운영 비용을 정당화하지 않는다.
D. 인메모리 비동기 체인 (스캔 없음) 기각 재시작/배포 시 실행을 잃는다; 다일 대기를 내구성 있게 표현할 수 없다. ADR-016이 이미 DB를 유일한 상태로 요구한다.

결과

장점
  • 구조적으로 멀티프로세스 안전: DB 클레임이 조정 기본 요소이며, 프로세스 수가 아니다.
  • 운영하거나 디버깅할 리더 없는 고가용성과 수평 확장.
  • 취소 안전: 단일 status 쓰기자 + 협력적 플래그 (cancel_requested); 클레임 보유 워커가 CANCELLED를 직접 적용. 쓰기별 status 가드 불필요; 부활 불가.
  • 크래시 복구는 리스 만료를 통해 자동 — 복구 저널 없음, 수동 단계 없음.
감수할 비용
  • 리스 튜닝: lease_seconds는 최대 호출 시간을 초과해야 한다; 너무 짧은 리스는 불필요한 (안전한) 작업을 유발; 너무 긴 리스는 크래시 후 복구를 지연시킨다.
  • 두 트랜잭션 분리는 단일 인프로세스 쓰기보다 움직이는 부분이 많다: 클레임 tx, 외부 호출, 결과 기록 tx를 별도로 추론해야 한다.
  • 리스 만료 구간: 크래시된 워커의 파이프라인은 재클레임 전까지 최대 한 리스 기간 동안 일시 중지된다.
  • 소프트 어드미션 목표 — runningPipelineCap과 slotCap 모두 소프트 게이트; 동시 어드미션 읽기는 최대 C − 1만큼 오버슈트 가능 (여기서 C = 동시 어드미션 액터 수: runningPipelineCap의 어드미션 트랜잭션, 슬롯 게이트의 워커들). 하드 캡은 명시적 counter-CAS 어드미션이 필요 — 연기됨.

운영 참고 (파라미터 & 메트릭)

이 섹션은 참고 카탈로그이다. 이것들은 정적 서버 설정 (예: Spring application.yaml) — API로 변경 가능한 런타임 파라미터가 아님 — 이며, 값은 이 ADR이 아닌 운영 설정에 있다. 특히 리스와 동시성 값은 설계 시 고정된 것이 아니라 관찰된 InfraManager 레이턴시에 대해 운영 중에 튜닝된다.

주요 메트릭

  • active workers — 현재 단계를 실행 중인 워커 스레드
  • total worker count — activePodCount × workerPerPod
  • concurrent API calls — InfraManager / 조건 확인에 대한 in-flight 외부 호출
  • RUNNING pipeline count — 현재 status = 'RUNNING' 행 수
  • pipeline-cap overshoot count — runningPipelineCap 초과 어드미션
  • empty-claim rate — 행을 반환하지 않은 클레임 폴 비율
  • claim QPS — 클레임 폴 처리량
  • claim latency — tx1 클레임 쿼리의 p50/p99 소요 시간
  • due-pipeline lag — 클레임되지 않은 처리 대상 행에 대한 now() − min(next_due_at)
  • stale-report discard count — claimed_by 가드 실패로 no-op된 tx2
  • lease-expired reclaim count — 리스 타임아웃 후 재클레임된 파이프라인
  • slot-full retry count — 슬롯 게이트로 연기된 TF 디스패치
  • TF-slot overshoot count — slotCap 초과 TF 디스패치
  • 429/503 count — InfraManager의 백프레셔 응답
  • API latency — 외부 호출 소요 시간 p50/p99
  • DB lock wait — 클레임 스캔 중 행 락 대기 시간
  • pipeline completion latency — 파이프라인 생성부터 종단 상태까지의 벽시계 시간

링크

  • ADR-016 — 이 실행 모델이 구동하는 내구성 도메인 모델

용어

  • Claim (클레임) — 짧은 커밋된 트랜잭션(tx1)에서 클레임 시점에 생성된 클레임별 펜싱 토큰 (신규 UUID)을 claimed_by에, 데드라인을 claimed_until에 스탬프하는 행위. 각 클레임은 새로운 고유 토큰을 생성하며; 리스 만료 후 동일 파이프라인을 재클레임하는 동일 워커는 다른 토큰을 받아, 이전 클레임의 in-flight tx2가 소유권 가드에서 거부된다. 클레임 시 워커는 진행 전에 cancel_requested도 확인한다.
  • Cooperative cancel (협력적 취소) — Admin/API 경로가 cancel_requested = true만 쓰는 (그리고 슬리핑 파이프라인을 깨우기 위해 next_due_at = now()으로 설정하는) 취소 모델 (Case B); 클레임 보유 워커가 안전 지점에서 이 플래그를 읽고 종단 CANCELLED 전환을 직접 적용한다. 유일한 status 쓰기자는 워커로 유지; 종단 부활을 방지하기 위한 쓰기별 status 가드 불필요.
  • Due pipeline (처리 대상 파이프라인) — next_due_at <= now()이고 리스가 만료되었거나 설정되지 않은 파이프라인.
  • Guarded write-back (가드된 쓰기백) — 소유권 가드 실패 시 no-op이 되는 UPDATE ... WHERE id = :id AND claimed_by = :token. 리스 만료 스트래글러의 덮어쓰기를 방지; 취소가 협력적이고 status가 단일 쓰기자이므로 (결정 6) status 가드는 불필요.
  • Lease (리스) — claimed_until 타임스탬프; 만료 시 자동으로 클레임을 해제하여 어느 워커든 재클레임 가능.
  • Two-transaction split (두 트랜잭션 분리) — tx1 (클레임)과 tx2 (결과 기록)는 별도의 커밋된 트랜잭션; 외부 호출은 그 사이에 트랜잭션 밖에서 실행된다.

개정 이력

  • 2026-06-27: ADR-016에서 분리하여 생성; 실행 모델을 여기로 추출함으로써 도메인 모델과 독립적으로 대체 가능하게 됨.
  • 2026-06-27: 동시 취소 및 멀티 세션 하에서 단일 쓰기자 전제가 깨지는 것이 발견된 후, 단일 서버/인메모리 모델을 멀티 워커 claim-pull로 교체.
