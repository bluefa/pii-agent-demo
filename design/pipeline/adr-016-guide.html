ADR-016

00 한눈에 (TL;DR)

무엇을. target source(약 2000개)의 설치/삭제를 브라우저 세션 없이 자동·복구가능하게 진행하고, 실행 히스토리를 일급으로 조회·알림한다.

어떻게. BFF 안에 durable state machine과 30초 주기 reconciler tick을 둔다. DB가 유일한 상태이고 tick이 각 task를 전진시키며 pipeline 상태는 task에서 파생된다.

대가. 다중 replica에선 BFF가 더 이상 stateless proxy가 아니라 리더 선출이 필요하다 — 이 ADR에서 가장 비싼 한 줄이지만 restart-safety·히스토리·조회성이 같은 뿌리에서 나온다.

이건 다음이 아니다:

❌ 워크플로 엔진 (Temporal/Airflow) ❌ exactly-once 기계 ❌ in-memory async 체인

01 문제와 맥락

현재 문제. 모든 Infra Manager Terraform 호출이 사람 수동이다. 브라우저 세션 없이 자동·복구가능하게, 실행 히스토리를 일급으로 만들어야 한다. 규모는 target source 약 2000개, provider × install/delete 파이프라인 정의 약 12종.

시스템 계층

Frontend (Admin 콘솔)
   │  Admin API (UI = API 동일 표면)
   ▼
[ BFF ]  ← 파이프라인 오케스트레이션이 여기 산다
   ├──► Backend Manager   (연동·승인·도메인 상태)
   └──► Infra Manager      (Terraform run API — 비동기, job_id 반환)
            │ pubsub
            ▼
      TerraformWorker  (k8s pod · dedup 없음)

확정 제약 7

1. 오케스트레이션 = BFF

파이프라인은 Admin 콘솔에서 생성되고 브라우저 없이 전진한다.

2. IM run = 비동기

job_id 반환, pubsub로 worker 실행. 일부 호출은 job_id 없이 조건으로 판정.

3. dedup 없음 → 멱등

중복 제출은 각각 실행되나 모든 execution API가 멱등이라 인프라 결과 무손상.

4. 결과 유실 가능

드문 worker 결함 — "실행 중"과 "유실"을 구분하지 않고 execution timeout으로 흡수.

5. 고정 풀 hard cap

동시 TerraformJob은 workerPoolSize (M)으로 hard-cap; BFF slotCap (N)은 그 위의 제출 throttle.

6. 히스토리·알림 일급 (2단 보존)

run/task/attempt/event spine은 무기한, 고빈도 task_check은 retention 창(기본 90일).

7. AI 운영 = 범위 외

단 모든 admin 액션이 BFF admin API라, 향후 AI 운영자는 또 하나의 API principal로 표면 변경 없이 추가.

02 핵심 결정

BFF 내부 durable state machine + reconciler tick을 둔다. DB가 유일한 상태이고, 30초 tick이 각 task를 전진시키며 pipeline 상태는 task에서 파생된다. 다중 replica는 advisory lock 리더 + 모든 전이 CAS로 안전하다.

먼저 — 두 개의 토대 (axiom)
이 두 가지는 아래 모든 섹션(상태기계·dispatch)이 전제로 깐다:
  • 단일 writer — 상태 전이는 tick만 쓴다. 관측(task_check)·산출(attempt.response)은 호출 스레드가 쓴다. 다음 tick이 그 관측을 소비해 전진한다.
  • 호출 deadline ≠ tick 주기 (D-T1) — 한 외부 호출이 tick 안에 끝날 필요가 없다(비블로킹 async 발사).

정본 결정 1~7

결정 1 — durable state machine + tick

DB=유일 상태. 다중 replica = advisory lock 리더 + CAS 전이. UI 액션은 외부 호출도 task 전진도 안 함.

결정 2 — TaskKind 2종

TERRAFORM_JOB · CONDITION_CHECK. 새 task = 새 코드 class(대개 기존 kind 재사용).

결정 3 — idempotency-by-construction

at-least-once dispatch + 멱등. exactly-once 기계 안 만듦.

결정 4 — Liveness

무한대기 없음. 4a timeout budget · 4b workerPoolSize hard cap + slotCap · 4c cancel=drain · 4d systemic 실패는 알림 롤업.

결정 5 — retry=새 run + unique

재개 없음(terminal 부활 없음). 동일 target unique 제약 → 생성 계약(↓ 2c).

결정 6 — tick 외부 호출 모델

D-T1~D-T7. 단일 writer·async 발사·per-call deadline·RLE.

결정 7 — Definition

코드 default recipe + run snapshot 박제(7.1~7.4). 코드=실행 권위.

02c 생성 계약 — "target당 실행자 1"

ADR이 가장 강하게 못 박은 외부 불변식
이 계약을 누락하면 단일 writer·slotCap·멱등 추론이 전부 깨진다 — 그래서 별도로 강조한다.

어느 경로로 생성되든 트리거는 다음 3단계를 충족해야 한다:

  1. Resolution — (type, provider) 코드 default recipe를 resolve(결정 7).
  2. 원자성 — task row 생성 + pipeline_def_snapshot 박제를 한 트랜잭션으로(snapshot == 실제 생성 구성).
  3. 중복 차단(필수) — 동일 target에 non-terminal pipeline이 있으면 부분 unique 제약이 INSERT를 막는다. 트리거는 그 23505(Postgres unique 위반)를 반드시 catch해 에러 대신 기존 non-terminal pipeline을 반환해야 한다.

[재시도]도 동일하다 — terminal에서 호출해도 target에 non-terminal이 있으면 그것을 반환한다(created=false). 토대 불변식이 ADR 밖 endpoint 코드에 의존하므로 이 계약(특히 3번)의 통합 테스트를 반드시 갖춘다(계약 회귀 방지).

03 검토한 대안

옵션결정이유
A. BFF 내부 durable reconciler (DB row + tick)채택실제 워크로드에 부합; 재시작 안전; 관리자 조정형 폴링이 next_check_at에 자연 대응; 최소 운영 footprint
B. 별도 오케스트레이터 마이크로서비스보류모듈 분량 로직에 서비스 분량 오버헤드. 모듈 경계가 추출 비용을 낮게 유지
C. 워크플로 엔진 (Temporal/Airflow)거부≥10분 폴링의 2~4 step 선형 체인은 그 도입 비용을 정당화 못 함. 큐·재현·수일 WAIT는 DB row+tick으로 충분
D. BFF 인메모리 async 체인거부재시작/배포에 run 유실; durable 큐·히스토리·수일 대기 표현 불가
E. Backend Manager에 파이프라인 상태거부로직·상태 분산; 원격 API 경유 원자적 slot 회계는 racy
F. Infra Manager 측 동시성 제한 (지금)보류현재 자동화 caller는 BFF뿐; 다른 caller 등장 시 IM 429/503 재검토(결정 4b)

별도 워커 분리도 배제한다 — 무거운 워크로드(실제 Terraform 실행)는 이미 TerraformWorker로 분리돼 있고 reconciler는 호출·폴링·기록만 한다. 부하가 강제하면 답은 워커 분리가 아니라 Option B다.

05 상태 기계

task 전이의 writer는 reconciler tick이다(단일 writer). 예외는 pipeline-level 사용자 전이 하나 — [중단] RUNNING→CANCELLING은 Admin API가 공통 전이 함수로 수행한다.

Pipeline 전이 (5상태)

          (생성) 즉시
             ▼
       RUNNING ──[중단] (가드: prior=RUNNING)──► CANCELLING
          │ 매 tick 파생 (¬CANCELLING)              │ 파생 ① (최우선)
          │  ② FAILED task 존재 → FAILED           │ 전 in-flight drain 완료
          │  ③ TTL EXPIRED      → FAILED           ▼
          │  ④ 전 task DONE     → DONE          CANCELLED
          ▼
   FAILED / DONE

Task 전이 (9상태)

(생성)→BLOCKED ──seq-1 DONE──► READY ──┐ kind로 분기
 TERRAFORM_JOB │ READY ──admit: COUNT(DISP|RUN)<slotCap──► DISPATCHING → RUNNING → DONE
 CONDITION_CHECK │ READY ─────────────────────► WAITING_EXTERNAL → DONE

tick 평가 순서 (한 task 안에서 — 처음 성립하는 규칙 하나만)

  1. pipeline=CANCELLING이면 취소 규칙 우선 — 일반 forward·재시도를 누른다.
  2. handler_key resolve(READY 이상) — 미해결이면 즉시 FAILED(HANDLER_NOT_FOUND).
  3. 완료 관측 우선 (timeout보다 앞) — 최신 observed=SUCCEEDED/MET면 DONE. "만료는 fresh 상태 재독 후 판정"이라 이미 완료된 작업을 timeout으로 실패시키지 않는다.
  4. timeout 판정 — execution timeout → attempt 실패 / TTL 초과 → EXPIRED.
  5. 그 외 일반 전이.

tick 바깥 순서 (task-pass → pipeline 파생 pass)

한 tick = due task를 next_check_at ASC, last_checked_at ASC NULLS FIRST, created_at ASC, seq ASC 정렬로 각 ≤1회 전진(독립 commit) → 그 tick에 커밋된 task 상태로 pipeline 파생 1회. 따라서 마지막 task terminal과 pipeline 수렴이 같은 tick(별도 지연 없음). BLOCKED→READY는 같은 tick 먼저 처리된 seq-1의 커밋 DONE을 본다(순차 chain 1 tick에 1칸). 최저 seq는 predecessor 공집합 → 첫 tick 무조건 READY.

CANCELLING precedence — 파생 ①~④
①CANCELLING(최우선) > ②FAILED task 존재 > ③TTL EXPIRED > ④전 task DONE. 취소 중 task가 FAILED여도 pipeline은 CANCELLED 단일 수렴 — 판정은 상태 기준(파생 시점 pipeline.status)이지 시간 기준이 아니다.

cancel / drain 매트릭스

대상중단 시
BLOCKED/READY (전 kind)즉시 CANCELLED (forward edge gate → 전진 불가)
WAITING_EXTERNAL (CONDITION_CHECK)즉시 CANCELLED (read-only 폴링 — drain 대상 없음)
DISPATCHING (TF)즉시 CANCELLED — handle 적재 여부 무관(RUNNING 승격 전이라 drain 안 함; 늦게 적재된 job_id는 버려져 orphan)
RUNNING (TF)drain — job_id 영속됨, 자연 terminal까지 폴링(forward edge만 gate). 실제 실패는 사실대로 마감(fail_count++)하되 pipeline은 CANCELLED 수렴

06 동시성 모델

네 개의 값이 모두 "동시성 제한"처럼 들리지만 hard cap은 하나뿐이다.

workerPoolSize (M)

동시성 hard cap. 고정 worker 풀 크기(IM 보호). 배포 설정 — settings API 비편집·비노출.

slotCap (N)

BFF 제출 throttle(slotCap ≈ workerPoolSize). pubsub 큐를 얕게. soft target — 안전장치 아님.

slotCap × maxFailCount

retry/orphan worst-case 제출량 sizing 값(큐로 흘러드는 총량). 동시 실행 상한 아님.

max_external_calls_per_tick

tick당 poll/check 발사 상한(기본 50). dispatch 제외 — dispatch는 slotCap admission으로 제한.

fail_count 가산 매트릭스

사건가산?
TERRAFORM_JOB poll 호출 오류 (잡 못 읽음)❌ 미가산 (관측 실패 ≠ 잡 실패)
TERRAFORM_JOB attempt 실패 — dispatch(IM_REJECTED·DISPATCH_NO_RESPONSE) 또는 RUNNING/job(JOB_FAILED·EXECUTION_TIMEOUT)✅ 가산
CONDITION_CHECK CHECK ERROR / CALL_TIMEOUT (비-backpressure)✅ 가산
NOT_MET ("아직")❌ 미가산
backpressure (429/503, 전부)❌ 미가산 ("나중에")
HANDLER_NOT_FOUND · 취소 정리(error_code=null)❌ 미소모 (영구 실패·취소는 재시도 가능 시도 아님)
drain 중 RUNNING TF 실제 실패/timeout✅ 가산 (사실대로 — 단 pipeline은 CANCELLED 수렴)

maxFailCount (K) 정의: TERRAFORM_JOB = 초기 dispatch 포함 최대 attempt 수(총 attempt ≤ maxFailCount, 재dispatch ≤ maxFailCount−1) / CONDITION_CHECK = 비-backpressure CHECK ERROR·CALL_TIMEOUT 허용 횟수.

slotCap ↔ execution-timeout 결합 (튜닝 주의). slotCap ≫ workerPoolSize이면 pubsub 큐가 깊어져 정상 job의 큐 대기 시간까지 execution timeout 경과에 포함돼 오발한다. 그래서 slotCap ≈ workerPoolSize로 큐를 얕게 유지하는 것이 timeout 정확도의 레버다.

08 dispatch · 멱등성 · 복구 (crash & N-pod)

dispatch 5단계 writer 분리

1 tick: DISPATCHING CAS + attempt 생성→ 2 호출스레드: DISPATCH PENDING 선기록→ 3 호출→ 4 (a) 관측 always 기록 (b) response 채택 CAS→ 5 다음 tick: RUNNING CAS

4(b) response 채택 가드: response IS NULL AND finished_at IS NULL AND status=DISPATCHING — write-once + 늦은-response 차단.

at-least-once + idempotency-by-construction

worker dedup 없음 → 중복 제출 무해(멱등 apply). "이미 원하는 상태=성공"(INSTALL 존재=성공 / DELETE 부재=성공).

DELETE not-found 함정. destroy가 "리소스 없음"을 에러로 던지면 crash 재dispatch가 멀쩡히 끝난 삭제를 FAILED로 종결시킨다 → "이미 부재=성공"이 멱등 정의에 포함돼야 한다. 멱등은 BFF가 검증하는 사실이 아니라 task 등록 계약이다(O28).
crash-point 매트릭스 — 최악의 순간들
Crash 지점복구
tick 도중, task 사이전이는 독립 commit; 완료분 유지, 나머지는 다음 tick 재도출
dispatch 후, response 영속 전DISPATCHING 잔류 → 복구가 재dispatch(fail_count++); 멱등이 중복 흡수
poll/check 응답 후, 기록 전관측 1회 유실; 다음 cadence 재독(읽기 idempotent)
실행 주체 사망 (status 미변경)다음 tick 재호출. DISPATCH는 PENDING으로 "시도 이력" 보존
상태 기록 후, 알림 전outbox 행 notified_at IS NULL → Notifier 재시도
advisory lock 보유 중session-scoped lock 자동 해제, 다음 tick 다른 pod 획득
장시간 outage복귀 첫 tick들에 밀린 작업 일괄 발화 — catch-up burst를 slotCap·max_external_calls가 wave로 흡수
N-pod Q&A — 다중 replica 우려
우려답
tick은 누가 도나매 tick 모든 pod가 advisory lock 시도; 1 pod 승리. failover 자동 ≤30초
split-brainlock 세션 유실로만 가능·무해: CAS + 반복 안전 side effect
Admin API on N podsstateless — 어느 pod든 읽기 + pipeline-level 전이(공통 CAS)
Notifier on N pods리더 불요: FOR UPDATE SKIP LOCKED로 분담
rolling deploysplit-brain과 동일 + snapshot이 구성을 박제(코드 동작은 현재 배포본)

backpressure (429/503)

fail_count 미소모. dispatch = Retry-After 있으면 그만큼, 없으면 다음 tick(cadence 하한 없음·동일 logical attempt 재사용). poll·check = max(Retry-After, kind cadence).

K headroom. response 영속 직전 좁은 pre-persist crash 창이 정상 job의 fail_count 1을 소비한다 → maxFailCount를 IM 최소가 아니라 약간의 여유를 두고 잡는다.

10 Definition · Snapshot (결정 7)

2-layer

코드 default recipe ((type,provider)당 1개)→ 생성 시 박제 → pipeline_def_snapshot (write-once, 1:1)→ 재현 권위

코드 = 실행 권위 / snapshot = 이력 권위.

절연 범위. default release를 올려도 in-flight·과거 run의 구성(task 목록·순서·ttl·polling·execution_timeout·max_fail_count)은 불변. 단 task class 코드 동작은 현재 배포본을 탄다(절연 대상 아님 — 코드=실행 권위).

handler 자동 레지스트리

  • 각 handler는 안정 key() 선언(클래스명 무관 — rename해도 키 유지).
  • 레지스트리 자동 수집(모든 handler 빈 → key()→handler 맵 부팅 시 파생; 중복 키 = 부팅 실패).
  • recipe는 handler를 class로 참조(컴파일 안전).
  • 2단계 검증 — pre-deploy CI 게이트가 authoritative(배포 전 prod DB non-terminal handler_key 검사; 부팅 assert는 defense-in-depth — 배포를 막으려면 배포 후인 부팅이 아니라 배포 전이어야).
  • 런타임 미해결 시 HANDLER_NOT_FOUND. 비호환 변경만 _V1/_V2 append-only(호환 bugfix는 현재 코드).

알림

모든 알림은 transactional outbox(pipeline_event)에서 나간다 — 상태 변경과 같은 tx라 유실 없음(at-least-once, 인앱 effectively-once).

  • WORKER_OUTAGE_SUSPECTED (critical) — execution timeout 연속 시 systemic worker 장애 의심 롤업.
  • QUEUE_WAIT_EXCEEDED — slot 대기 임계(기본 30분) 초과.
운영자 대시보드 멘탈모델: workerPoolSize · slotCap≈workerPoolSize · 큐 깊이 · timeout rate만 보면 된다.

없는 버튼

❌ force-check 없음 ❌ task 재시작 없음 ❌ pause/resume 없음 ✅ 제어는 cancel · retry 둘뿐

14 용어 사전

본문의 점선 밑줄 용어에 마우스를 올리면 여기 정의가 툴팁으로 뜬다(클릭하면 해당 항목으로 점프). 이 사전이 모든 툴팁의 단일 출처다.

시스템 · 계층

CSR
Client-Side Rendering. 브라우저 클라이언트 렌더 — "브라우저 세션"의 주체.
SSR
Server-Side Rendering(대비 용어).
Infra Manager (IM)
Terraform run API 제공자(비동기, job_id 반환). pubsub로 TerraformWorker 실행.
Backend Manager
연동·승인·target source 도메인 상태 소유.
TerraformWorker
실제 terraform 실행 k8s pod. dedup 없음 — 중복 제출은 각각 실행된다.
Terraform

코어 아키텍처

reconciler
30초 tick으로 깨어나 due task를 전진시키는 단일 논리 스케줄러. task를 전진시키지 "파이프라인을 돌리"지 않는다.
tick
reconciler가 깨어나는 고정 주기(기본 30초). 호출 deadline과 별개 시계(D-T1).
durable state machine
DB row로 영속되는 상태기계. 메모리 의존 0 → 재시작 안전.
advisory lock
pg_try_advisory_lock. 다중 replica 중 한 pod만 tick 돌리는 리더 선출(session-scoped — 세션 끊기면 자동 해제).
리더 선출
advisory lock 보유 pod가 리더. failover 자동 ≤30초.
CAS
Compare-And-Swap. WHERE에 기대 prior 상태를 실은 조건부 갱신 — 낡은/중복 writer는 0행. 다중 replica 안전.
replica
BFF 다중 인스턴스(pod). 상태는 DB에만 있어 어느 replica가 죽어도 안전.
단일 writer
상태 전이는 tick만, 관측(task_check)·산출(attempt.response)은 호출 스레드(D-T4).
Notifier
notified_at IS NULL 이벤트를 소비해 인앱 알림 발송하는 BFF 루프. 리더 불요(FOR UPDATE SKIP LOCKED 분담).
FOR UPDATE SKIP LOCKED
N pod가 리더 없이 outbox 행을 나눠 처리하는 행 잠금절.

작업 모델

pipeline (run)
한 target의 설치/삭제 실행 1건. type×provider. 실행 단위=target_source_id(1 pipeline : 1 target).
task
pipeline 내 순차 chain의 한 단계. seq 순서·9상태.
TaskKind
task 흐름 shape. v1 2종(TERRAFORM_JOB·CONDITION_CHECK). slot 소비 여부가 kind에서 정해진다.
TERRAFORM_JOB
dispatch→job_id→poll→terminal. IM slot 소비. attempt 있음.
CONDITION_CHECK
dispatch 없이 조건 MET까지 폴링. attempt 없음. slot 비소비.
GENERAL_JOB
비-terraform 비동기 job kind. v2 defer(RECONNECT와 함께 도입).
handler_key
안정적 코드 class 식별자(예 aws.tf.network). reconciler 라우팅 키. 미해결=HANDLER_NOT_FOUND.
handler
task 흐름을 구현한 코드 class. 안정 key() 선언, 자동 레지스트리 수집.
레지스트리(자동)
모든 handler 빈을 주입받아 key()→handler 맵을 부팅 시 파생(수동 목록 없음·중복 키=부팅 실패).
stable key / key()
handler가 선언하는 클래스명 무관 안정 키(rename해도 불변; 키 문자열의 단일 출처).
seq
task 순서 번호. predecessor(seq-1) DONE이면 승격. unique(pipeline_id, seq).
predecessor
직전 seq task. 그게 DONE이어야 BLOCKED→READY 승격.
recipe
(type,provider)당 코드 default 정의(task 시퀀스, release version 1개). 생성 시 snapshot으로 박제.
target source
설치/삭제 대상. 약 2000개. 1 pipeline : 1 target.
task_attempt (attempt)
dispatch action의 생애주기 1행. dispatch당 1개. action=side-effect(관측값 저장 아님).
logical attempt
동일 attempt 단위. 429/503 재호출은 같은 logical attempt 재사용(새 행 없음).
task_check (check)
외부 호출 관측 장부. DISPATCH=호출당 1행 / CHECK=관측 run당 1행(RLE).
dispatch
IM run API 호출로 job_id 획득(side-effect 발생). TERRAFORM_JOB만.
poll
dispatch한 job의 상태를 cadence마다 반복 확인(read-only, kind=CHECK).
관측 run
RLE 단위. 연속 동일 (api_result,observed,error_code) 폴을 1행으로 접고 poll_count++.
external_handle
task_check가 확인한 id의 참조(home은 attempt.response, handle 저장소 아님).
handle
dispatch 산출 식별자(terraform=job_id). attempt:handle = 1:1.
fan-out
한 dispatch가 handle 여러 개 반환. 금지 — 독립 작업이면 task를 나눈다.

동시성

workerPoolSize (M)
고정 worker 풀 크기 = 동시성 hard cap(IM 보호). 배포 설정 — settings API 비편집·비노출. 예 4~8.
slotCap (N, 구 N-cap)
BFF admission 제출 throttle(slotCap ≈ workerPoolSize). pubsub 큐를 얕게. soft target — 동시성 안전장치 아님. 초안 3.
slot
TERRAFORM_JOB이 DISPATCHING/RUNNING 동안 점유하는 admission 단위. slotCap개 한도.
slot 큐
admit 못 받은 READY TERRAFORM_JOB(별도 상태 없이 READY∧kind=TF로 표현; 순번=admission 순서 파생).
admission
READY TF를 COUNT(DISPATCHING|RUNNING) < slotCap일 때만 DISPATCHING 승격.
soft target
CAS 불변식이 아니라 근사 목표(slotCap; COUNT→admit read-then-act라 leader 단일성 의존). split-brain 일시 초과는 풀이 흡수.
slotCap × maxFailCount (구 N·K)
retry/orphan worst-case 제출량 sizing 값(큐로 흘러드는 총량). 동시 실행 상한 아님.
maxFailCount (K, = max_fail_count 컬럼)
task FAILED 전 최대 비성공 시도. TF=초기 dispatch 포함 최대 attempt 수 / CONDITION_CHECK=비-backpressure CHECK ERROR·CALL_TIMEOUT 허용 횟수.
fail_count
"성공하지 못한 시도 횟수". maxFailCount 도달 시 task FAILED. 미가산: backpressure·NOT_MET·TERRAFORM_JOB poll 오류(잡 못 읽음). 가산: TF dispatch/job 실패·CONDITION_CHECK CHECK_ERROR/CALL_TIMEOUT·drain 중 실제 실패.
max_external_calls_per_tick
tick당 poll/check 발사 상한(기본 50). dispatch 제외(slotCap admission으로 제한). burst 완화.
split-brain
lock 세션 유실로 두 pod가 잠시 둘 다 리더. 무해(workerPoolSize가 cap·CAS 수렴).
catch-up burst / wave
outage 복귀 첫 tick들의 밀린 작업 일괄 발화. slotCap·max_external_calls가 wave로 배수.

정합성 · 신뢰성

멱등성
중복 실행이 결과를 손상하지 않음. 모든 dispatch task의 필수 계약.
idempotency-by-construction
exactly-once 기계 대신 모든 dispatch를 멱등으로 만들어 at-least-once를 안전화.
at-least-once
최소 1회 보장(중복 가능). dispatch·알림 전달 모델.
exactly-once
정확히 1회(채택 안 함 — 멱등으로 대체).
write-once
attempt.response·snapshot은 1회 기록 후 불변(재시도=새 attempt=새 response).
pre-persist crash
response 영속 직전 crash. 정상 job의 fail_count 1을 소비 → maxFailCount headroom 근거.
늦은 response
cancel/terminal 후 도착한 dispatch 응답. CAS guard로 무시.
backpressure
IM 429/503. 실패 아님 → requeue, fail_count·errorCode 미소모.
Retry-After
429/503 재시도 지연 힌트. dispatch는 cadence 하한 없이 이걸로(없으면 다음 tick) 미룸.
cadence
폴링 주기(job-poll 30~60초 / condition ≥10분 guard).
transactional outbox
상태 변경과 같은 tx에 이벤트를 써 유실 없이 알림(pipeline_event).
effectively-once
at-least-once 전달 + read-dedup(event id)로 인앱 알림 중복 제거.
crash recovery
response 미영속 DISPATCHING을 dispatch_recovery_timeout 후 재dispatch.
fresh 상태 재독
timeout 판정 전 최신 상태를 읽어 이미 완료된 작업을 timeout으로 실패시키지 않음(완료관측 > timeout).
orphan job
BFF가 추적을 끊은 in-flight job(crash 재dispatch·취소·HANDLER_NOT_FOUND). 멱등이라 무해; worker terraform 자연 종료가 bound.

시간 · 생명주기

execution timeout
dispatch→terminal 경과 상한(기본 30분). 초과 시 attempt 실패(흡수).
TTL (WAIT_EXTERNAL)
CONDITION_CHECK 총 체류 상한(기본 7일). 초과 시 EXPIRED→pipeline FAILED.
per-call HTTP deadline
단일 호출 타임아웃(전역 30초 + TaskKind 오버라이드). tick 주기와 별개.
next_check_at
다음 폴 예정 시각(스케줄 힌트, 상태 전이 아님). 일반=tick이, backpressure 재설정=호출 스레드가 쓴다.
last_checked_at
tick이 task를 마지막 서비스(발사)한 시각. 기아 방지 정렬 키(ASC NULLS FIRST).
deadline_at
현재 적용 timeout의 절대 만료 시각(reconciler 파생값, 내부 비노출).
last_activity_at
마지막 상태 전이 시각. 보드 기본 정렬 키(started_at과 다름).
started_at
(pipeline)RUNNING 진입=생성 시각·overlap 필터 기준 / (task)첫 실행 개시=READY→DISPATCHING(TF)·READY→WAITING_EXTERNAL(CONDITION_CHECK) 전이 시각(BLOCKED→READY 승격이 아님) / (attempt)dispatch 시작 / (task_check)run 첫 발사.
finished_at
terminal 도달 시각(pipeline/task/attempt 각 grain).
created_at
pipeline 생성 시각(생성 즉시 RUNNING이라 started_at과 동일).
checked_at
task_check run 마지막 관측 시각(DISPATCH PENDING이면 null).
dispatch 복구 timeout
response 없는 DISPATCHING을 재dispatch하는 기준(기본 5분).

상태 (task 9종 · pipeline 5종)

BLOCKED
의존 미해소(predecessor 미완). reconciler가 쳐다볼 후보 아님.
READY
전진 가능. READY∧TF = slot 큐 대기(WAITING_SLOT 없음).
DISPATCHING
dispatch 발사 ~ 다음 tick의 RUNNING 전이 전(response 적재돼도 RUNNING 승격은 다음 tick; 그 사이 cancel도 DISPATCHING 취급 — drain 안 함). slot 점유.
RUNNING (task)
job_id 폴링 중. slot 점유.
WAITING_EXTERNAL
CONDITION_CHECK 조건 폴링 중.
DONE
성공 terminal.
FAILED (task)
실패 terminal(maxFailCount 소진/HANDLER_NOT_FOUND 등).
EXPIRED
WAIT_EXTERNAL TTL 초과(task). → pipeline FAILED 파생.
CANCELLED (task)
폴링할 handle 없는 task의 중단 terminal.
CANCELLING
pipeline 중단 진행. 파생 최우선. forward edge gate, in-flight drain.
CANCELLING precedence
파생 ① 최우선. 취소 중 task가 FAILED여도 pipeline은 CANCELLED 단일 수렴(상태 기준, 시간 기준 아님).
drain
죽일 수 없는 in-flight job(RUNNING TERRAFORM_JOB, job_id 영속)을 자연 terminal까지 폴링. forward edge만 gate.
terminal
나가는 전이 없는 상태("terminal은 terminal"). task 4종(DONE·FAILED·EXPIRED·CANCELLED) / pipeline 3종(DONE·FAILED·CANCELLED).

관측 · 결과 enum

api_result
호출 결과 enum(PENDING|OK|ERROR).
PENDING (api_result)
DISPATCH 선기록("시도했으나 결과 미정"). 상태 아님.
observed
원시 관측값(RUNNING|SUCCEEDED|FAILED|MET|NOT_MET). canonical — 통합 enum 저장 안 함.
PENDING (파생 verdict)
reconciler가 (kind,observed)에서 파생하는 통합 판정 "진행 중". api_result=PENDING(호출 미완)과 다른 개념(폴 결과 판정).
result (OK|FAIL)
attempt terminal 결과. EXECUTION_TIMEOUT은 별도 값이 아니라 error_code로 표현.
outcome
API 파생 표현(SUCCEEDED|FAILED|EXECUTION_TIMEOUT). result+error_code에서 파생(저장값 아님).
poll_count
관측 run에 접힌 폴 수(RLE). NOT_MET 1000폴 = poll_count=1000.
RLE
Run-Length Encoding. 연속 동일 관측을 1행+count로 접기. CHECK에만(DISPATCH는 호출당 1행).
latency_ms
마지막 폴 지연(overwrite, 누적/평균 아님). DISPATCH PENDING·synthetic은 null.
task_check.kind=CHECK
관측 kind 값. TF job poll·CONDITION_CHECK 조건 확인 둘 다 이 kind(RLE collapse 대상). DISPATCH(호출당 1행)와 구분.
task_check.kind=DISPATCH
dispatch 호출 관측 kind 값. 호출당 1행(RLE 안 함, poll_count=1).

errorCode 카탈로그 (8종)

CALL_TIMEOUT
어느 호출이든 1회 timeout(dispatch/poll/check 공통). task_check 귀속. 호출 1회 실패이지 attempt 직접 실패 아님.
EXECUTION_TIMEOUT
dispatch→terminal 미도달(30분+). attempt 귀속.
TTL_EXPIRED
WAIT_EXTERNAL TTL 초과. status=EXPIRED 파생(행 없음).
IM_REJECTED
dispatch 하드 거부(비-backpressure). attempt 귀속.
CHECK_ERROR
check 관측 실패(비-backpressure). task_check 귀속.
DISPATCH_NO_RESPONSE
recovery_timeout까지 응답 미영속. attempt 귀속(대응 DISPATCH PENDING 잔류).
HANDLER_NOT_FOUND
handler_key 미해결. 즉시 FAILED·fail_count 미소모. synthetic task_check 1행(kind=CHECK, poll_count=1).
JOB_FAILED
TERRAFORM_JOB poll이 job 자체 FAILED 관측. errorCode는 attempt 귀속(관측은 task_check.observed=FAILED).

Definition · Snapshot

pipeline_def_snapshot
생성 시 박제한 실행 구성(write-once, 1:1). 재현 권위.
definition_version
recipe 버전(불변; snapshot에만 보유 — pipeline.definition_version 컬럼은 제거).
spec (jsonb)
snapshot의 resolve된 전체 recipe(name + 순서 있는 task 목록). snake_case.
frozen (박제)

코드 = 실행 권위

버전 정책 (_V1/_V2)
비호환 동작 변경만 append-only _V1/_V2. 호환 bugfix는 현재 배포 코드.
부팅 assert
부팅 시 default recipe handler 등록 검증(defense-in-depth).
pre-deploy CI 게이트
배포 전 prod DB non-terminal handler_key 검증(authoritative — 배포 차단; 부팅 assert보다 강함).
defense-in-depth
부팅 assert가 CI 게이트 뒤의 2차 방어.
definition_key
recipe 식별자(예 AWS_INSTALL). snapshot이 보유.
release version
default recipe의 코드 릴리스 1개. 다중 버전 공존 lifecycle 불요(이력은 snapshot).
promotion
배포 승격. pre-deploy CI 게이트는 promotion 전에 실행돼 깨질 배포를 막는다.
