ADR-022: Install/Delete Pipeline — 완료 검증(postCheck)과 종단 상태 알림

제안됨
제안됨 — 2026-07-01 · ADR-016(도메인)·ADR-021(실행)의 후속

한눈에

두 가지 후속 결정을 다룬다. ① postCheck — 자기 보고 완료 판정과 별개로 end-state를 독립 검증. ② 종단 알림 — 파이프라인 완료/실패/취소를 신뢰성 있게 알리기.

핵심 통찰

종단 전이는 이미 pipeline.status에 durable하게 있다. 그러니 별도 이벤트 저장소(트랜잭션 아웃박스)를 두지 말고, 마커 컬럼 notified_at 하나로 상태에서 알림을 파생하고, 전달은 ADR-021의 claim/lease를 그대로 쓴다. 이벤트가 이미 도메인 행에 있으니 dual-write가 없고, 파이프라인 행에는 lease/fencing이 이미 있으니 별도 relay가 겪던 lease 문제가 애초에 없다.

아웃박스(기각) vs 상태 파생(채택)

✗ 트랜잭션 아웃박스

  • 이벤트가 이미 status에 있는데 두 번째 저장소를 둠
  • relay가 외부 전달에 lease가 필요한데 SKIP LOCKED만으론 락을 외부 호출에 물리거나(ADR-021 §3 위반) 이중 전달
  • 다중 sink·poison·pruner를 새로 떠안음
2 테이블 + 3 루프 + 8 이벤트

✓ 상태 파생 (notified_at)

  • 이벤트 = 상태 → dual-write 없음
  • 알림 대상은 pipeline 행 → ADR-021 claim/lease/two-tx 그대로 재사용 → relay-lease 문제 소멸
  • 운영 알림은 기존 metrics/alerting으로 분리
컬럼 3개 + 스캔 술어 1개

배경

postCheck — "작업이 끝났다" ≠ "인프라가 실제로 원하는 상태다"

ADR-016 §5의 완료 판정 check(attempt, task)은 InfraManager의 자기 보고 결과를 읽어 "job이 성공적으로 끝났는가"를 답한다. 이는 "대상 인프라가 지금 의도한 상태인가"와 다르다:

  • INSTALL: job은 성공 보고했지만 리소스가 eventual-consistency로 아직 조회 불가
  • 부분 적용: apply가 절반만 반영됐는데 job 응답엔 안 드러남(silent partial)
  • DELETE: job은 성공 보고했으나 의존 리소스(PII 저장소 등)가 잔존

즉 자기 보고 신뢰만으론 종단 상태의 실제 정합성을 보장하지 못한다 — end-state를 독립 재확인할 수단이 필요하다.

종단 알림 — 신뢰성 있게 "끝났다" 전달하기

파이프라인이 종단에 도달해도 오퍼레이터·다운스트림에 신뢰성 있게 알릴 경로가 없다. 순진한 두 방법은 모두 틀렸다:

  • 전이 트랜잭션 안에서 알림 호출 → dual-write. 알림 성공+트랜잭션 롤백이면 갈라지고, 느린 알림 서버가 전이를 막는다.
  • 커밋 후 best-effort 알림 → 커밋~알림 사이 크래시로 조용히 유실.

관건은 "전이는 커밋됐는데 알림은 아직"을 durable하게 기억하는 것. 그런데 그 상태는 이미 도메인 행에 있다 — status가 종단이면 "알림 대상"이라는 뜻. 필요한 건 "이미 알렸는가" 마커 하나뿐이다.

규모 (ADR-016/021과 동일)

대상 ~2,000개, 분 단위 job, 단일 조직 내부 도구. 별도 이벤트 저장소·relay·CDC/브로커를 정당화하지 않는다.

결정

1 postCheck는 "게이팅"이 기본, "종단 후 자문"은 필요 시에만

검증이 파이프라인 성공 자체를 게이팅해야 하는지로 배치가 갈린다.

현재 · 기본
(A) 게이팅 postCheck
  • recipe의 마지막 task로 기존 CONDITION_CHECK를 둔다
  • 새 task kind·상태·실패 의미 0 (개념 최소화)
  • 실패 → task FAILED(§6) → projection으로 파이프라인 FAILED(§2)
  • 순수 recipe 규약 — 새 메커니즘 없음
유보
(B) 종단 후 자문 postCheck
  • 종단을 커밋한 뒤 검증(드리프트·장기 eventual-consistency)
  • 종단 부활 금지(§7) — 되돌리지 않고 불일치 신호만
  • 도입 시 §2의 상태-파생 claim-pull 재사용, MISMATCH만(OK 없음)
  • 드리프트 요구 생기기 전까지 코드·스키마 안 만듦

2 종단 알림은 상태에서 파생한다 — 별도 저장소 없음

pipeline에 nullable 컬럼 notified_at 하나. 알림 대상은 쿼리로 파생된다:

status IN ('DONE','FAILED','CANCELLED') AND notified_at IS NULL

"전이는 커밋됐는데 알림은 아직"이 이 술어로 완전히 표현된다 — 별도 이벤트 행 INSERT가 없다. 종단 전이 자체가 durable하므로 dual-write가 원천적으로 없다. payload는 이미 커밋된 pipeline/task 행에서 구성한다(실패 시 실패 task+error_code 포함) — TASK_FAILED를 별도 이벤트로 둘 필요가 없다.

3 알림 전달은 ADR-021의 claim/lease 메커니즘을 재사용 (쿼리는 별개)

핵심 이점: 알림 대상은 pipeline 행이고, 그 행에는 ADR-021의 lease/fencing 컬럼(claimed_by, claimed_until)이 이미 있다. 그래서 알림은 "종단 파이프라인에 대한 또 하나의 클레임 가능한 작업"일 뿐 — 별도 relay가 재발명해야 했던 lease 문제가 애초에 발생하지 않는다.

tx1 · claim 종단+미알림+backoff·lease 게이트 행을 SKIP LOCKED로 집고 fresh UUID lease 스탬프
→
외부 호출 알림 sink에 전달 (트랜잭션 밖, per-call 타임아웃)
→
tx2 · guarded SET notified_at=now() WHERE claimed_by=:token — 성공 시 클레임 해제 
-- tx1: 종단-미알림 파이프라인 claim (RUNNING 스캔과 별개 술어)
SELECT id FROM pipeline
 WHERE status IN ('DONE','FAILED','CANCELLED')
   AND notified_at IS NULL
   AND (notify_next_at IS NULL OR notify_next_at <= now())
   AND (claimed_until IS NULL OR claimed_until < now())
 ORDER BY notify_next_at NULLS FIRST
 LIMIT 1 FOR UPDATE SKIP LOCKED;

ADR-021 §3의 two-transaction split(락을 외부 호출에 안 물림)과 fencing이 그대로 적용된다. 종단 알림은 별도 work-kind로 자체 claim 쿼리를 쓴다(전용 loop 또는 공유 워커의 추가 claim 쿼리). 종단 행에는 READY task·cancel_requested·slot-gate가 없으므로 RUNNING 전용 분기를 타지 않는다.

실패·격리·보장
  • 실패 backoff: 전달 실패 시 notify_attempts++, notify_next_at=now()+backoff(지수, ADR-021 429/503→next_due_at와 동형); 상한 초과 시 크게 밀고 운영 알림으로 승격("가장 오래된 미알림 종단 행 age" 지표로 감시).
  • 워커풀 격리: 종단 알림은 전용 풀/loop 또는 작은 동시성 상한으로 처리 — 느린 sink가 파이프라인 실행을 굶기지 않게. notify 클레임은 runningPipelineCap/slotCap에 계상 안 함.
  • at-least-once: notified_at은 1회 마커(알림 상태 1개)지만 외부 전달은 at-least-once — 중복 가능 → 소비자는 pipeline_id로 멱등 dedupe(파이프라인당 종단 1개).
  • stale straggler 안전: lease 만료 후 되살아난 tx2는 claimed_by 가드에서 no-op(ADR-021 §4).

V1은 단일 논리 sink 가정. 독립 재시도가 필요한 다중 sink가 생기면 그때 per-sink 상태(또는 작은 outbox)를 도입한다.

4 운영 알림(worker-outage/queue-wait)은 이 메커니즘 밖

WORKER_OUTAGE/QUEUE_WAIT는 상태 전이가 아니라 지표 임계에서 나온다(도메인 행 없음). ADR-021이 이미 정의한 지표(lease-expired reclaim count, due-pipeline lag)에 대한 임계 알림으로, 조직이 이미 운영하는 metrics/alerting 스택에서 처리한다. 상태-파생 경로에 억지로 태우면 도메인 행 없는 이벤트를 위해 범용 저장소를 되살려야 하므로 그러지 않는다.

5 보장과 한계 (수용)

  • exactly-once 없음 — at-least-once + 멱등 소비자로 충분(ADR-016 §5와 동일 이유).
  • 소비자 계약 — dedupe 키 pipeline_id 장기 보관; payload에 schema_version; PII 최소화: 허용 필드만(pipeline_id, type, terminal_status, target, 실패 시 failed_task/error_code, schema_version).
  • 알림 지연 = 스캔 주기 — 저지연 필요 시 LISTEN/NOTIFY 힌트를 나중에 얹음.
  • 알림 전용 지표 — 미알림 종단 행 최고 age, 재시도/실패 수, give-up 승격 수.

검토한 대안

종단 알림

대안판정이유
A. 상태 파생 + notified_at 채택 이벤트가 이미 도메인 행에 있어 dual-write 없음; ADR-021 claim/lease/two-tx를 그대로 재사용해 relay-lease 문제 없음; 새 테이블·relay·pruner 0.
B. 트랜잭션 아웃박스(별도 event_outbox + relay) 기각 이벤트가 이미 status에 있어 불필요; relay가 외부 전달에 lease 필요(§3 위반 or 이중 전달); 다중 sink·poison·pruner를 떠안음; ADR-016이 이미 잘라낸 메커니즘.
C. 전이 트랜잭션 내 동기 알림 호출 기각 dual-write; 느린/실패 알림이 전이를 롤백·차단.
D. 커밋 후 best-effort 알림 기각 커밋~알림 사이 크래시로 조용히 유실; 재시도·복구 없음.
E. CDC/브로커(Debezium/Kafka) 기각 규모 대비 운영 비용 과다. 이미 DB 소유 → "상태 스캔 파생"이 같은 아이디어의 경량판.
F. Postgres LISTEN/NOTIFY 부분 채택(선택) 대체는 못 하나(휘발성) 저지연 wake-up 힌트로 얹을 수 있음. 지연 문제 시 도입, V1 불필요.

postCheck

대안판정이유
A. 게이팅=트레일링 CONDITION_CHECK + 종단 후 자문(유보) 채택 흔한 경우는 기존 task kind 재사용(새 개념 0); 자문은 필요 시에만.
B. 새 POST_CHECK task kind 도입 기각 동작이 같은데 enum만 늘림 — 개념 최소화 위반.
C. 종단 후 mismatch 시 파이프라인 FAILED로 되돌림 기각 ADR-016 §7 "종단 부활 금지" 위반. 올바른 신호는 자문 방출.

결과

장점
  • 신뢰성 있는 종단 알림을 dual-write·별도 저장소 없이 — 전달은 ADR-021의 검증된 claim/lease를 탄다.
  • relay-lease 딜레마 원천 소멸 — 알림이 pipeline 행 작업이라 two-tx split·fencing이 그대로.
  • 움직이는 부품 최소: 새 테이블·relay·pruner·taxonomy 없음 → 컬럼 + 스캔 술어/분기.
  • ADR-016 취지와 일치(잘라낸 outbox를 되살리지 않음). 도메인 상태(status·enum) 불변.
감수할 비용
  • pipeline에 알림 메타데이터 컬럼 3개(notified_at/notify_next_at/notify_attempts, ADR-021 실행 컬럼과 동일 범주) + 부분 인덱스 + 워커 분기.
  • at-least-once → 멱등 소비자 필수(pipeline_id).
  • 알림 전달을 실행 워커풀에서 격리해야 함(relay 없앤 대가).
  • 회귀(수용): per-event 감사 추적 상실(파이프라인당 notified_at 1개; 로그/지표로 재구성).
  • 회귀(수용): 종단만·1회성 — 중간 이벤트("시작됨"·"step N 완료")나 성공 내 transient task 실패는 이 경로로 안 나감 → metrics 소관.
  • 자문 postCheck는 유보; 도입 시 같은 lease 1쌍을 공유해 알림과 직렬화(실행과는 무경합).

스키마

종단 알림 (현재 결정) — 모두 ADR-022 소유 알림 메타데이터 컬럼. ADR-021 실행 컬럼과 동일 범주이며 도메인 상태 컬럼이 아니다. lease 토큰(claimed_by/claimed_until)은 ADR-021 것을 공유.

pipeline.notified_at (nullable) — 종단 알림 전달 완료(sink ack) 마커
pipeline.notify_next_at(nullable) — 실패 backoff 게이트(다음 재시도 시각)
pipeline.notify_attempts (int, def 0) — backoff 지수·give-up 임계 계산용
※ 새 테이블 없음. 대상은 status IN(종단) AND notified_at IS NULL 파생.
인덱스: (notify_next_at) WHERE status IN('DONE','FAILED','CANCELLED') AND notified_at IS NULL

종단 후 자문 postCheck — 유보(현재 미구현, 범위 밖; §1(B)의 상태-파생 claim-pull 재사용)

pipeline.post_check_due_at (nullable) — 자문 검증 due 마커(ADR-022 소유)
※ 드리프트 이력 필요 시 결과 1행 UPDATE-in-place (반복 이력 없음)
불변식
  1. 알림/진단 메타데이터는 reconciler·claim·스케줄링·전이의 의미에 관여하지 않는다(claim 술어가 읽는 건 "알림 대상 선별"이지 도메인 전이가 아니다).
  2. 종단 알림 대상은 커밋된 종단 상태에서 파생 — 전이가 durable해야만 대상이 되고, durable하면 반드시 대상(dual-write 없음, 유실 없음).
  3. 알림 메타데이터 손상/롤백은 도메인 상태를 오염시키지 않는다 — 최악은 재전달(멱등 흡수)/재클레임(delay)일 뿐 부정확 아님.

  • 유보 모드 전용: 자문 postCheck 도입 시 종단 후 불일치 신호 1종만 추가(OK 없음).

링크

  • ADR-016 — 얹는 도메인 모델(event outbox를 잘라낸 원출처 — 이 ADR은 그 취지를 이어 상태 파생 채택). 알림 메타데이터 컬럼은 ADR-022 소유.
  • ADR-021 — 알림 전달이 그대로 재사용하는 claim/lease 실행 모델(§2 claim, §3 two-tx, §4 guarded write-back).
  • adr-016-history.md — event outbox 등 최대 모델 요소가 재범위 축소로 정리된 경위.

용어

  • completion check — ADR-016 §5의 check(attempt, task). 자기 보고 결과로 task 완료 판정. postCheck와 다름(자기 보고 vs 독립 end-state 검증).
  • postCheck — end-state 독립 검증. 게이팅(트레일링 CONDITION_CHECK) 또는 종단 후 자문(read-only·부활 금지·불일치 신호만) 두 배치 방식.
  • derive-from-state — 별도 저장소 없이 도메인 행 상태(status 종단 + notified_at IS NULL)에서 알림 대상을 쿼리로 파생. 이벤트가 이미 상태에 있어 dual-write 없음.
  • notified_at — 파이프라인당 종단 알림 전달 완료(sink ack) 마커(알림 메타데이터, 도메인 상태 아님).
  • dual-write — DB write와 외부 부작용을 한 트랜잭션 경계에서 함께 시도해 부분 실패 시 갈라지는 안티패턴. 상태 파생이 이를 제거.
