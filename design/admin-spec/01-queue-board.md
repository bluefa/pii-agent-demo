# 화면 1 — Queue Board (작업 큐 보드)

> 공통 제약·셸·어휘는 `00-overview.md` 참조. (브랜드 #0064FF·한국어·데스크톱 / 좌측 사이드바 + 우측 콘텐츠)

---

## 1. 이 화면은 무엇인가

관리자가 **지금 처리해야 할 승인 작업**을 한 화면에서 발견하고 처리하는 보드.
"오늘 내가 손대야 할 일"을 빠르게 파악 → 클릭 → 승인/반려까지 끝내는 동선.

**처리 작업은 2종류**다:

| 작업 유형 | 언제 생기나 | 관리자가 하는 일 |
|-----------|-------------|-----------------|
| **연동 승인** | TargetSource가 `PENDING`(승인 대기) | 요청 내용을 보고 **승인** 또는 **반려(사유)** |
| **설치 확정** | TargetSource가 `CONNECTED`(연결 테스트 성공) | 설치 결과를 보고 **확정**(→완료) 또는 **반려(사유, →재테스트)** |

---

## 2. 사용자 / 맥락

- 사용자: 운영 관리자. 하루에 여러 번 들러 "쌓인 일"을 처리.
- 진입: 사이드바 첫 메뉴(기본 랜딩). 처리 대기 건수를 배지로 노출해 주의를 끈다.

---

## 3. 화면 구성 (영역)

세 영역으로 구성한다. (배치·스타일은 자유)

### A. 상단 요약 KPI (4개)
한눈에 부하를 보여주는 수치 카드.

| 카드 | 예시 값 | 의미 |
|------|---------|------|
| 연동 승인 대기 | **4건** | PENDING 건수 |
| 설치 확정 대기 | **2건** | CONNECTED 건수 |
| 오늘 처리됨 | **7건** | 오늘 승인+확정 완료 |
| 반려 | **1건** | 오늘 반려 |

- "연동 승인 대기"·"설치 확정 대기"는 **할 일** — 0이 아니면 시선이 가야 함.
- 카드 클릭 시 해당 탭으로 이동(선택).

### B. 탭 + 도구줄
```
[ 연동 승인 대기 ④ ]  [ 설치 확정 대기 ② ]  [ 처리 완료 ]        🔍 검색   ⟳ 새로고침
```
- 탭마다 건수 배지.
- 우측에 검색(서비스/TargetSource ID)과 새로고침.
- **탭에 따라 테이블 컬럼이 다르다**(아래 C).

### C. 작업 테이블 (탭별 컬럼 상이)

**① 연동 승인 대기 탭**

| 컬럼 | 예시 | 비고 |
|------|------|------|
| 서비스 | svc-alpha | |
| TargetSource | ts-aws-001 | |
| Provider | AWS | provider 칩 |
| 관리자 입력 | `완료 ✓` / `미입력 2건 ⚠` | **게이트 표시** — 입력 미완 건 식별 |
| 요청자 | 김영희 | |
| 요청일 | 06-18 | |
| 액션 | `[상세]` | 행 클릭과 동일 |

**② 설치 확정 대기 탭** (컬럼이 다름 — 설치/연결 결과 중심)

| 컬럼 | 예시 | 비고 |
|------|------|------|
| 서비스 | svc-beta | |
| TargetSource | ts-idc-003 | |
| Provider | IDC | |
| 연결 테스트 | `성공 3/3` | 리소스별 연결 결과 요약 |

**③ 처리 완료 탭** — 오늘(또는 최근) 승인/확정/반려된 건 이력. 결과(승인/확정/반려) + 처리자 + 시각.

## 4. 핵심 인터랙션 — 행 클릭 → 처리 모달

행 클릭 시 **상세 모달**이 뜬다. 모달의 역할 = "요약 확인 + 빠른 처리". (보드를 벗어나지 않음)

### 모달 내용 (연동 승인)
- 헤더: 서비스 · TargetSource · Provider · 상태 pill
- **관리자 입력 완성도 칩**: `관리자 입력 완료 ✓` 또는 `미입력 2건 ⚠`
- 요청 리소스 목록(요약): 리소스 ID · DB 타입 · 포함/제외(제외 사유 칩)
- 액션:
  - 게이트 통과 시 → **[승인 ▸]** Primary + **[반려]**
  - 게이트 미통과(필수 입력 누락) 시 → **[승인] 비활성**, 대신 **[상세에서 입력하기 →]** 가 Primary로 승격 (Service 관리 상세로 이동)
  - 어느 경우든 **[상세 페이지로 이동 →]** 링크 제공

### 모달 내용 (설치 확정)
- 헤더 동일
- **연결 테스트 결과 요약**: 성공 3/3 · 06-19 09:12 · 리소스별 성공/실패
- 액션: **[설치 확정 ▸]** Primary + **[반려]**

### 반려
- 반려 클릭 → **사유 입력(필수)** 모달 → 제출.
- 연동 승인 반려 → 상태 `PENDING → IDLE`(서비스 화면에 사유 노출).
- 설치 확정 반려 → 상태 `CONNECTED → INSTALLED`(연결 테스트 재요청).

## 5. 상태 (이 화면이 가져야 할 상태들)

| 상태 | 표현 |
|------|------|
| 로딩 | 테이블 스켈레톤 |
| **빈 상태 (해당 탭 0건)** | "처리할 연동 승인이 없어요 👍" 같은 안내 — 한가함을 긍정적으로 |
| 에러 | 조회 실패 + [다시 시도] |
| 처리 중 | 승인/반려 제출 시 버튼 로딩 → 성공 시 행 제거 + 토스트 |
| 게이트 미통과 행 | 입력 미완 칩 강조, 모달 승인 비활성 |

---

## 6. 디자인 우선순위 (무엇이 가장 중요한가)

1. **할 일 건수가 즉시 보인다** — KPI 카드의 "승인 대기/확정 대기"가 0인지 아닌지 한눈에.
2. **두 작업 유형이 헷갈리지 않는다** — 탭/컬럼/액션이 명확히 분리.
3. **막힌 건(입력 미완)이 구별된다** — 바로 처리 가능한 건 vs 상세로 가야 하는 건.
4. **처리가 빠르다** — 행 클릭 → 모달 → 한 번에 승인/반려.

---

## 7. 도메인 메모 (참고)

- 같은 승인/반려/확정 액션은 **Service 관리 상세 페이지에서도** 가능하다(동일 결과). 큐 보드는 "발견 + 빠른 처리"용 보조 동선, 상세 페이지는 "값 입력까지 포함한 단일 작업대".
- "관리자 입력 게이트": `nlb_index`(IDC)는 승인 전 필수(하드), `agent_id·credential`은 승인 시 경고(소프트)·확정 시 필수(하드). 큐 보드는 이 완성도를 칩으로 노출만 한다.
- 상태 전이 요약:
  - 연동 승인: 승인 → CONFIRMING / 반려 → IDLE
  - 설치 확정: 확정 → COMPLETED / 반려 → INSTALLED
