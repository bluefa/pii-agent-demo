# 화면 2 — Service 관리 (목록 + TargetSource 상세 3탭)

> 공통 제약·셸·어휘는 `00-overview.md` 참조.
> 이 화면은 **하위 화면 5개**로 구성된다. 디자인 도구엔 2A → 2B → 2C … 순으로 하나씩 넘기길 권장.
>
> - **2A. 목록** — 서비스 선택 → TargetSource 목록
> - **2B. 상세 셸** — TargetSource 상세의 공통 헤더 + 3탭 구조
> - **2C. 탭① 연동 워크플로우** — 라이프사이클 + "지금 할 일" + 연동 구성 표(편집)
> - **2D. 탭② 연동 데이터** — 요청/승인/확정 3객체 열람·비교 (읽기 전용)
> - **2E. 탭③ 설치 관리** — 설치/삭제 파이프라인 시작 + 진행 + 이력

---

## 이 화면 묶음은 무엇인가

한 TargetSource(연동 대상 DB 하나)에 대한 **관리자의 모든 작업이 끝나는 단일 작업대**.
값 입력 → 연동 승인 → 확정 입력 → 검증 → 설치 → 설치 확정까지 여기서 처리한다.
(Queue Board는 발견·빠른 처리용 보조 동선이고, 진짜 작업은 여기서.)

# 2A. 목록 (서비스 → TargetSource)

## 구성
좌우 2분할.

```
┌─ 서비스 ─────────┐   ┌─ svc-alpha 의 TargetSource ───────────────────┐
│ 🔍 검색          │   │ Provider   TargetSource     상태       Health  │
│ ● svc-alpha     │   │ AWS        ts-aws-001        COMPLETED  ●정상   ›│
│   svc-beta      │   │ GCP        ts-gcp-002        CONNECTED  ●정상   ›│
│   svc-gamma     │   │ IDC        ts-idc-003        PENDING    -       ›│
└─────────────────┘   │ Azure      ts-az-004         INSTALLED  ●확인중 ›│
                      └─────────────────────────────────────────────────┘
```

- **좌측 서비스 리스트**: 검색 + 선택. 선택된 서비스 강조.
- **우측 TargetSource 테이블**: 선택 서비스의 연동 대상 목록.
  - 컬럼: Provider 칩 · TargetSource ID · **프로세스 상태 pill** · Health(설치된 건만) · 진입 화살표.
  - 행 클릭 → 2B 상세로 이동.
- 상태 pill 색은 상태군 일관(대기=앰버 / 진행=파랑 / 완료=중립·초록 / 이상=빨강).

## 상태
- 빈 상태: 서비스에 TargetSource 0건 → "연동 대상이 아직 없어요".
- 로딩/에러: 공통 패턴.

# 2B. 상세 셸 (공통 헤더 + 3탭)

상세 진입 시 항상 보이는 **헤더 + 탭 바**. 탭 내용만 바뀐다.

```
┌─ Service 관리 › svc-alpha › ts-aws-001 ────────────────────────────────┐
│  ts-aws-001   [AWS]   상태: CONNECTED   Health: 정상   [⟳ 상태 새로고침] │
│                                                                        │
│  [ 연동 워크플로우 ]   [ 연동 데이터 ]   [ 설치 관리 ]                    │
└────────────────────────────────────────────────────────────────────────┘
```

- 헤더: 경로(breadcrumb) · TargetSource ID · Provider 칩 · 현재 상태 pill · Health · 상태 새로고침 버튼.
- 탭 3개. 기본 진입 = **연동 워크플로우**.
- Provider(AWS/Azure/GCP/IDC)에 따라 탭 *내용물*은 달라지지만 헤더·탭 골격은 동일.

# 2C. 탭① 연동 워크플로우 — "지금 무엇을 해야 하나"

이 탭의 목표: **현재 상태가 화면을 결정한다.** 관리자가 "상태 → 할 일"을 머리로 변환하지 않게 한다.
세 블록으로 구성.

## (1) 라이프사이클 스테퍼 + 데이터 인디케이터 (상단 고정)

```
IDLE ─✓ PENDING ─● CONFIRMING ─○ CONFIRMED ─○ INSTALLED ─○ CONNECTED ─○ COMPLETED
        현재
─────────────────────────────────────────────────────────────────────────────
진행 구성: 승인 정보 (승인 06-18 · 김영희)   ┊   확정정보: 있음 ⚠ 불일치   [데이터 보기 →]
```

- 7단계 스테퍼: 지난 단계 ✓ / 현재 강조(브랜드색) / 미래 dim.
- **하단 "데이터 인디케이터"는 단계와 분리해서 표시**:
  - `진행 구성`: PENDING이면 "요청 정보", CONFIRMING 이후면 "승인 정보" (전환형 — 동시에 둘 다는 아님).
  - `확정정보`: `없음` / `있음 ✓ 일치` / `있음 ⚠ 불일치` / `있음 ⚠ 잔존` — **단계와 무관하게 항상 표시**.
- 재요청 사이클(운영 중인데 다시 PENDING)이면 헤더에 **`운영 중 재연동`** 배지.
- 반려 이력 있으면 `반려됨 · 사유 보기` 칩.

## (2) "지금 할 일" 작업 카드 + CTA (상태별로 내용이 바뀜)

현재 상태에 맞는 **단 하나의 카드**가 무엇을 해야 하는지와 Primary 버튼을 제시한다.

| 상태 | 카드 메시지 | 체크리스트 | Primary | Secondary |
|------|------------|-----------|---------|-----------|
| IDLE | "연동 요청이 없어요" (반려 직후면 반려 사유) | — | — | (잔존 확정정보 시) 확정정보 삭제 |
| **PENDING** | "연동 승인 대기 — 승인 전에 관리자 입력을 완료하세요" | 관리자 입력 2/3 · nlb_index 0/2(IDC) | **승인** | 반려 |
| CONFIRMING (확정정보 없음) | "승인 완료 — 승인 정보를 점검하고 확정을 입력하세요" | 미입력 필드 0 확인 | **승인정보대로 확정 입력** | — |
| CONFIRMING (확정정보 잔존·불일치) | "승인정보와 확정정보가 달라요 — 확정정보를 삭제한 뒤 다시 입력해야 해요" | — | **확정정보 삭제** | 데이터 보기 |
| CONFIRMED | 확정 검증 패널 `✓ 승인↔확정 일치 (리소스 3/3 · 필드 21/21)` | — | **설치 시작** (→설치 관리 탭) | 확정정보 삭제 |
| INSTALLED | "설치 완료 — 서비스 담당자의 연결 테스트 대기 중" | — | — | 설치 관리 보기 |
| CONNECTED | "연결 테스트 성공 (3/3 · 06-19 09:12) — 설치 확정 대기" | TC 결과 요약 | **설치 확정** | 반려(→INSTALLED) |
| COMPLETED | "연동 완료" + 요약(확정 일시·리소스 수) | — | — | — |

- 카드는 1개만(현재 상태). 메시지는 평이한 한국어 안내체.
- **강등 경고 모달**: CONFIRMED 이후에 승인 정보를 수정·저장하면 프로세스가 CONFIRMING으로 되돌아간다 →
  저장 시 "저장하면 프로세스가 CONFIRMING으로 돌아가요. 확정정보 삭제 → 재확정 후 설치 확인·연결 테스트·설치 확정을 다시 진행해야 해요." 확인 모달.

## (3) 연동 구성 표 (편집 대상 명시 + 편집 모드)

연동 리소스 목록 + 관리자가 채울 값. **편집 대상이 무엇인지 표 헤더에 명시**한다.

```
┌─ 연동 구성 · 편집 대상: 승인 정보 ───────────────────────────── [편집] ─┐
│ DB     Resource ID      Region   Name      ┊ agent_id   raw    credential │
│ MySQL  db-prod-01… 📋   ap-ne2   orders    ┊ ag-101     true   cred-55    │
│ MySQL  db-prod-02… 📋   ap-ne2   payments  ┊ 미설정 ⚠   true   cred-56    │
│ Redis  cache-01…   📋   ap-ne2   session   ┊ ag-101     false  cred-57    │
└──────────────────────────────────────────────────────────────────────────┘
              ↑ 읽기(서비스 담당자 정보)            ↑ 관리자 편집 영역 (옅은 tint로 구분)
```

- **편집 대상 라벨**: PENDING → `편집 대상: 연동 요청 정보`, CONFIRMING 이후 → `편집 대상: 승인 정보`.
- 좌측 = 읽기 컬럼(DB 타입·Resource ID·Region·Name·포함/제외). 우측 = 관리자 편집 컬럼(옅은 배경으로 구분).
- **Provider별 편집 컬럼이 다름**:
  - **AWS/Azure/GCP**: `agent_id` · `does_support_raw`(토글) · `db_credential`
  - **IDC**: `nlb_index`(= Source IP) — 별도 [설정] 모달로 선택 (아래 §IDC)
- 미입력 값은 모두 **`미설정 ⚠`** 표기 (입력 완성도 가시화).
- 제외 리소스는 행을 흐리게 + 제외 사유 칩, 관리자 컬럼 비활성.
- **편집 방식 = 편집 모드 + 일괄 저장**: `[편집]` → 관리자 컬럼 전체가 입력 상태 → 변경 셀 하이라이트 → `[저장]` 1회로 일괄 커밋. (셀마다 모달 띄우지 않음)
- 같은 값을 여러 리소스에 넣는 `agent_id`는 컬럼 헤더 메뉴 `모든 리소스에 적용` 제공(선택).

### IDC 전용 — nlb_index 설정
IDC는 편집 컬럼이 `nlb_index` 하나. 값은 직접 타이핑이 아니라 **NLB 테이블에서 선택**한다.

```
미설정 행의 [설정] 클릭 → 모달:
┌─ NLB Index 선택 ───────────────────────────────┐
│ Index   IP List                점유 Listener     │
│ ○ #2    10.0.1.3, 10.0.1.4     12               │
│ ● #3    10.0.1.5, 10.0.1.6     4   ← 여유 많은 순 │
│ ○ #4    10.0.1.7               0                │
│                          [취소]   [선택 적용]     │
└────────────────────────────────────────────────┘
```
- 점유 Listener 적은 순 정렬 + 기본 선택 추천.
- nlb_index는 **연동 승인 전 필수**(미설정이면 승인 버튼 비활성 + 사유 표시).

# 2D. 탭② 연동 데이터 — 3객체 열람·비교 (읽기 전용)

3-객체 모델(요청 → 승인 → 확정)을 **열람**하고, 승인↔확정을 **비교**하는 탭. 편집 없음.

```
[ 요청 정보 (참고) ]  [ 승인 정보 ]  [ 확정정보 ]   |   [ 비교 ]

┌─ 승인 정보 ── 생성 06-18 14:02 · 연동 승인 시 요청정보에서 승계 ──────────┐
│ (리소스 스냅샷 테이블 — 읽기 전용)                                        │
└──────────────────────────────────────────────────────────────────────────┘

[비교] (승인 ↔ 확정 — CONFIRMED 이후에만 활성):
│ 리소스       필드        승인 정보    확정정보            │
│ db-prod-01  agent_id    ag-101      ag-101  ✓           │
│ ▸ 전체 21개 필드 일치 ✓                                  │
```

- 서브 탭 4개: **요청 정보(참고)** · **승인 정보** · **확정정보** · **비교**.
- 각 객체 헤더에 생성 시각·생성 경위 표기(요청: 서비스 담당자 / 승인: 요청정보 승계 / 확정: 승인정보 복사).
- **미생성 객체는 빈 상태**: "승인 정보가 아직 없어요 — 연동 승인 시 생성돼요" → 모델 학습 장치.
- **요청 정보는 참고용 읽기 전용** (반려·재요청 맥락 조사용).
- **비교는 승인 ↔ 확정 한 쌍만**. CONFIRMED 이후에만 의미. 그 전(불일치·잔존)엔 diff 대신
  "확정정보는 수정 불가 — 삭제 후 다시 입력" 안내 카드.

# 2E. 탭③ 설치 관리 — 설치/삭제 파이프라인

이 TargetSource의 **설치·삭제 실행과 모니터링**. 4블록.

## (1) 설치/삭제 시작 + 수행 가능한 파이프라인 미리보기
```
┌─ 설치 관리 ───────────────────────────────────────────────────────────┐
│ 현재 설치 상태: 설치됨 · Health 정상            [⟳ 상태 확인]            │
│                                                                       │
│ 수행 가능한 동작:   [ 설치 시작 ▸ ]   [ 삭제 시작 ▸ ]                    │
│                                                                       │
│ ▾ 설치 시 수행할 단계 (AWS 자동):                                       │
│   ○ TF 권한 확인 →  ○ SVC TF →  ○ BDC Common TF →  ○ BDC Service Level │
│      (← 파이프라인 화면과 동일한 노드 플로우 컴포넌트로 미리보기)         │
└───────────────────────────────────────────────────────────────────────┘
```
- **설치 시작 / 삭제 시작** 버튼. 누르면 해당 파이프라인 생성 확인 모달 → 실행.
- **수행할 단계 미리보기(DAG)**: 실제로 어떤 Task를 순서대로 도는지 *읽기용 노드 플로우*로 보여준다.
  - **반드시 파이프라인 화면(`03-pipeline.md`)의 노드 플로우와 동일한 컴포넌트**를 재사용한다(미리보기 상태 = 전부 '대기').
  - Provider/동작별 단계는 §도메인 메모 참조.
- 진행 중 파이프라인이 있으면 새 동작 버튼은 비활성(중복 방지).

## (2) 진행 중 파이프라인 (있을 때만)
- 현재 실행 중인 설치/삭제가 있으면: 진행률 + 현재 Task + 경과 + **[파이프라인에서 보기 →]**(03 화면으로).
- 노드 플로우로 현재 위치 표시.

## (3) Pipeline 동작 이력
이 TargetSource에서 과거에 실행된 설치/삭제 파이프라인 목록.

| 컬럼 | 예시 |
|------|------|
| 동작 | 설치 / 삭제 |
| 결과 | 완료 ✓ / 실패 ✕ / 취소 |
| 단계 | 4/4 |
| 시작 | 06-15 10:02 |
| 소요 | 21분 |
| → | 상세 보기 |

- 행 클릭 → 03 화면의 파이프라인 상세(모달).
- 빈 상태: "아직 실행된 파이프라인이 없어요".

---

## 상태 (이 화면 묶음 전체)
- 로딩 / 빈 상태 / 에러: 공통 패턴.
- 편집 중(저장 전): 변경 셀 하이라이트 + 저장/취소 노출, 다른 탭 이동 시 경고.
- 위험 액션(확정정보 삭제·강등·삭제 파이프라인): 결과를 명확히 고지하는 확인 모달.

## 디자인 우선순위
1. **현재 상태가 곧 화면** — 2C의 작업 카드가 "지금 할 일"과 Primary를 분명히.
2. **어디서 고치고 어디서 확인하는가** — 워크플로우(편집) vs 데이터(열람)의 역할 분리가 명확.
3. **편집이 가볍다** — 편집 모드 + 일괄 저장, 셀 모달 왕복 없음.
4. **설치 미리보기는 기존 노드 플로우 재사용** — 새 비주얼 요소를 만들지 말 것.

## 도메인 메모 (참고)
- **Provider별 설치 단계(DAG)** — 미리보기·진행에 쓰임:
  - AWS(자동): TF 권한 확인 → SVC TF → BDC Common TF → BDC Service Level TF
  - AWS(수동): SVC TF 설치 확인 → BDC Common TF → BDC Service Level TF
  - Azure: AzureVM 설치 확인 → PE 승인 요청 → PE 승인 확인
  - GCP: TF 권한 확인 → Subnet 생성 확인 → SVC TF → BDC TF
  - IDC: BDC CX TF → BDP TF
