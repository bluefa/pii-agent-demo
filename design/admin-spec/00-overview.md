# PII Agent — Admin 콘솔 디자인 명세 · 개요

> **목적**: 이 폴더는 운영 관리자(Admin) 콘솔의 화면을 Claude Design / Figma 등 디자인 도구로
> 생성하기 위한 **핸드오프 명세**다. 각 파일은 한 화면(또는 한 흐름)을 독립적으로 다루며,
> 디자인 도구에 하나씩 붙여넣어 시안을 만들고 → 화면을 보며 피드백하는 워크플로우를 전제로 한다.
>
> **이 명세가 정하는 것**: 각 화면이 *무엇을 보여주고 / 어떤 액션이 있고 / 어떤 상태를 가지는가*.
> **이 명세가 정하지 않는 것**: 구체적 레이아웃·색·간격·컴포넌트 스타일 — 이건 도구가 탐색하고
> 사용자가 화면을 보며 결정한다. (아래 §2 "고정 vs 자유" 참고)

---

## 1. 한 줄 정의

PII Agent는 서비스의 DB(연동 대상)에 PII 탐지 에이전트를 설치·연동하는 제품이다.
**서비스 담당자**는 별도 화면(SIT 연동, 7단계 프로세스)에서 자기 연동을 진행하고,
**운영 관리자(Admin)** 는 이 콘솔에서 *모든 서비스의 연동 승인·값 관리·설치 자동화·운영 모니터링*을 한다.

Admin 콘솔은 서비스 담당자 화면과 **완전히 분리된 별도 페이지**다.

## 2. 고정 vs 자유 (디자인 도구 제약)

### 🔒 고정 (반드시 지킬 것)

| 항목 | 값 |
|------|-----|
| **브랜드 컬러** | Primary `#0064FF` (파랑). 강조·주요 액션·활성 상태의 기준색 |
| **언어** | 한국어 UI (문구는 본 명세의 한국어를 그대로 쓰거나 자연스럽게 다듬기) |
| **대상 기기** | 데스크톱 전용 (가로 ≥ 1280px). 모바일·반응형 불필요 |
| **톤** | 운영 도구 — 신뢰감 있고 밀도 높은 정보 표시. 장난스럽지 않게 |

### 🎨 자유 (도구가 탐색 → 사용자가 보고 결정)

레이아웃, 여백·간격, 그림자·깊이, 라운드, 타이포 스케일, 컴포넌트 스타일(테이블/카드/칩/버튼/모달),
색 팔레트(브랜드 파랑 위에 얹는 보조색·상태색), 아이콘, 애니메이션 — **모두 자유**.

> 핵심: 사용자는 "무엇을 보여줄지"는 알지만 "어떻게 보여야 멋진지"를 말로 표현하기 어렵다.
> 그래서 이 명세는 기능·정보·상태에 집중하고, 비주얼은 도구가 제안 → 화면을 보며 좁혀간다.

## 4. 도메인 어휘 (디자이너가 알아야 할 최소한)

화면 곳곳에 등장하는 개념. 깊이 이해할 필요는 없고, "이런 데이터가 있다" 정도면 충분하다.

- **서비스(Service)** — 고객사/팀 단위. 예: `svc-alpha`. 여러 TargetSource를 가진다.
- **TargetSource** — 연동 대상 DB 하나. 예: `ts-aws-001`. **Provider** 와 **상태**를 가진다.
- **Provider** — 인프라 종류: **AWS · Azure · GCP · IDC** (+ SDU). Provider마다 연동·설치 절차가 다르다.
- **프로세스 상태 (7단계)** — TargetSource의 연동 진행 단계:
  `IDLE → PENDING(승인 대기) → CONFIRMING(승인됨) → CONFIRMED(확정됨) → INSTALLED(설치됨) → CONNECTED(연결 테스트 성공) → COMPLETED(연동 완료)`
- **3-객체 모델** — 연동 정보는 세 단계 산출물로 존재한다:
  **① 요청정보**(서비스 담당자가 요청) → **② 승인정보**(관리자가 승인 시 승계) → **③ 확정정보**(실제 인프라에 반영된 스냅샷, immutable).
- **파이프라인(Pipeline) / Task** — 설치·삭제를 자동화하는 작업 묶음. Pipeline 1개 = 설치 또는 삭제 1회 실행.
  내부는 여러 **Task**(단계)로 구성되고 **순차 실행**된다. Task는 *직접 실행형*과 *외부 대기형* 2종이 있다.
- **확인주기 / TTL / 실패 횟수** — Task 운영 파라미터: 외부 대기를 얼마나 자주 확인하는지(≥10분),
  최대 얼마나 기다리는지(TTL), 몇 번 실패하면 멈추는지(max_fail_count).

> 더 깊은 도메인·상태 규칙은 각 화면 파일의 "도메인 메모"와 원본 `design/admin-page-requirements.md` 참조.
> 디자인 단계에서는 위 어휘만으로 충분하다.

## 5. 공통 컴포넌트 / 패턴 (화면 간 일관성)

여러 화면에서 반복되는 요소. 같은 의미는 같은 비주얼로 표현한다(도구가 한 번 정하면 전체 재사용).

| 패턴 | 쓰임 | 비고 |
|------|------|------|
| **상태 pill** | 프로세스 상태·Task 상태·Pipeline 상태 라벨 | 상태군마다 색 일관 (진행=차분 / 대기=앰버 / 실패=빨강 / 완료=중립·초록) |
| **Provider 칩** | AWS/Azure/GCP/IDC 식별 | provider별 고정 색/마크. 화면 전체 동일 |
| **KPI 카드** | 상단 요약 수치 (건수·비율·시간) | Queue·Pipeline·API 화면 공통 |
| **데이터 테이블** | 목록(큐 행·서비스·파이프라인·API) | 정렬·필터·검색·행 클릭 진입. 밀도 높되 읽기 쉽게 |
| **빈 상태(empty)** | 데이터 0건 | 무엇이 없고 어떻게 채우는지 안내 |
| **모달** | 상세 확인·승인/반려·파라미터 입력·위험 액션 확인 | 위험 액션(삭제·강등)은 결과를 명확히 고지 |
| **사유 입력** | 반려 시 사유 필수 입력 | 텍스트 영역 + 제출 |
| **확인주기 표시** | 외부 대기 카운트다운 "다음 확인 N분 후" | Pipeline·Service 설치 탭 |

### 공통 상태 4종 (모든 데이터 화면이 가져야 함)
1. **로딩** — 조회 중
2. **빈 상태** — 데이터 없음(+ 다음 행동 안내)
3. **에러** — 조회 실패(+ 재시도)
4. **정상** — 데이터 표시

## 7. 디자인 도구 사용법(권장)

1. 먼저 이 `00-overview.md`를 도구에 전달해 **공통 셸·제약·어휘**를 인지시킨다.
2. 그다음 화면 파일 하나(`01`~`05`)를 전달해 해당 화면 시안을 생성한다.
3. 시안을 보고 피드백 → 같은 파일 기준으로 반복.
4. 화면 간 공통 컴포넌트(§5)는 한 번 정해진 스타일을 다른 화면에도 재사용하도록 지시한다.

> 각 화면 파일은 자체적으로 읽어도 되도록 핵심 맥락을 반복 수록했다(도구에 단독 전달 가능).
