## 1. 배경 / 목표

서비스 담당자용 SIT 화면(v13)과 별개로, **운영 관리자(Admin)** 가 사용하는 콘솔이 필요하다.
관리자의 일은 크게 4가지다:

1. **Task Queue Board** — 처리해야 할 승인 작업을 한눈에 보고 승인/반려
2. **Service 관리** — 서비스 → TargetSource 목록 → 상세. provider별로 다른 값(nlb_index, agent_id 등)을 승인 전/후에 설정·수정, 확정정보 조회/삭제/재입력, 설치 관리
3. **리소스 설치·삭제 파이프라인** — Terraform 작업을 DAG 파이프라인으로 자동화하고 현황 모니터링
4. **API Dashboard** — API 응답시간 통계와 최근 실패 내역 확인
5. **가이드 관리** — 프로세스 단계별 가이드 콘텐츠(ko/en) 편집 (기존 Guide CMS 설계 편입)

Admin 콘솔은 **서비스 담당자 화면과 완전히 분리된 별도 페이지**다.
v13 프로토타입에서 상단 "관리자" navigation 클릭 시 **아예 다른 페이지로 전환**되는 구조로 처리한다 (결정 #1).

## 2. 현황 파악 요약

### 2.1 이미 존재하는 것 (app/ + swagger 기준)

| 영역 | 현황 | 근거 |
|------|------|------|
| 승인 Queue | `GET /task-admin/approval-requests` + `app/integration/task_admin` 페이지. PENDING / IN_PROGRESS / APPROVED / REJECTED 탭, 승인·반려(사유) 모달 | `task_dashboard.yaml` |

| 설치 완료 확정 | `POST /target-sources/{id}/pii-agent-installation/confirm` (CONNECTED 상태에서 호출) | `confirm.yaml` |
| 확정정보 모델 | 3-객체 모델: 확정정보(=Infra Manager의 현재 TF 상태, 수정 API 없음) / 승인 요청 / 승인 완료 | ADR-006 |

| Provider별 설치 상태 | aws/azure/gcp/idc/sdu 각각 `installation-status`(GET, 캐시) + `check-installation`(POST, 강제 갱신) | 각 provider yaml |
| 서비스 측 대기 지점 | AWS 수동 TF 실행 / Azure PE 승인·VM TF / GCP TF 권한 / IDC 방화벽 — 문서화됨 | `cloud-provider-states.md` |
| Credential 수정 | `PUT /target-sources/{id}/resources/credential` (resourceId + credentialId) | `confirm.yaml` |
| TestConnection | 비동기: `POST /test-connection` → 202 → `GET .../latest` 폴링 | `test-connection.yaml` |
| 가이드 CMS | `GET/PUT /admin/guides/{name}` (ko/en 콘텐츠), Admin UI 설계 기존 존재 | `guides.yaml` |

### 2.2 존재하지 않는 것 (이번 스펙에서 신규 정의)

- ❌ **NLB table API** — nlb_index / IP List / Listener 점유 수. swagger·도메인 문서 모두 없음
- ❌ **ResourceId 기준 admin 값 수정 API** — agent_id, does_support_raw (credential만 PUT 존재)
- ❌ **확정정보 삭제 / "승인정보대로 확정 입력"** 액션
- ❌ **설치/삭제 작업 Queue** — 현재는 Infra Manager API를 순차 수동 호출
- ❌ **API 메트릭** — admin_dashboard.yaml은 연동 KPI만 있고 응답시간/실패 통계 없음

### 4.2 Task Queue Board

#### 작업 유형 2종

| 유형 | 대상 상태 | 액션 | 의미 |
|------|----------|------|------|
| **연동 승인** | `PENDING` (승인 대기) | 승인 / 반려(사유) | 미래 상태(연동 대상 구성)를 승인 |

#### 4.3.2 TargetSource 관리 상세 — 탭 2개: 연동 정보 관리 | 설치 관리 (결정 #4)

#### 4.3.5 IDC 전용 — nlb_index 설정

- 백엔드의 `nlb_index` = SIT 화면의 Source IP. **연동 승인 전에 설정**해야 함
- nlb table: `Index / IP List(최대 2) / 현재 점유 Listener 수`. API Call로 값이 갱신됨
- 확정 시점에 nlb table의 비어 있는 값을 업데이트하는 구조

```
│  ── ① 리소스 값 관리 (IDC) ───────────────────────────────────│
│  │ 구분        연동 대상         Port   nlb_index             ││
│  │ [Single]   10.20.30.40      3306   #3 (10.0.1.5, 10.0.1.6) ✎ ││
│  │ [Domain]   db.svc-a.io      5432   미설정 ⚠ [설정]          ││
│                                                                │
│  nlb_index 설정 모달:                                           │
│  ┌─ NLB Index 선택 ────────────────────────────────┐           │
│  │ Index  IP List              점유 Listener        │           │
│  │ ○ #2   10.0.1.3, 10.0.1.4   12                  │           │
│  │ ● #3   10.0.1.5, 10.0.1.6   4   ← 여유 많은 순?  │           │
│  │ ○ #4   10.0.1.7             0                   │           │
│  │                              [취소] [선택 적용]   │           │
│  └─────────────────────────────────────────────────┘           │
```

- 미설정 리소스가 있으면 승인 차단? → **Q3-c 답변 필요**

### 4.4 리소스 설치/삭제 파이프라인 (명칭: 결정 #9)

> 명칭 결정: "Queue"는 쌓이는 대기열 뉘앙스라 실체와 다름. 실체는 **정해진 단계(DAG)를 스케줄에 따라
> 순차 실행 + 외부 승인 대기(manual gate) + 실패 시 중단/재시도** — CI/CD **파이프라인** 멘탈 모델과 일치.
> 이하 작업 Set = **Pipeline**, 개별 작업 = **Task** 로 부른다.

#### 4.4.1 개념 모델 — DAG 기반 (결정 #5)

Pipeline은 **DAG(Directed Acyclic Graph)** 구조로 정의하되, **실행은 엄격 순차** 다 (결정 #14) —
동시에 RUNNING인 Task는 1개뿐이며, 선행 Task가 DONE이 되어야 다음 Task가 시작된다. 병렬 진행은 없다.

```
Pipeline (설치/삭제 1회 실행 단위, 구 JobSet)
 ├─ 트리거: TargetSource 상세의 [설치 시작] / [삭제 시작]
 ├─ 생성 시 provider별 Task DAG가 한 번에 등록
 │   (삭제 Pipeline은 설치 DAG의 역방향, BDC Common 제외)
 ├─ Pipeline 상태: QUEUED / RUNNING / DONE / FAILED / CANCELLED
 └─ Task
     ├─ 유형 2종
     │   · EXECUTE        — 시스템이 직접 수행 (Infra Manager Terraform API 호출)
     │   · WAIT_EXTERNAL  — 서비스 측 액션 대기 (확인주기마다 상태 체크)
     ├─ depends_on: [선행 Task ID...]  — 비어 있으면 즉시 실행 가능
     └─ 수정 가능 속성 (결정 #6, #8): 관리자가 Task 상세에서 수정 가능 (DONE/RUNNING 제외)
         · polling 주기(확인주기) — ≥10분 가드
         · TTL(최대 체류시간)
         · max_fail_count(최대 허용 실패 횟수) — ∞ 허용 (확인성 Task는 무한 권장)
```

> 모든 provider DAG(§4.4.2)는 선형 체인이며, 실행 모델도 순차만 가정한다 (결정 #14 — 병렬 진행 없음).
> Q4-e(병렬 갈래 실패 처리)는 이 결정으로 해당 없음 처리.

#### 4.4.2 Provider별 Task DAG — ✅ 확정 (결정 #7)

`→` 는 의존성(선행 완료 후 실행). Task 유형(EXECUTE/WAIT_EXTERNAL)은 AI 추정이므로 틀린 곳이 있으면 정정 필요.

**설치 DAG**

| Provider | 설치 DAG |
|----------|------------|
| AWS (자동) | ① TF 권한 확인 `WAIT_EXTERNAL` → ② SVC TF `EXECUTE` → ③ BDC Common TF `EXECUTE` → ④ BDC Service Level TF `EXECUTE` |
| AWS (수동) | ① SVC TF 설치 확인 `WAIT_EXTERNAL` → ② BDC Common TF `EXECUTE` → ③ BDC Service Level TF `EXECUTE` |
| Azure | ① AzureVM 설치 확인 `WAIT_EXTERNAL` (DB-only여도 생략 없이 확인 — 결정 #13) → ② PE 승인 요청 `EXECUTE` → ③ PE 승인 확인 `WAIT_EXTERNAL` |
| GCP | ① TF 권한 확인 `WAIT_EXTERNAL` → ② Subnet 생성 확인 → ③ SVC TF 동작 `EXECUTE` → ④ BDC TF 동작 `EXECUTE` |
| IDC | ① BDC CX TF `EXECUTE` → ② BDP TF `EXECUTE` — **방화벽 오픈 확인은 파이프라인 범위 밖** (기존 수동 확정 유지, 결정 #13) |

**삭제 DAG** — 설치의 역순. 단, **BDC Common TF는 삭제하지 않는다** (결정 #7).

| Provider | 삭제 DAG |
|----------|------------|
| AWS (자동) | ① BDC Service Level TF 삭제 → ② SVC TF 삭제 (BDC Common 유지) |
| AWS (수동) | ① BDC Service Level TF 삭제 → ② SVC TF 삭제 확인 (BDC Common 유지) |
| Azure | ① PE 삭제 → ② (AzureVM은 서비스 측 자산 — 삭제 범위 외 추정) |
| GCP | ① BDC TF 삭제 → ② SVC TF 삭제 |
| IDC | ① BDP TF 삭제 → ② BDC CX TF 삭제 |

> 현재 확정된 provider DAG는 모두 선형이지만, 모델 자체는 병렬 갈래를 지원한다(§4.4.1).

**확인 필요 (잔여)**:
- 삭제 DAG의 세부 단계(특히 Azure·수동 AWS)는 위 추정이 맞는가?

#### 4.4.3 Task 상태 머신 — 실패 카운트 기반 (결정 #8)

시도가 실패하면 `fail_count`를 올리고, **`max_fail_count` 이내면 자동 재시도**(다음 스케줄에),
초과하면 그때 FAILED로 확정된다. 확인성 Task(TF 권한 확인 등)는 `max_fail_count = ∞` 가능.

```
            ┌─────────┐
 QUEUED ──→ │ RUNNING │ ──→ DONE ──→ (후행 Task 실행 가능)
            └─────────┘ ──→ (시도 실패) fail_count++
                              ├─ fail_count ≤ max_fail_count → 재시도 대기 (다음 주기)
                              └─ 초과 → FAILED → Pipeline 중단, 관리자 [재시도]/[중단]
 QUEUED ──→ WAITING_EXTERNAL ──(확인주기마다 체크, 미충족은 실패로 안 침)──→ DONE
                            └──(TTL 초과)──→ EXPIRED(타임아웃) ──→ Pipeline FAILED (결정 #11)
                                              └─ 관리자 [재시도(타이머 리셋)] / [중단]
```

> 구분 주의: WAIT_EXTERNAL의 "아직 안 됨"(외부 액션 미완)은 실패가 아니라 대기 — TTL로만 관리.
> "확인 API 호출 자체가 에러"인 경우가 fail_count 대상이다.

#### 4.4.4 정책 (초안)

| 항목 | 초안 | 비고 |
|------|------|------|
| 확인주기(polling 주기) | Task별 기본값 보유, **관리자가 Task 상세에서 수정 가능** (결정 #6). **10분 미만 불가** 가드 | DONE/RUNNING Task는 수정 불가 |
| 최대 체류시간(TTL) | WAIT_EXTERNAL Task별 설정 (예: 7일), Task 상세에서 수정 가능 | 초과 시 EXPIRED |
| TTL 초과 처리 | Task **EXPIRED(타임아웃) → Pipeline FAILED** (결정 #11). 보드에 경고, 관리자가 [재시도(타이머 리셋)] / [중단] | 보류(ON_HOLD) 상태 폐기 |
| 최대 실패 횟수(max_fail_count) | Task별 설정·수정 가능, **∞ 허용** (결정 #8). 기본값 확정 (결정 #12): EXECUTE TF = 1, 확인성(WAIT/권한 확인) = ∞ | |
| 실패 처리 | fail_count ≤ max → 다음 주기에 자동 재시도. 초과 → FAILED, 관리자가 로그 확인 후 [재시도(카운트 리셋)] / [중단] | Q4-b는 본 정책으로 통합 해소 |
| 실행 순서 | **엄격 순차** (결정 #14) — 동시에 RUNNING Task 1개, 병렬 진행 없음 | |
| 중복 방지 | 동일 TargetSource에 진행 중 Pipeline 있으면 신규 생성 차단 | |
| 설치↔삭제 충돌 | 진행 중 설치 Pipeline이 있는데 삭제 요청 → 차단 (중단 후 삭제만 허용) | |

## 9. 연동 정보 관리 UX 개선안 (R2 — 검토용)

> 배경: v14 프로토타입 사용 리뷰에서 "연동 정보 관리"의 답답함이 확인됨.
> 연동 승인 프로세스는 `IDLE → PENDING → CONFIRMING → CONFIRMED` 로 진행되며, 단계마다
> **연동 요청 정보 → 승인 정보 → 확정정보** 라는 산출물(3-객체, ADR-006)이 생긴다.
> 관리자는 요청/승인 정보에 값(agent_id, does_support_raw, nlb_index 등)을 채워 넣고,
> 승인 정보를 토대로 확정한 뒤 **확정정보에 값이 제대로 들어갔는지 확인**할 수 있어야 한다.
> 현행 화면은 이 모델을 표현하지 못한다.

### 9.0 도메인 모델 — 상태와 3객체의 관계 (✅ R2 Q&A로 확정)

| 상태 | 요청정보 | 승인정보 | 확정정보 |
|------|---------|---------|---------|
| IDLE | X | X | 있을 수 있음 — 레거시 (process 개념 도입 전 데이터) |
| PENDING | O | X | 있을 수 있음 — 재요청 사이클 (기존 연동 Target에 신규 연동 요청) |
| CONFIRMING | 비활성 (데이터는 뒷단에 보존) | O | X, **또는** O인데 승인과 다른 상태 |
| CONFIRMED | 비활성 | O | O + 승인과 일치 |

**확정 룰:**

1. **승계** — 연동 승인 시 요청정보의 값(agent_id, nlb_index 등)은 승인정보로 그대로 승계된다. 요청정보 레코드는 삭제되지 않고 뒷단에 보존된다 (프로세스상 비활성일 뿐).
2. **확정정보는 immutable** — 생성 경로는 「승인정보대로 확정 입력」 단 하나. 수정(Update) 불가, 확정정보가 이미 있으면 확정 입력도 불가 → 변경은 반드시 **삭제 후 재생성**.
3. **불일치 → CONFIRMING 강등** — 확정 생성 후 승인정보를 수정하거나, 잔존 확정정보가 있는 채로 새 사이클이 승인되면 불일치가 되고, **어느 단계(INSTALLED/CONNECTED/COMPLETED 포함)에 있든 CONFIRMING으로 복귀**한다. 이후 재확정 → 설치 확인 → TestConnection → 설치 확정을 다시 밟는 것은 **의도된 운영 비용**이다.
4. **확정정보 ≠ 단계 산출물** — 확정정보는 Infra Manager에 반영된 인프라 상태 객체로, 프로세스 단계와 독립적으로 존재할 수 있다 (IDLE/PENDING에서도 가능). confirmStatus에 영향을 주는 요소지만 절대적 요소는 아니다.
5. **검증 표시 원칙** — 상태 판정은 백엔드 책임. CONFIRMED에서는 frontend가 승인·확정 GET 2회 후 **표시용으로만** 비교한다. CONFIRMING + 확정 존재 상태에서는 필드 비교가 무의미 — 해소 경로가 「확정정보 삭제 → 승인정보 기준 재입력」 하나로 결정되어 있기 때문. diff 없이 액션 유도만 한다.

| P2 | **확정 검증 수단 부재** | "승인정보대로 확정 입력" 후 값이 제대로 들어갔는지 비교할 방법이 없음. 확정정보는 접힌 토글 뒤에 있고, 편집 표와 컬럼 구성마저 달라(agent_id/raw/cred ↔ Host/Port/Credential) 눈 대조도 불가 | 높음 |
| P3 | **액션 분산** | 승인/반려·설치 확정은 큐 보드 모달에만, 값 편집·확정 입력은 상세 페이지에만 → 한 건 처리에 화면 왕복 강제 | 높음 |
| P4 | **관리자 입력 완성도 비가시** | 미입력 집계가 없음 (IDC nlb_index의 `미설정 ⚠`만 존재). 실제 운영은 빈 값에서 시작하므로 agent_id 등 누락 위험 | 중간 |
| P5 | **편집 인터랙션 비용** | 셀마다 ✎ → 모달 → 저장. 리소스 N개 × 필드 3종이면 모달 왕복 3N회 | 중간 |
| P6 | **이력 부재** | 누가 언제 어떤 값을 바꾸고 승인·확정·삭제했는지 추적 불가 | 중간 |

### 9.2 개선 원칙

1. **상세 페이지 = 단일 작업대** — 한 TargetSource에 대한 관리자 행동 전부(값 입력 → 승인 → 확정 입력 → 검증 → 설치 → 설치 확정)가 상세 페이지에서 끝난다. 큐 보드는 발견 + 게이트 통과 건의 빠른 처리용 보조 동선.
2. **단계가 화면을 결정한다** — process-status가 "지금 할 일" 카드와 Primary CTA를 결정한다. 관리자가 상태 → 할 일을 머리로 변환하지 않게 한다.
3. **전이마다 검증 가능한 산출물** — 요청/승인/확정 각 객체를 열람·비교할 수 있게 하고, 특히 승인 ↔ 확정은 자동 대조한다.

### 9.4 연동 워크플로우 탭

#### 9.4.1 라이프사이클 스테퍼 + 데이터 인디케이터 (상단 고정)

```
┌──────────────────────────────────────────────────────────────────────────┐
│ IDLE ──● PENDING ──○ CONFIRMING ──○ CONFIRMED ──○ INSTALLED ──○ CONNECTED ──○ COMPLETED │
│           현재                                          [운영 중 재연동]    │
├──────────────────────────────────────────────────────────────────────────┤
│ 진행 구성: 요청 정보 (요청자 김영희 · 06-10)  ┊  확정정보: 있음 ⚠ 잔존        │
└──────────────────────────────────────────────────────────────────────────┘
```

- 현재 단계 강조(provider accent), 지난 단계 ✓, 미래 단계 dim
- **데이터 인디케이터(하단 줄)는 단계와 분리** — §9.0 모델 반영:
  - `진행 구성`: PENDING이면 요청 정보, CONFIRMING 이후면 승인 정보 (**전환형** — 동시에 존재하지 않음). 클릭 시 연동 데이터 탭으로
  - `확정정보`: 없음 / 있음 ✓ 일치 / 있음 ⚠ 불일치 / 있음 ⚠ 잔존(IDLE·PENDING) — **단계와 무관하게 항상 표시** (§9.0 룰 4)
- 재요청 사이클(PENDING + 잔존 확정정보)이면 헤더에 `운영 중 재연동` 배지 — 실제 인프라 연동은 살아있는 채로 프로세스만 앞 단계로 돌아갔음을 알림
- 반려 이력이 있으면 해당 위치에 `반려됨 · 사유 보기` 칩

#### 9.4.2 상태별 작업 카드 + CTA — "지금 할 일"

| 상태 | 카드 메시지 | 체크리스트 | Primary | Secondary |
|------|------------|-----------|---------|-----------|
| IDLE | "연동 요청이 없어요" (반려 직후면 반려 사유 표시). 잔존 확정정보 있으면 레거시 안내 카드 (§9.5) | — | — | (잔존 시) 확정정보 삭제 |
| PENDING | "연동 승인 대기 — 승인 전에 관리자 입력을 완료하세요". 잔존 확정정보 있으면 `운영 중 재연동` 안내 추가 | 관리자 입력 N/M 리소스 · nlb_index N/M (IDC) | **승인** | 반려 |
| CONFIRMING ① 확정정보 없음 | "승인 완료 — 승인 정보를 점검하고 확정을 입력하세요" | 미입력 필드 0건 확인 | **승인정보대로 확정 입력** | — |
| CONFIRMING ② 확정정보 있음 (불일치·잔존) | "승인정보와 확정정보가 다른 상태예요 — 확정정보를 삭제한 뒤 다시 입력해야 해요" (필드 diff 없음 — §9.0 룰 5) | — | **확정정보 삭제** | 데이터 보기 |
| CONFIRMED | 확정 검증 패널 (§9.5 — frontend 비교 ✓ 요약) | 승인 ↔ 확정 일치 확인 | **설치 시작** (→ 설치 관리 탭) | 확정정보 삭제 |
| INSTALLED | "설치 완료 — 서비스 담당자의 연결 테스트를 기다리는 중" | — | — | 설치 관리 보기 |
| CONNECTED | "연결 테스트 성공 (성공 3/3 · 06-11 09:12) — 설치 확정 대기" | TestConnection 결과 요약 | **설치 확정** | 반려 (→ INSTALLED) |
| COMPLETED | "연동 완료" + 요약 (확정 일시·리소스 수) | — | — | — |

- **승인/반려/설치 확정을 상세 페이지에 추가** (P3 해소). 큐 보드 모달과 동일 액션·동일 API — 어디서 처리해도 결과 동일
- 반려는 기존 사유 모달(admRejectModal) 재사용
- **강등 경고 모달**: CONFIRMED 이후 단계에서 승인 정보를 수정·저장하면 프로세스가 CONFIRMING으로 복귀하므로(§9.0 룰 3), 결정 #10의 안내를 한 단계 강화 — "저장하면 프로세스가 CONFIRMING으로 돌아가요. 확정정보 삭제 → 재확정 후 설치 확인·연결 테스트·설치 확정을 다시 진행해야 해요."

#### 9.4.3 연동 구성 표 — 편집 대상 명시 + superset 컬럼

- 표 헤더에 **편집 대상 라벨**: PENDING → `편집 대상: 연동 요청 정보`, CONFIRMING 이후 → `편집 대상: 승인 정보` (P1 해소의 절반). CONFIRMED 이후 단계에서 저장하면 강등 모달(§9.4.2)을 거친다
- 컬럼은 **서비스 담당자가 보던 정보의 superset** + 관리자 전용 필드(시각적으로 구분 — 옅은 배경 tint):

| 영역 | cloud (AWS/Azure/GCP) | IDC |
|------|----------------------|-----|
| 공통 (읽기) | Database Type · Resource ID · Region · Resource Name · 포함 여부(제외 사유 칩) | 구분 · Database Type · 연동 대상(Host) · Port · 포함 여부 |
| 관리자 (편집) | agent_id · does_support_raw · db_credential | nlb_index (Source IP) |

- 미입력 값은 모두 `미설정 ⚠` 표기 — 현행 IDC nlb 패턴을 전 관리자 필드로 확장 (P4 해소)
- 제외 리소스는 현행 `row-excluded` + 사유 칩 유지, 관리자 필드는 비활성

```
┌─ 지금 할 일 ──────────────────────────────────────────────────────────┐
│ ◉ 연동 승인 대기                                                       │
│   승인 전에 관리자 입력을 완료하세요.                                     │
│   ☐ 관리자 입력  2/3 리소스 완료 — agent_id 1건 미설정                    │
│                                              [반려]      [승인 ▸]      │
└───────────────────────────────────────────────────────────────────────┘
┌─ 연동 구성 · 편집 대상: 연동 요청 정보                        [편집]      │
│ DB      Resource ID     Region   Name      ┊ agent_id   raw    credential │
│ MySQL   db-prod-01… 📋  ap-ne2   orders    ┊ ag-101     true   cred-55  │
│ MySQL   db-prod-02… 📋  ap-ne2   payments  ┊ 미설정 ⚠    true   cred-56  │
│ Redis   cache-01…   📋  ap-ne2   session   ┊ ag-101     false  cred-57  │
└───────────────────────────────────────────────────────────────────────┘
```

#### 9.4.4 승인·확정 게이트 (✅ Q6-d 확정, 결정 #24 — 구 Q3-c 통합 해소)

필드별로 "필수가 되는 시점"을 분리한다:

| 필드 | 승인 시점 | 확정 시점 |
|------|----------|----------|
| nlb_index (IDC) | **하드 게이트** — 미배정 시 승인 비활성 + "nlb_index 미배정 N건" 사유 표시 ("승인 전 설정" 명시 요건) | — |
| agent_id · db_credential | 소프트 게이트 — 승인 클릭 시 "미입력 N건, 그래도 승인할까요?" 확인 모달 | **하드 게이트** — 미입력 시 확정 입력 비활성 |
| does_support_raw | 기본값(false) 보유 — 게이트 없음 | — |

### 9.5 확정정보 카드 — 상태별 표시 원칙 (P2 해소, Q6-c ✅ 결정 #20)

상태 판정(CONFIRMING/CONFIRMED)은 백엔드 책임이고, 프론트의 비교는 **표시용 보조 확인**이다.
확정정보가 immutable(§9.0 룰 2)이므로 상태에 따라 보여줄 것이 다르다:

**① CONFIRMED (및 이후 단계)** — frontend가 승인·확정 GET 2회 후 필드 단위 비교한 검증 패널 상주:

```
┌─ 확정 검증 ─────────────────────────────────────────────────┐
│ ✓ 승인 정보와 확정정보가 일치해요 — 리소스 3/3 · 필드 21/21    │
│                                          [필드 비교 보기 →]   │
└─────────────────────────────────────────────────────────────┘
```

(모델상 CONFIRMED ⟺ 일치이므로 ⚠는 예외 상황 — 표시상 차이가 보이면 경고 + [확정정보 삭제 후 다시 입력] 유도)

**② CONFIRMING + 확정정보 존재 (불일치·잔존)** — **필드 diff를 보여주지 않는다.**
해소 경로가 「삭제 → 승인정보 기준 재입력」 하나로 결정되어 있어 비교가 의미 없기 때문 (§9.0 룰 5):

```
┌─ 확정정보 불일치 ───────────────────────────────────────────┐
│ ⚠ 승인정보와 확정정보가 다른 상태예요.                         │
│   확정정보는 수정할 수 없어요 — 삭제한 뒤 승인정보 기준으로      │
│   다시 입력해야 해요.                                         │
│   [데이터 보기 →]                     [확정정보 삭제]          │
└─────────────────────────────────────────────────────────────┘
        └─ 삭제 후 카드가 CONFIRMING ①로 전환 → [승인정보대로 확정 입력]
```

**③ IDLE/PENDING + 잔존 확정정보 (레거시·재요청)** — "기존 연동의 확정정보가 남아 있어요" 안내 카드 + [조회] [삭제].
재요청 사이클이면 승인 후 어차피 CONFIRMING ②로 합류하므로 이 시점에 삭제를 강제하지 않는다.

- 비교는 기존 GET 2회(approved/confirmed-integration) + 클라이언트 비교 — **신규 API 불필요**

### 9.6 연동 데이터 탭 — 3객체 열람·비교

```
[ 요청 정보 (참고) ]  [ 승인 정보 ]  [ 확정정보 ]  |  [ 비교 ]

┌─ 승인 정보 ─── 생성 06-10 14:02 · 연동 승인 시 요청정보에서 승계 ─┐
│ (approval-table 스냅샷 — 읽기 전용)                                │
└───────────────────────────────────────────────────────────────────┘

비교 (승인 ↔ 확정 — CONFIRMED 이후에서만 활성):
│ 리소스         필드         승인 정보      확정정보               │
│ db-prod-01    agent_id     ag-101        ag-101 ✓               │
│ ▸ 전체 21개 필드 일치 ✓                                           │
```

- 미생성 객체는 빈 상태로: "승인 정보가 아직 없어요 — 연동 승인 시 생성돼요" → 객체 모델의 학습 장치 역할
- 각 객체 헤더에 생성 시각·생성 경위(요청: 서비스 담당자 / 승인: 요청정보 승계 / 확정: 승인정보 복사) 표기
- **요청 정보는 참고용 읽기 전용** (결정 #21) — 승인 후에도 "원래 뭘 요청했는지" 조사용으로 열람 가능 (뒷단 보존 데이터, §9.0 룰 1)
- **비교 쌍은 승인 ↔ 확정 하나뿐** — 요청↔승인은 승계 모델이라 승인 직후 항상 동일하므로 제공하지 않음. CONFIRMING + 확정 존재 상태에서 비교 진입 시 diff 대신 §9.5 ② 안내 카드를 보여줌

### 9.7 편집 방식 — 편집 모드 + 일괄 저장 (P5 해소, ✅ Q6-b 확정, 결정 #23)

- `[편집]` → 관리자 컬럼 전체가 입력 상태로 전환 (text input / does_support_raw 토글 / nlb [선택] 버튼)
- 변경된 셀은 하이라이트, `[저장]` 1회로 일괄 커밋 — 확정정보 존재 시 경고 모달도 세션당 1회만
- 같은 값을 여러 리소스에 쓰는 agent_id는 컬럼 헤더 메뉴 `모든 리소스에 적용` 제공 (선택 사항)
- 효과: 모달 왕복 3N회 → 편집 세션 1회

### 9.8 큐 보드 연계

- 상세 모달(admDetailModal)에 **관리자 입력 완성도 칩** 추가: `관리자 입력 완료 ✓` / `미입력 2건 ⚠`
- 게이트 미통과 건은 모달의 [승인] 비활성 + **[상세에서 입력하기 →]가 Primary로 승격** — 모달의 역할을 "요약 확인 + 게이트 통과 건의 빠른 처리"로 명확화
- 보드 행에도 완성도 칩을 노출해 바로 처리 가능한 건을 식별 (선택 사항)

### 9.9 시나리오

**S1. AWS 신규 연동 (해피 패스)**

1. 서비스 담당자가 리소스를 선택해 연동 요청 → `PENDING`. 요청 정보 생성
2. 관리자: 큐 보드 "연동 승인 대기" 뱃지 증가 → 행 클릭 → 모달에 `미입력 3건 ⚠` → [상세에서 입력하기 →]
3. 상세 워크플로우 탭: 스테퍼 PENDING 강조, 작업 카드 체크리스트 `관리자 입력 0/3`
4. 연동 구성 표 [편집] → agent_id 3건 입력(공통값은 "모든 리소스에 적용") · credential 확인 · raw 토글 → [저장]
5. 체크리스트 전부 ✓ → [승인] → `CONFIRMING`. 스테퍼 전진, 승인 정보 칩 ✓ 점등
6. 작업 카드 "승인 정보를 점검하고 확정을 입력하세요" → 표의 편집 대상 라벨이 `승인 정보`로 변경 → [승인정보대로 확정 입력]
7. 확정 검증 패널 자동 표시: `✓ 리소스 3/3 · 필드 21/21 일치` → `CONFIRMED`
8. Primary [설치 시작] → 설치 관리 탭에서 파이프라인 진행 → 완료 시 `INSTALLED`
9. 서비스 담당자가 TestConnection 수행 → 성공 시 `CONNECTED`
10. 관리자: 큐 보드 "설치 확정 대기" → 상세 카드에 TC 결과(성공 3/3 · 시각) → [설치 확정] → `COMPLETED`

**S2. IDC — nlb_index 배정 (승인 전 필수)**

1. `PENDING`. 체크리스트 `nlb_index 0/2 배정 ⚠ (승인 전 필수)` — [승인] 비활성 (하드 게이트)
2. 표의 [설정] → NLB 모달(점유 Listener 적은 순 정렬) → 배정 × 2건 → 게이트 해제 → [승인]
3. 확정 입력 시점에 nlb table의 빈 값이 시스템에 의해 업데이트됨

**S3. 운영 중 값 정정 — 강등 → 재확정 (§9.0 룰 3)**

1. `COMPLETED` 운영 중, db-prod-02의 agent_id를 바꿔야 함 → 연동 구성 표 [편집] → 저장 시 **강등 경고 모달**: "저장하면 프로세스가 CONFIRMING으로 돌아가요. 확정정보 삭제 → 재확정 후 설치 확인·연결 테스트·설치 확정을 다시 진행해야 해요."
2. 저장 → 승인정보 수정 → **CONFIRMING 복귀**. 스테퍼 후퇴 + 확정정보 칩 `있음 ⚠ 불일치`
3. 작업 카드 = CONFIRMING ② (§9.5): diff 없이 "삭제 후 재입력" 안내 → [확정정보 삭제]
4. 카드가 CONFIRMING ①로 전환 → [승인정보대로 확정 입력] → `CONFIRMED` + 검증 패널 `✓ 일치`
5. 설치 확인 → TestConnection → 설치 확정 재수행 (의도된 운영 비용)

**S4. 반려 2종**

- PENDING 반려: 사유 입력 → `IDLE`. 스테퍼에 `반려됨 · 사유 보기` 칩, 서비스 화면에 사유 노출(기존 동작)
- CONNECTED 반려(설치 확정 반려): 사유 입력 → `INSTALLED`. 작업 카드가 "연결 테스트 재요청됨 — 서비스 담당자 대기"로 전환 (결정 #2)

**S5. 승인 후 값 수정 (CONFIRMING 단계)**

1. 승인 후 확정 입력 전에 agent_id 오기재 발견 → 표에서 바로 수정 (편집 대상: 승인 정보)
2. [승인정보대로 확정 입력] 시 수정값이 그대로 확정에 반영 — 별도 재승인 불필요
3. 확정정보가 이미 있는 상태라면 S3 경로(삭제 → 재입력)로 합류

**S6. 재요청 사이클 — 기존 연동 Target에 신규 연동 요청 (§9.0 룰 4)**

1. `COMPLETED` 운영 중 서비스 담당자가 구성 변경(리소스 추가 등) 연동 요청 → 프로세스 `PENDING` 복귀. 기존 확정정보는 잔존, 헤더에 `운영 중 재연동` 배지
2. 관리자: 요청 검토 + 관리자 값 입력 → [승인] → `CONFIRMING` — 잔존 확정정보와 신규 승인정보가 다른 상태 (CONFIRMING ② 카드). 이 시점까지 잔존 확정정보 삭제는 강제하지 않음 (§9.5 ③)
3. [확정정보 삭제] → [승인정보대로 확정 입력] → `CONFIRMED` → 설치 파이프라인 → TestConnection → 설치 확정
4. 사이클 내내 기존 인프라 연동은 물리적으로 유지 — 화면은 단계 복귀(스테퍼)와 인프라 잔존(확정정보 칩·배지)을 구분해 표현

**S7. 레거시 — 확정정보만 있는 IDLE**

1. process 개념 도입 전 데이터: `IDLE`인데 확정정보 존재
2. 작업 카드: "연동 요청이 없어요" + 잔존 확정정보 안내 카드 [조회] [삭제] (§9.5 ③)
3. 신규 연동을 시작하려면 서비스 담당자의 연동 요청부터 — 이후 S6 흐름에 합류

### 9.10 변경 이력 탭 — ❌ 이번 라운드 제외 (✅ Q6-e 확정, 결정 #22)

신규 이력 API(`GET /admin/target-sources/{id}/history`)가 전제라 이번 범위에서 제외한다.
P6(이력 부재)은 유효한 문제로 남아 있으며, 백엔드 이력 적재가 준비되면 후속 라운드에서
타임라인(값 변경 old→new · 상태 전이 · 확정 액션 · 파이프라인 이벤트, 행위자·시각 포함) 설계를 진행한다.
