# IDC Provider 연동 플로우 요구사항

> 대상: `design/SIT Prototype Athena v12.html` (→ v13 반영 예정)
> 상태: **확정** (Q1~Q4 사용자 답변 반영 완료)
> 작성일: 2026-06-10

---

## 1. 배경 / 목표

IDC(온프레미스) 인프라는 AWS/Azure/GCP와 **본질적으로 동일한 7단계 프로세스**를 따른다.
차이는 단 세 가지:

1. **Infra Scan이 없다** — 연동 대상 DB를 사용자가 직접 입력한다 (버튼 → 모달).
2. **리소스 식별자가 다르다** — Resource ID / Resource Name이 없고, IP·Domain 기반으로 식별한다.
3. **설치 단계가 다르다** — `BDC 측 리소스 설치 → 방화벽 확인` 2단계로 구성된다.

기존 7단계 프로세스 바(pbar)는 그대로 유지하되, Step 1과 Step 4의 내용물만 IDC 분기로 교체한다.

## 3. 상세 요구사항

### 3.1 Step 1 — 연동 대상 DB 입력 (Scan 대체)

- 카드 제목: `연동 대상 DB 입력`
- 부제: `IDC 인프라는 자동 스캔이 지원되지 않아요. 연동할 DB 접속 정보를 직접 입력해주세요.`
- 헤더 우측 버튼 2개 — 기존 `Run Infra Scan`/`Last Scan` 위치 대체:
  - `기존 연동 요청 정보 불러오기` (Outline/Secondary)
  - `+ 연동 대상 추가` (Primary)
- 화면 상태:

| 상태 | 조건 | UI |
|------|------|-----|
| 빈 상태 | 입력 0건 | 일러스트 + "연동 대상을 추가해주세요" + 안내 문구 |
| 목록 | 입력 ≥ 1건 | 테이블 + 하단 `연동 대상 승인 요청` 버튼 (기존과 동일) |

- 행 단위 액션: 승인 요청 전까지 **수정 / 삭제** 허용 — 행 hover 시 액션 아이콘 노출. 수정은 「연동 대상 추가」 모달을 해당 값으로 채워 재사용.

#### 등록 건수

- **등록 한도 없음** — `연동 대상 추가` 버튼은 항상 활성 (결정 #46, 기존 5건 한도 폐지).
- 목록은 **기존 pagination-row 패턴**(표시 건수 select + 페이지 이동)을 그대로 제공.

#### 「기존 연동 요청 정보 불러오기」 동작

1. 클릭 시 **확인 모달** 노출. 모달에는 **불러올 기존 연동 정보의 미리보기 목록**을 함께 보여준다 (구분 / 연동 대상 / DB Type) — 목록은 **pagination으로 제공**.
2. 경고 문구: `현재 입력한 정보는 모두 사라지고 아래 기존 연동 정보를 불러옵니다.`
3. `불러오기` 확정 시 현재 입력 목록을 **전부 대체**하고 기존 연동 요청의 대상 목록으로 갱신 (row-in 애니메이션).
4. **불러올 정보가 없으면**: 모달 내 빈 상태 문구(`불러올 기존 연동 정보가 없어요`) + `불러오기` 버튼 **disabled**.

### 3.2 「연동 대상 추가」 모달

입력 순서(상→하):

| # | 필드 | 타입 | 규칙 |
|---|------|------|------|
| 1 | 입력 방식 | segmented `[ IP | Domain ]` | 기본값 **IP**. 전환 시 입력값 초기화 confirm 불필요(초안) |
| 2-a | IP (IP 모드) | text + `+ IP 추가` | 기본 1칸. `+` 클릭 시 칸 추가 + **주의 배너 노출**. 추가 칸은 `×`로 제거 가능. IPv4 형식 검증. **끝 공백 문자 감지 시 경고 + 저장 차단** |
| 2-b | Domain (Domain 모드) | text 1칸 | 단일 도메인만. FQDN 형식 검증. **끝 공백 경고** 동일. 상시 안내 배너: "Web Server가 아닌 **DB에 대한 주소**를 입력해야 해요. DB에 Domain을 연결하는 것은 **DB IP가 유동적으로 변경되는 경우에만** 권장돼요." |
| 3 | Port | number | 1–65535, 필수. Multiple IP여도 **단일 입력만 가능** (확정) |
| 4 | Database Type | select | 기존과 동일: MySQL / PostgreSQL / Oracle / MSSQL / MariaDB / MongoDB / Redis |
| 5 | Oracle SID | text | **Oracle 선택 시에만 노출 + 필수**. 그 외 DB Type에서는 미노출. Multiple IP여도 **단일 입력만 가능** |

- 주의 배너 (IP 2개 이상일 때만):
  > ⚠ 여러 IP 등록은 멀티 노드 구성(예: Oracle RAC)에서만 권장돼요. 가능하면 **단일 IP**로 등록해주세요.
- CTA: `[취소]` `[추가]` — 필수값 미충족 시 `추가` disabled.
- 검증 실패는 필드 인라인 에러로 표시 (toast 금지).

### 3.3 연동 리소스 목록 (IDC 컬럼 세트)

Resource ID / Region / Resource Name 컬럼을 제거하고 아래로 대체.
Step 1·4·5 등 목록이 노출되는 모든 단계에서 동일 컬럼 세트 사용 (단계별 상태 컬럼만 상이 — 기존 패턴 동일).

| 컬럼 | 내용 | 비고 |
|------|------|------|
| 구분 | `Single` / `Multiple IP` / `Domain` 뱃지 | IP 1개 → Single, 2개 이상 → Multiple IP, 도메인 → Domain |
| 연동 대상 | host(IP 또는 domain)만 — **Port 미포함** | 컬럼 폭 220px 고정. 긴 host는 **Resource ID와 동일 패턴**(ellipsis + copy-on-hover + hover 전체값 floating tooltip). Multiple IP는 기본 접힘 + `IP N개 더보기 ▾ / 접기 ▴` 토글 |
| Port | 별도 컬럼 (80px) | 연동 대상 셀에 합쳐 표기하지 않음 |
| Database Type | 기존과 동일 뱃지 | Oracle은 뱃지 아래 별도 줄에 `SID: …` 표기. SID는 50자 이상일 수 있으므로 **ellipsis(말줄임) + hover 시 전체값 tooltip** |
| Source IP | BDC Agent 출발지 IP **1~2개** + ⓘ | **Step 2부터 노출** (Step 1에는 컬럼 없음). 2개일 땐 셀 안에 두 줄로 스택 표시, 각각 복사 버튼. **tooltip**: 방화벽 등록 안내 (아래 3.4) |
| 액션 | 수정 / 삭제 아이콘 | Step 1 한정, 행 hover 시 노출 |
| (단계별 상태) | Step 1: 연동 완료 여부 / Step 4: 방화벽 상태 등 | 기존 패턴 유지 |

### 3.3-1 연동 제외 + 제외 사유 (Step 1)

> 상태: **확정** (구현 완료 — 제외 행은 Step 2·3에 비대상 pill + 사유 칩으로 노출, Step 4~7은 연동 대상만. 결정 #39)

IDC도 입력한 대상을 연동에서 제외할 수 있어야 하며, 제외 시 사유가 필수다.

#### 동작 규칙

| # | 규칙 |
|---|------|
| 1 | Step 1 목록에 체크박스 복원. 기본 전체 체크(=연동 대상) |
| 2 | 체크 **해제** 시 해당 행에 즉시 **사유 선택 popover** 노출: `임시DB` / `StageDB` / `DevDB` / `사유 직접 입력` |
| 3 | 사유를 선택해야만 제외가 확정. **선택 없이 popover를 닫으면(외부 클릭/ESC) 체크박스 자동 복귀** (제외 불가) |
| 4 | `사유 직접 입력` 클릭 → 텍스트 입력 모달. 저장해야 제외 확정, 취소하면 체크 복귀 |
| 5 | 제외 확정된 행: 흐림 처리(row-excluded) + 제외 사유 셀에 **Step 3과 동일한 reason-chip** (ⓘ + 사유, hover 시 floating tooltip로 전체 사유 + 등록자·일시) |

#### 방화벽 확인 상세 모달

「방화벽 확인」 task 클릭 시 노출. **연동 대상 1건 = 1행** — `Source IP → 연동 대상` 매핑이 한 행에 들어가고, 행을 IP별로 펼치지 않는다. 건수가 많아지면 목록과 동일하게 pagination 제공.

```
┌─ 방화벽 확인 ──────────────────────────────────────────────── ✕ ─┐
│  Source IP → 연동 대상 방화벽 오픈 여부를 확인합니다.                │
│                                                                   │
│  Source IP        →   연동 대상              오픈 여부             │
│  ────────────────────────────────────────────────────────────     │
│  172.16.0.11      →   10.20.30.40:3306      [방화벽 오픈]          │
│  172.16.0.11      →   10.20.31.10:1521 +2   [방화벽 오픈되지 않음]  │
│  172.16.0.12                                                      │
│  172.16.0.12      →   db.svc-a.io:5432      [방화벽 오픈]          │
│                                                                   │
│                                                  [  닫기  ]       │
└───────────────────────────────────────────────────────────────────┘
```

- Source IP가 2개인 행은 셀 안에 두 줄 스택 (목록과 동일한 표현).
- Multiple IP 연동 대상은 대표 IP + `+N` 표기 (hover 시 전체 IP tooltip) — 목록과 동일.
- 상태 뱃지는 **2종뿐**: `방화벽 오픈`(green) / `방화벽 오픈되지 않음`(red).
  - 한 행에 경로가 여러 개(Source IP 2개, 대상 IP 다수)면 **모든 경로가 열려야** `방화벽 오픈`, 하나라도 닫혀 있으면 `방화벽 오픈되지 않음`.
