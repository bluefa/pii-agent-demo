# AWS 설치 UX 요구사항 정리 (조치 중심)

> 작성일: 2026-03-02
> 목적: 설치 상태 상세 노출 대신, 사용자 조치 필요 여부 중심으로 AWS 설치 UX를 정리

## 1. 확정된 전제

1. 모드(`AUTO`/`MANUAL`)에 따라 다른 UX를 만들지 않는다.
2. `AUTO` 모드에서 Service 측 Terraform Script 실행 주체는 BDC다.
3. Service 측 상태는 완료/미완료 중심으로만 신뢰 가능하다.
4. BDC 상세 상태(`RUNNING/SUCCESS/FAIL`)는 사용자에게 직접 노출하지 않는다.

## 2. UX 원칙

1. 화면의 핵심 질문은 두 가지로 제한한다.
   - 서비스 담당자가 조치할 내용이 있는가
   - BDC 측 설치가 아직 필요한가
2. 내부 구현 상태(스크립트 실행 중, 실패 코드)는 운영/개발용 정보로 간주하고 기본 화면에서 숨긴다.
3. Resource 단위 표시는 단순화한다.
   - `설치 확인중`
   - `설치 완료`

## 3. 공통 상태 모델

### 3.1 내부 신호(비노출)

- Service Script: `NOT_COMPLETED | COMPLETED | UNKNOWN`
- BDC Script: `PENDING | RUNNING | SUCCESS | FAIL`

### 3.2 사용자 노출 상태(핵심)

- 서비스 담당자 조치: `필요 | 불필요`
- BDC 설치 필요 여부: `필요 | 불필요`
- Resource 설치 상태: `설치 확인중 | 설치 완료 | 확인 필요`

### 3.3 매핑 규칙

| 조건 | 서비스 담당자 조치 | BDC 설치 필요 | Resource 표시 |
|---|---|---|---|
| Service Script 미완료 존재 | 필요 | 필요(또는 대기) | 설치 확인중 |
| Service Script 모두 완료 + BDC != SUCCESS | 불필요 | 필요 | 설치 확인중 |
| Service Script 모두 완료 + BDC == SUCCESS | 불필요 | 불필요 | 설치 완료 |
| Service/BDC 상태 불명확(UNKNOWN 등) | 필요(확인) | 필요(확인) | 확인 필요 |

### 4.2 상단 요약 카드 규칙

```text
[서비스 담당자 조치 카드]
- 필요: 미완료/확인필요 Service Script가 1개 이상
- 불필요: 모든 Service Script가 완료

[BDC 설치 카드]
- 필요: BDC가 아직 최종 완료되지 않음
- 불필요: BDC 완료 확인됨
```

### 4.3 리소스 상태 표현 규칙

```text
설치 확인중  : Service 미완료 또는 BDC 미완료
설치 완료    : Service 완료 + BDC 완료
확인 필요    : 스크립트 상태를 신뢰할 수 없는 경우(UNKNOWN 등)
```

설명:
- BDC는 사실상 단일 Script이므로 같은 시점의 여러 Resource가 동일 상태를 가질 수 있다.
- 사용자에게는 내부 실행 상태가 아니라, "지금 조치가 필요한지"와 "설치 완료 확인이 끝났는지"만 노출한다.

## 5. 채택 결정

1. Terraform Script 중심 UI는 **대안 B(Service Accordion + Resource 패널)** 으로 확정한다.
2. 모드(`AUTO`/`MANUAL`)에 따른 화면 분기는 두지 않는다.
3. BDC 상세 상태(`RUNNING/SUCCESS/FAIL`)는 사용자 화면에서 숨기고, 상단 요약 카드(`필요/불필요`)만 노출한다.
4. API는 변경하지 않는다. 기존 설치 상태 API 응답을 화면 매핑만 변경해서 사용한다.

## 6. 채택안(B) 레이아웃 상세

### 6.1 메인 구조

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 서비스 담당자 조치: 필요   |   BDC 설치: 필요   | [상태 다시 확인]          │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ ┌────────────────────────────┐  ┌─────────────────────────────────────────┐ │
│ │ Service Scripts             │  │ 선택된 Script 리소스                    │ │
│ │                            │  │                                         │ │
│ │ ▾ svc-vpc-apne2            │  │ Script: svc-vpc-apne2                  │ │
│ │   [서비스 측 확인 필요]     │  │                                         │ │
│ │   리소스 2개                │  │ ┌──────────┬──────────────┬───────────┐ │ │
│ │ ▸ svc-athena-glue          │  │ │ Resource │ 설치 상태     │ 조치      │ │ │
│ │   [설치 완료]               │  │ ├──────────┼──────────────┼───────────┤ │ │
│ │   리소스 1개                │  │ │ RDS-A    │ 설치 확인중   │ 가이드    │ │ │
│ │ ▸ svc-dynamodb-role        │  │ │ RDS-B    │ 설치 확인중   │ 가이드    │ │ │
│ │   [확인 필요]               │  │ └──────────┴──────────────┴───────────┘ │ │
│ │   리소스 1개                │  │                                         │ │
│ └────────────────────────────┘  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 6.2 좌측 Script Accordion 규칙

1. 각 row는 `terraformScriptName` 단위로 묶는다.
2. row 정보: Script명, Service 상태 라벨, 포함 Resource 수, 조치 버튼(조건부).
3. 기본 펼침 우선순위:
   - `서비스 측 확인 필요`
   - `확인 필요`
   - 그 외 첫 번째 Script
4. 클릭 시 우측 Resource 패널이 해당 Script 기준으로 갱신된다.

### 6.3 우측 Resource 패널 규칙

1. 선택된 Script에 매핑된 Resource만 표시한다.
2. Resource 상태 라벨:
   - `설치 확인중`
   - `설치 완료`
   - `확인 필요`
3. 조치 버튼:
   - `설치 확인중`: `[설치 가이드]`
   - `확인 필요`: `[상태 확인]`
   - `설치 완료`: 버튼 없음

## 7. 상태 매핑 규칙 (채택안 B)

### 7.1 Script 상태 라벨 (Service 중심)

| Service Script 내부 상태 | 화면 라벨 |
|---|---|
| `NOT_COMPLETED` | 서비스 측 확인 필요 |
| `COMPLETED` | 설치 완료 |
| `UNKNOWN` | 확인 필요 |

### 7.2 Resource 상태 라벨

| Service Script | BDC 요약 상태 | Resource 라벨 |
|---|---|---|
| `COMPLETED` | 불필요(완료) | 설치 완료 |
| `COMPLETED` | 필요 | 설치 확인중 |
| `NOT_COMPLETED` | 필요/불필요 무관 | 설치 확인중 |
| `UNKNOWN` | 필요/불필요 무관 | 확인 필요 |

### 7.3 BDC 실패 처리 정책

1. BDC 내부 실패(`FAIL`)는 상세 코드를 노출하지 않는다.
2. 상단 `BDC 설치` 카드는 `필요`로 유지한다.
3. Resource 라벨은 `확인 필요`로 승격한다(오해 방지 목적).

## 8. 구현 범위 (MVP)

1. `terraformScriptName` 기준 Service Accordion UI 추가
2. 선택 Script 기준 Resource 패널 연동
3. 상단 요약 카드 2개(`서비스 담당자 조치`, `BDC 설치`) 유지
4. BDC 상세 상태 문구/배지 제거 (요약 필요 여부만 유지)
5. 기존 API 재사용 (엔드포인트/응답 스키마 변경 없음)

## 9. 검증 기준

1. Script 클릭 시 우측 Resource 목록이 정확히 변경된다.
2. 동일 Script 매핑 Resource들이 같은 기준으로 상태 계산된다.
3. BDC 완료 전에는 `설치 완료`가 잘못 노출되지 않는다.
4. BDC 실패 시 Resource가 `설치 확인중`이 아닌 `확인 필요`로 표시된다.
5. 모드(`AUTO`/`MANUAL`) 변경과 무관하게 화면 레이아웃/정보 구조가 동일하다.
6. 기존 설치 상태 API만으로 화면 렌더링이 가능하다.

## 10. 비채택 대안 기록

1. 대안 A(Service만 노출): Resource 가시성이 부족해 비채택
2. 대안 C(Service+BDC 상세): 정보량 과다로 비채택

## 11. 간소화안 스케치 (스크립트명 기반 Process Bar 재활용)

요청 반영:
1. 단계명은 일반 문구 대신 `terraformScriptName`을 사용한다.
2. `3) 최종 확인` 단계는 제거한다.
3. `기존 Process Bar` 컴포넌트를 재활용한다.
4. `상태 다시 확인` 버튼은 노출하지 않는다.

### 11.1 기본 레이아웃 (텍스트 도식)

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ AWS 설치 진행 (Process Bar 재사용)                                          │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  [1] svc-vpc-apne2  ──  [2] svc-athena-glue  ──  [3] svc-dynamodb-role  ──  │
│                                                                              │
│                                  [4] BDC 리소스 설치 확인                   │
│                                                                              │
│  상태 예시                                                                   │
│  - 현재 단계: [2] svc-athena-glue (설치 확인중)                              │
│  - 완료 단계: [1] svc-vpc-apne2                                              │
│  - 대기 단계: [3] svc-dynamodb-role, [4] BDC 리소스 설치 확인               │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 11.2 단계 구성 규칙

1. Process Bar 단계 수는 `serviceScripts.length + 1` 이다.
2. 1~N 단계는 Service Terraform Script 이름(`terraformScriptName`)으로 표시한다.
3. 마지막 단계(N+1)는 고정 라벨 `BDC 리소스 설치 확인`으로 표시한다.
4. `최종 확인` 같은 별도 후행 단계는 두지 않는다.

### 11.3 상태 매핑 규칙

```text
Service Script 상태
- COMPLETED      -> 완료 단계
- PENDING/FAILED -> 진행중 또는 대기 단계(현재 선택/순서에 따라)

BDC 단계 상태
- bdcInstallationRequired = true  -> 진행중/대기
- bdcInstallationRequired = false -> 완료
```

### 11.4 상세 패널과의 관계

1. 상단 Process Bar는 "현재 어디까지 진행됐는지"만 보여준다.
2. 하단 Script/Resource 패널은 기존처럼 상세 대상과 설치 상태를 보여준다.
3. 사용자는 상단에서 흐름을 보고, 하단에서 실제 조치 대상을 확인한다.
