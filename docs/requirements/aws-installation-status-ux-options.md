# AWS 설치 UX 요구사항 정리 (조치 중심)

> 작성일: 2026-03-02
> 목적: 설치 상태 상세 노출 대신, 사용자 조치 필요 여부 중심으로 AWS 설치 UX를 정리

## 1. 확정된 전제

1. 모드(`AUTO`/`MANUAL`)에 따라 다른 UX를 만들지 않는다.
2. `AUTO` 모드에서 Service 측 Terraform Script 실행 주체는 BDC다.
3. Service 측 상태는 완료/미완료 중심으로만 신뢰 가능하다.
4. BDC 상세 상태(`RUNNING/SUCCESS/FAIL`)는 사용자에게 직접 노출하지 않는다.

## 2. UX 원칙

1. 화면의 핵심 질문은 두 가지로 제한한다.
   - 서비스 담당자가 조치할 내용이 있는가
   - BDC 측 설치가 아직 필요한가
2. 내부 구현 상태(스크립트 실행 중, 실패 코드)는 운영/개발용 정보로 간주하고 기본 화면에서 숨긴다.
3. Resource 단위 표시는 단순화한다.
   - `설치 확인중`
   - `설치 완료`

## 3. 공통 상태 모델

### 3.1 내부 신호(비노출)

- Service Script: `NOT_COMPLETED | COMPLETED | UNKNOWN`
- BDC Script: `PENDING | RUNNING | SUCCESS | FAIL`

### 3.2 사용자 노출 상태(핵심)

- 서비스 담당자 조치: `필요 | 불필요`
- BDC 설치 필요 여부: `필요 | 불필요`
- Resource 설치 상태: `설치 확인중 | 설치 완료 | 확인 필요`

### 3.3 매핑 규칙

| 조건 | 서비스 담당자 조치 | BDC 설치 필요 | Resource 표시 |
|---|---|---|---|
| Service Script 미완료 존재 | 필요 | 필요(또는 대기) | 설치 확인중 |
| Service Script 모두 완료 + BDC != SUCCESS | 불필요 | 필요 | 설치 확인중 |
| Service Script 모두 완료 + BDC == SUCCESS | 불필요 | 불필요 | 설치 완료 |
| Service/BDC 상태 불명확(UNKNOWN 등) | 필요(확인) | 필요(확인) | 확인 필요 |

## 4. 화면 레이아웃 (와이어프레임)

### 4.2 상단 요약 카드 규칙

```text
[서비스 담당자 조치 카드]
- 필요: 미완료/확인필요 Service Script가 1개 이상
- 불필요: 모든 Service Script가 완료

[BDC 설치 카드]
- 필요: BDC가 아직 최종 완료되지 않음
- 불필요: BDC 완료 확인됨
```

### 4.3 리소스 상태 표현 규칙

```text
설치 확인중  : Service 미완료 또는 BDC 미완료
설치 완료    : Service 완료 + BDC 완료
확인 필요    : 스크립트 상태를 신뢰할 수 없는 경우(UNKNOWN 등)
```

설명:
- BDC는 사실상 단일 Script이므로 같은 시점의 여러 Resource가 동일 상태를 가질 수 있다.
- 사용자에게는 내부 실행 상태가 아니라, "지금 조치가 필요한지"와 "설치 완료 확인이 끝났는지"만 노출한다.

## 5. Terraform Script 중심 UI 대안

요청사항에 맞춰 Script를 1급 객체로 올리고, 각 Script를 확장 가능한 컴포넌트(Accordion)로 표현하는 안을 비교한다.

### 5.1 대안 A: Service Script 전용 Accordion (Service만 노출)

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 서비스 담당자 조치: 필요   |   BDC 설치: 필요                               │
├──────────────────────────────────────────────────────────────────────────────┤
│ Service Terraform Scripts                                                   │
│                                                                              │
│ ▸ svc-vpc-apne2                [서비스 측 확인 필요]  (리소스 2개)          │
│   - RDS-A, RDS-B                                                              │
│   - [설치 가이드] [상태 확인]                                                  │
│                                                                              │
│ ▾ svc-athena-glue              [설치 완료]           (리소스 1개)            │
│   - Athena-C                                                                 │
│                                                                              │
│ ▸ svc-dynamodb-role            [확인 필요]           (리소스 1개)            │
│   - Dynamo-D                                                                 │
│   - [상태 확인]                                                                │
└──────────────────────────────────────────────────────────────────────────────┘
```

장점:
- Service 담당자 입장에서 가장 직관적
- BDC 상세를 숨겨 정보 과다 노출 방지

단점:
- Resource 전체 현황을 표로 빠르게 훑기 어려움
- BDC 필요 영향이 리소스별로 어떻게 보이는지 약함

### 5.2 대안 B: Service Accordion + Resource 패널 (권장)

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 서비스 담당자 조치: 필요   |   BDC 설치: 필요                               │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ ┌────────────────────────────┐  ┌─────────────────────────────────────────┐ │
│ │ Service Scripts             │  │ 선택된 Script 리소스                    │ │
│ │                            │  │                                         │ │
│ │ ▾ svc-vpc-apne2            │  │ Script: svc-vpc-apne2                  │ │
│ │   [서비스 측 확인 필요]     │  │                                         │ │
│ │ ▸ svc-athena-glue          │  │ ┌──────────┬──────────────┬───────────┐ │ │
│ │   [설치 완료]               │  │ │ Resource │ 설치 상태     │ 조치      │ │ │
│ │ ▸ svc-dynamodb-role        │  │ ├──────────┼──────────────┼───────────┤ │ │
│ │   [확인 필요]               │  │ │ RDS-A    │ 설치 확인중   │ 가이드    │ │ │
│ │                            │  │ │ RDS-B    │ 설치 확인중   │ 가이드    │ │ │
│ │                            │  │ └──────────┴──────────────┴───────────┘ │ │
│ └────────────────────────────┘  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────────────┘
```

장점:
- Script 중심과 Resource 가시성을 둘 다 만족
- 서비스 담당자 조치 맥락을 유지하면서 테이블 가독성 확보

단점:
- 레이아웃 복잡도 증가(2패널)
- 모바일 대응 시 축약 규칙 추가 필요

### 5.3 대안 C: Service + BDC 병렬 노출(상세) Accordion

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Service Script: svc-vpc-apne2                                               │
│ 상태: 서비스 측 확인 필요 | BDC: 설치 필요                                   │
│                                                                              │
│ ┌──────────┬──────────────────┬──────────┬────────────────┐                │
│ │ Resource │ Service 상태      │ BDC 상태 │ 최종 표시       │                │
│ ├──────────┼──────────────────┼──────────┼────────────────┤                │
│ │ RDS-A    │ 확인 필요         │ 필요     │ 설치 확인중     │                │
│ │ RDS-B    │ 확인 필요         │ 필요     │ 설치 확인중     │                │
│ └──────────┴──────────────────┴──────────┴────────────────┘                │
└──────────────────────────────────────────────────────────────────────────────┘
```

장점:
- 운영 관점 디버깅에 유리
- Service/BDC 역할을 명시적으로 구분

단점:
- 사용자에게 불필요한 상태 정보가 늘어남
- 현재 요구사항(조치 중심 단순 UX)과 충돌 가능성 높음

## 6. 대안별 비교 요약

| 구분 | 대안 A (Service만) | 대안 B (Service+Resource) | 대안 C (Service+BDC 상세) |
|---|---|---|---|
| Script 중심성 | 높음 | 높음 | 높음 |
| Resource 가시성 | 중간 | 높음 | 높음 |
| 사용자 단순성 | 매우 높음 | 높음 | 낮음 |
| 구현 복잡도 | 낮음 | 중간 | 높음 |
| 현재 요구 적합성 | 높음 | **매우 높음** | 낮음 |

## 7. 현재 권장 방향

1. 1차는 대안 B로 진행한다.
2. BDC는 상단 요약(`필요/불필요`)만 노출하고, Script/Resource 행에는 상세 상태를 넣지 않는다.
3. Script 상태 라벨은 Service 중심으로 단순화한다.
   - `서비스 측 확인 필요`
   - `설치 완료`
   - `확인 필요`
4. Resource 상태는 기존 정책대로 유지한다.
   - `설치 확인중`
   - `설치 완료`
   - `확인 필요`

## 8. MVP 구현 범위

1. Service Script Accordion 컴포넌트 추가(expand/collapse)
2. 선택 Script 기준 Resource 패널 연동
3. 상단 요약 카드 2개 상태(서비스 조치 필요, BDC 설치 필요) 유지
4. BDC 상세 상태는 화면에서 숨김(내부 로직에서만 사용)

## 9. 확인 필요 항목

1. 초기 릴리즈를 대안 B로 확정할지
2. 대안 B에서 기본 펼침 Script를
   - 미완료 Script 우선
   - 최근 변경 Script 우선
   중 어느 규칙으로 할지
3. `BDC == FAIL` 시 사용자 노출을 `설치 확인중`으로 유지할지, `확인 필요`로 올릴지
