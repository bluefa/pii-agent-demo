# AWS 설치 UX 요구사항 정리 (조치 중심)

> 작성일: 2026-03-02
> 목적: 설치 상태 상세 노출 대신, 사용자 조치 필요 여부 중심으로 AWS 설치 UX를 정리

## 1. 확정된 전제

1. 모드(`AUTO`/`MANUAL`)에 따라 다른 UX를 만들지 않는다.
2. `AUTO` 모드에서 Service 측 Terraform Script 실행 주체는 BDC다.
3. Service 측 상태는 완료/미완료 중심으로만 신뢰 가능하다.
4. BDC 상세 상태(`RUNNING/SUCCESS/FAIL`)는 사용자에게 직접 노출하지 않는다.

## 2. UX 원칙

1. 화면의 핵심 질문은 두 가지로 제한한다.
   - 서비스 담당자가 조치할 내용이 있는가
   - BDC 측 설치가 아직 필요한가
2. 내부 구현 상태(스크립트 실행 중, 실패 코드)는 운영/개발용 정보로 간주하고 기본 화면에서 숨긴다.
3. Resource 단위 표시는 단순화한다.
   - `설치 확인중`
   - `설치 완료`

## 3. 공통 상태 모델

### 3.1 내부 신호(비노출)

- Service Script: `NOT_COMPLETED | COMPLETED | UNKNOWN`
- BDC Script: `PENDING | RUNNING | SUCCESS | FAIL`

### 3.2 사용자 노출 상태(핵심)

- 서비스 담당자 조치: `필요 | 불필요`
- BDC 설치 필요 여부: `필요 | 불필요`
- Resource 설치 상태: `설치 확인중 | 설치 완료 | 확인 필요`

### 3.3 매핑 규칙

| 조건 | 서비스 담당자 조치 | BDC 설치 필요 | Resource 표시 |
|---|---|---|---|
| Service Script 미완료 존재 | 필요 | 필요(또는 대기) | 설치 확인중 |
| Service Script 모두 완료 + BDC != SUCCESS | 불필요 | 필요 | 설치 확인중 |
| Service Script 모두 완료 + BDC == SUCCESS | 불필요 | 불필요 | 설치 완료 |
| Service/BDC 상태 불명확(UNKNOWN 등) | 필요(확인) | 필요(확인) | 확인 필요 |

## 4. 화면 도식도

### 4.1 상단 요약 카드

```text
┌──────────────────────────────────────────────┐
│ 서비스 담당자 조치: 필요                      │
│ BDC 설치: 필요                               │
│ 안내: 설치 대상 리소스를 확인하고 설치를 진행하세요. │
└──────────────────────────────────────────────┘
```

### 4.2 Resource 테이블

```text
┌──────────┬──────────────────────┬────────────────┐
│ Resource │ terraformScriptName  │ 설치 상태      │
├──────────┼──────────────────────┼────────────────┤
│ A        │ svc-vpc-apne2        │ 설치 확인중    │
│ B        │ svc-vpc-apne2        │ 설치 확인중    │
│ C        │ svc-athena-glue      │ 설치 완료      │
└──────────┴──────────────────────┴────────────────┘
```

설명:
- BDC는 사실상 단일 Script이므로 동일 시점에는 다수 Resource가 같은 상태를 가질 수 있다.
- 사용자는 각 Resource를 보되, 판단 기준은 "아직 설치 확인중인지/완료인지"만 본다.

## 5. 다양한 의견과 선택 근거

1. 상태 정밀도 중심(`실행중/성공/실패`) 의견
   - 장점: 운영 추적 용이
   - 미선택 이유: 현재 목표(사용자 조치 판단) 대비 정보 과다
2. 모드별 UX 분리 의견
   - 장점: 구현 상태와 화면 정합성 높음
   - 미선택 이유: 사용자 학습 비용 증가, 정책상 모드별 UX 분기 회피
3. 조치 중심 단순화 의견
   - 장점: 사용자가 다음 액션을 빠르게 이해
   - 선택 이유: 이번 요구사항에 가장 직접적으로 부합

## 6. 최종 선택안

1. 모드 공통 UX를 유지한다.
2. 상단은 `서비스 담당자 조치 필요 여부`와 `BDC 설치 필요 여부`만 보여준다.
3. Resource는 `설치 확인중/설치 완료` 중심으로 표시한다.
4. 기술 상세 상태(BDC RUNNING/FAIL 등)는 기본 화면에서 숨기고, 필요 시 내부 운영 로그에서만 확인한다.

## 7. MVP 구현 범위

1. 상단 요약 카드 2개 상태(서비스 조치 필요, BDC 설치 필요) 추가
2. Resource 테이블 상태를 `설치 확인중/설치 완료/확인 필요`로 표준화
3. 기존 상세 상태 문구(`실행중/실패` 등) 기본 화면 제거
4. 상태 매핑 로직을 `terraformScriptName` 기준으로 일관 적용

## 8. 확인 필요 항목

1. `BDC == FAIL`일 때 사용자 노출을 `설치 확인중`으로 유지할지, `확인 필요`로 올릴지
